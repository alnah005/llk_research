# 3.1 Blaze vs TTNN Lifecycles

Both Blaze and TTNN ultimately produce a `Program` object enqueued to a device command queue, but the paths diverge in where, when, and in what language the critical decisions are made. This section traces each lifecycle end-to-end and introduces labeled decision points (A through H) that thread through the rest of this chapter and into [Section 3.2](02_translation_challenges.md). Rather than repeating internals from [Chapter 1](../ch01_ttnn_registration_contract/index.md) (TTNN) and [Chapter 2, Section 2.2](../ch02_generic_op_dispatch/02_blaze_compilation_pipeline.md) (Blaze), the focus is on the specific points where each system's decision locus makes the other system's assumptions invalid.

---

## 3.1.1 The Blaze Lifecycle

A Blaze operation moves through five stages, each implemented in Python (until the final dispatch into TT-Metal's C++ runtime). The stages accumulate mutable state into a shared `FusedProgram` / `BlazeProgram` object.

### Stage 1: Op Definition (Author Time)

The op author defines a Python class inheriting from `MicroOp` or `FusedOp`:

```python
# Source: blaze/ops/matmul/op.py (representative)
class Matmul(MicroOp):
    name = "matmul"
    a: Input = Input()
    b: Input = Input()
    out: Output = Output()

    @staticmethod
    def emit(f, a, b, output_tensor, *, transpose_b=False, ...):
        a_cb = f.cb_from_tensor(a)             # CB allocation: decision A
        b_cb = f.cb_from_tensor(b)             # CB allocation: decision A
        out_cb = f.cb_from_tensor(output_tensor)
        f.unified_ct_args([                    # CT args: decision B
            ("matmul.in0_w", a_cb.inner_dim),
            ("matmul.in1_w", b_cb.inner_dim),
            ("matmul.transpose_b", int(transpose_b)),
        ])
        f.per_core_unified_ct_args([           # Per-core specialization: decision D
            f.flag("matmul.is_active", out_cb.core_ranges),
        ])
        return f.output("matmul", out_cb, src=a_cb)
```

At this stage, CB IDs, CT arg names and values, per-core specialization flags, and semaphore names are all determined by Python code. No C++ is involved.

### Stage 2: Composition (Compile Time)

When `BlazeCompiler.compile()` runs, it creates a `FusedProgram` and invokes the op's composition function. For a `FusedOp`, the `compose()` function chains multiple `emit()` calls:

```python
# FusedOp composition (representative)
@classmethod
def compose(cls, f, tensors, output, user_args):
    intermediate = RMSNorm.emit(f, tensors["x"], ...)
    result = Matmul.emit(f, intermediate, tensors["w"], output, ...)
    return result
```

Each `emit()` call mutates `f` in-place, appending CB descriptors, CT args, semaphore allocations, and shadow graph nodes. The `FusedProgram` accumulates state across all `emit()` calls -- this is the **stateful accumulation** that has no analog in TTNN.

### Stage 3: Engine Pipeline (Compile Time)

The `BlazeCompiler` runs three engines in sequence:

1. **`CBEngine.assign(graph)`** -- Assigns CB IDs from the dataflow graph, optimizing for temporal reuse (CBs with non-overlapping lifetimes share IDs). *Decision A.*
2. **`SemEngine.assign(graph)`** -- Assigns global semaphore indices, tracking per-device allocation. *Decision C.*
3. **`CTArgEngine`** -- Resolves compile-time arguments, including per-core specialization via `PerCoreCompileTimeDescriptor`. *Decisions B, D.*

These engines operate on the `BlazeGraph` IR and form an **integrated resource allocation pipeline** that jointly optimizes CB IDs, semaphore slots, and CT arg layouts across all constituent MicroOps.

### Stage 4: Kernel Codegen and Assembly (Compile Time)

`kernel_codegen.py` generates C++ kernel source from the shadow graph:

```cpp
// Generated by kernel_codegen.py (representative)
#include "blaze/ops/reduce/reduce.hpp"
#include "blaze/ops/mcast/mcast.hpp"
#include "blaze/ops/multiply/multiply.hpp"

void MAIN {
    // Phase 0: Reduce
    reduce::run(ct::reduce.src, ct::reduce.dst, ct::reduce.num_pages);
    // Phase 1: Mcast
    mcast::run(ct::mcast.src, ct::mcast.sem_addr);
    // Phase 2: Multiply
    multiply::run(ct::multiply.a, ct::multiply.b, ct::multiply.dst);
}
```

The generated source is JIT-compiled by TT-Metal's kernel compilation infrastructure. *Decision E.* Then `BlazeProgram.build()` constructs a `ProgramDescriptor` from the accumulated state:

```python
# Source: blaze/program.py (simplified)
def build(self) -> ttnn.ProgramDescriptor:
    ukd = UnifiedKernelDescriptor(
        kernel_source=self._kernel,
        core_ranges=self._core_ranges,
        ncrisc_named_compile_time_args=self._ncrisc_args,
        per_core_compile_time_descriptors=self._per_core_descriptors,
    )
    return ttnn.ProgramDescriptor(
        kernels=ukd.get_kernel_descriptors().kernels,
        cbs=self._cb_descriptors,
        semaphores=self._semaphore_descriptors,
    )
```

At this point, the `ProgramDescriptor` is complete and all semantic metadata (op names, graph topology, port semantics, validation rules) is discarded -- the "metadata discard boundary" documented in [Section 2.2](../ch02_generic_op_dispatch/02_blaze_compilation_pipeline.md).

### Stage 5: Dispatch (Runtime)

```python
# Dispatch: Python calls generic_op, which does pass-through execution
ttnn.generic_op(io_tensors, mesh_program_descriptor)
```

Execution passes through `GenericOp::invoke()` into `device_operation::launch<GenericOpDeviceOperation>()`. The C++ side hashes the descriptor, checks the program cache, and on a miss, calls `GenericMeshProgramFactory.create_mesh_workload()` which mechanically translates the `ProgramDescriptor` into `CreateKernel()`, `CreateCircularBuffer()`, and `CreateSemaphore()` calls. **No decisions are made -- it is a transcription.**

## 3.1.2 The TTNN Native Lifecycle

A native TTNN operation follows a fundamentally different path, with all program-construction decisions made in C++ inside the framework's dispatch pipeline.

### Stage 1: Op Definition (C++, Author Time)

The op author defines a `DeviceOperation` struct satisfying `DeviceOperationConcept`:

`BinaryDeviceOperation` defines typed attributes, a 7-alternative `program_factory_t` variant, and all three required methods (`validate_on_program_cache_miss`, `compute_output_specs`, `create_output_tensors`). See [Section 1.1.7](../ch01_ttnn_registration_contract/01_device_operation_concept.md) for the full 92-line struct listing with all type aliases, factory variants, optional extensions, and performance model.

### Stage 2: Registration (C++ Compile Time)

```cpp
constexpr auto add = ttnn::register_operation<
    "ttnn::add",
    operations::binary::BinaryOperation<operations::binary::BinaryOpType::ADD>>();
```

This stamps the name `"ttnn::add"` into the binary for profiling, graph tracing, and cache hashing.

### Stage 3: Dispatch and Factory Execution (Runtime)

When Python calls `ttnn.add(a, b)`, the full dispatch chain executes:

```
ttnn.add(a, b)                                   [Python -> nanobind]
  |
  +-- registered_operation_t::operator()          [C++]
        |
        +-- invoke() -> (attrs, tensor_args)
        +-- device_operation::launch<BinaryDeviceOperation>()
              |
              +-- compute_output_specs()                 [decision F: shape inference]
              +-- create_output_tensors()                [decision G: allocation]
              +-- compute_mesh_workload_hash()           [O(1) semantic hash]
              +-- Cache lookup
              |
              +-- MISS:
              |     +-- validate_on_program_cache_miss() [decision H: validation]
              |     +-- select_program_factory()
              |     +-- factory.create_mesh_workload()
              |           +-- CreateKernel()             [decision E: kernel]
              |           +-- CreateCircularBuffer()     [decision A: CB sizing]
              |           +-- CreateSemaphore()          [decision C: semaphores]
              |           +-- SetRuntimeArgs()           [decisions B, D: args]
              |           +-- return CachedMeshWorkload{shared_variables}
              |
              +-- HIT:
              |     +-- override_runtime_arguments()     [patch addresses]
              |
              +-- enqueue_mesh_workload()
```

Every resource allocation decision -- which CBs to create, how large they should be, which kernels to compile, what CT args to pass -- happens inside the C++ program factory. The factory is a **stateless pure function**: it receives immutable `(operation_attributes_t, tensor_args_t)` and produces a `Program` in a single call.

The critical observation: decisions A through E happen **before dispatch** in Blaze but **inside the factory on cache miss** in TTNN. This positional inversion is the source of every translation challenge.

## 3.1.4 Decision Locus Table

The following table maps each engineering decision to where it occurs in each system. Every row marked "Yes" represents a decision that Blaze makes in Python but TTNN expects to be made in C++.

| Decision | Blaze Phase | Blaze Owner | TTNN Phase | TTNN Owner | Mismatch? |
|----------|------------|-------------|------------|------------|----------:|
| **A. CB sizing and ID assignment** | Composition (emit) | `BlazeProgram.cb_from_tensor()`, `CBEngine` | Factory (cache miss) | `CreateCircularBuffer()` inside factory | Yes |
| **B. Compile-time arg computation** | Composition (emit) | `f.unified_ct_args()`, `CTArgEngine` | Factory (cache miss) | Inline C++ in factory `create()` | Yes |
| **C. Semaphore allocation** | Composition (emit) | `FusedProgram.semaphore()`, `SemEngine` | Factory (cache miss) | `CreateSemaphore()` inside factory | Yes |
| **D. Per-core specialization** | Composition (emit) | `PerCoreCompileTimeDescriptor`, `f.flag()` | Factory (cache miss) | Per-core `SetRuntimeArgs()` loop | Yes |
| **E. Kernel selection/generation** | Compilation (build) | `kernel_codegen.py`, `UnifiedKernelDescriptor` | Factory (cache miss) | Static kernel path in factory `create()` | Yes |
| **F. Output shape inference** | Pre-compilation | Implicit in `compose()` / user code | Dispatch (launch) | `compute_output_specs()` static method | Yes |
| **G. Output tensor allocation** | Pre-compilation | User or Blaze allocator pre-allocates | Dispatch (launch) | `create_output_tensors()` static method | Yes |
| **H. Semantic validation** | Compilation | `BlazeProgram.validate()`, per-op checks | Dispatch (cache miss) | `validate_on_program_cache_miss()` | Yes |
| **Factory selection** | Implicit (single merged program) | `BlazeCompiler._compile_single()` | Dispatch (cache miss) | `select_program_factory()` | Partial |
| **Cache hash computation** | Not computed (or `custom_program_hash`) | Optional field on descriptor | Dispatch | `compute_mesh_workload_hash()` | Partial |

**Reading the table:** Every "Yes" row must be bridged by the adapter -- either by pre-computing the decision in Python and passing it through structured attributes, or by making the pre-computed result look like a C++ factory output. The "Partial" rows indicate concerns where the mismatch is navigable without architectural changes.

## 3.1.5 The Three Incompatibilities

The decision locus table reveals three fundamental incompatibilities:

**1. Decision Language.** Blaze makes decisions A--E in Python; TTNN expects them in C++ inside `create_mesh_workload()`. No mechanism exists for the TTNN dispatch pipeline to call back into Python during factory execution (GIL contention, lifetime hazards, and build-system coupling all prevent this).

**2. Decision Timing.** Blaze makes CB allocation and CT arg decisions *before* entering the TTNN dispatch pipeline; TTNN makes them *inside* the pipeline during factory execution. This means Blaze cannot use TTNN's `create_output_tensors()` (it needs `Buffer*` pointers before calling `generic_op`), and validation timing differs (Blaze: before descriptor assembly; TTNN: after hash computation).

**3. Decision Scope.** Blaze's engine pipeline operates on the *entire fused graph*, assigning CB IDs and semaphore IDs globally across all constituent MicroOps. TTNN factories operate on a *single operation*. A FusedOp that chains three MicroOps sharing CBs and semaphores cannot be decomposed into three separate TTNN factory calls -- the phases share a single CB and semaphore namespace (see Challenge 4 in [Section 3.2](02_translation_challenges.md) for a concrete example).

---

## 3.1.6 Why the Inversion Exists

The temporal and language inversions are not oversights -- they are the enabling design decisions behind Blaze's value proposition:

1. **Python-first authoring.** Blaze allows op authors to define CB layouts, CT args, and kernel compositions in Python. If these decisions were deferred to a C++ factory, the entire `emit()` / `compose()` API would need to be reimplemented in C++, eliminating the rapid-iteration benefit.

2. **Cross-op resource sharing.** A `FusedOp` composes multiple MicroOps into a single program with shared CBs and semaphores. This requires a stateful builder (`FusedProgram`) that accumulates resources across `emit()` calls. TTNN's stateless factory model has no equivalent.

3. **Dynamic kernel generation.** Blaze's `kernel_codegen.py` generates kernel source specialized to the exact graph topology. The kernel includes phase-switching logic, CB aliases, and CT arg accessors determined by the composition. TTNN kernels are static source files.

4. **Compile-once, run-many.** Blaze separates compilation from execution. A `MeshCompiledProgram` can be compiled once and run many times with different tensor addresses (via runtime arg patching), amortizing the Python-side compilation cost.

These benefits are worth preserving. The challenge -- addressed in [Chapter 4](../ch04_adapter_pattern/index.md) -- is to bridge the inversion without sacrificing them.

## 3.1.7 The Cache Interaction Asymmetry

The program cache introduces a further asymmetry in per-invocation costs. Both systems have a first-invocation compilation cost and a steady-state cost. The key distinction is what each pays on *repeated* invocations.

**First invocation** (compile + cache miss):

$$C_{\text{TTNN}}^{\text{first}} = C_{\text{hash}} + C_{\text{validate}} + C_{\text{factory}}$$

$$C_{\text{Blaze}}^{\text{first}} = C_{\text{engines}} + C_{\text{emit}} + C_{\text{build}} + C_{\text{descriptor}} + C_{\text{hash}} + C_{\text{create\_at}}$$

**Steady-state** (reusing a compiled `MeshCompiledProgram` with a cache hit):

$$C_{\text{TTNN}}^{\text{steady}} = C_{\text{hash}} + C_{\text{override}}$$

$$C_{\text{Blaze}}^{\text{steady}} = C_{\text{hash}} + C_{\text{override}}$$

The per-invocation structure is the same on cache hits: both systems hash and then patch runtime arguments. The asymmetry lies in the *hash cost*: TTNN hashes $O(1)$ semantic attributes (shapes, dtypes, config flags), while Blaze hashes the full `ProgramDescriptor` at $O(\sum |K_i| + \sum |CB_j| + \sum |S_k|)$ unless `custom_program_hash` is set (which Blaze currently does not use). The Python compilation pipeline ($C_{\text{engines}} + C_{\text{emit}} + C_{\text{build}}$) is paid only on the first invocation -- subsequent `run()` calls on the same `MeshCompiledProgram` reuse the already-built descriptor, as described in Section 3.1.6.

---

## 3.1.8 Implications for Registration

The lifecycle comparison yields three structural constraints for any bridge between Blaze and TTNN:

1. **The bridge cannot move decisions A--E into TTNN factories** without reimplementing the Blaze engine stack in C++. The engine stack is tightly coupled to the Python `BlazeGraph` and `FusedProgram` abstractions.

2. **The bridge must accept that the `ProgramDescriptor` arrives pre-built.** Since Blaze makes decisions A--E before dispatch, the adapter must consume a finished descriptor rather than constructing a `Program` from semantic attributes.

3. **The bridge must inject identity and metadata without changing the program.** The adapter's job is to wrap the pre-built `ProgramDescriptor` in a typed `DeviceOperation` struct that provides a unique name, semantic hash, output-spec inference, and validation -- all without altering the kernels, CBs, or semaphores that Blaze constructed. This is the adapter pattern described in [Chapter 4](../ch04_adapter_pattern/index.md).

Both pipelines converge at the `Program` object submitted to `enqueue_mesh_workload()`. Blaze already produces valid programs that execute correctly on silicon. The adapter does not need to change what runs on hardware -- it only needs to change how the program reaches the `enqueue_mesh_workload()` call, adding the metadata that `generic_op` discards.

---

## Key Takeaways

- **Three incompatibilities** separate the two lifecycles: decision language (Python vs C++), decision timing (before dispatch vs inside the factory), and decision scope (whole-graph vs single-op). All ten decision categories in the locus table are affected.

- **The inversion exists for good reasons** -- Python-first authoring, cross-op fusion, dynamic kernel generation, and compile-once-run-many execution. Any registration strategy must preserve these benefits.

- **The cache hash cost is asymmetric:** TTNN hashes $O(1)$ semantic attributes; Blaze hashes the full $O(n)$ descriptor contents. The Python compilation pipeline itself is paid only once per `MeshCompiledProgram`, not on every invocation.
