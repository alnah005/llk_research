# Chapter 5: Compile-Time Configuration, Preprocessor Defines, and `build.h`

## 5.3 Matmul-Specific Configuration Structs

The matmul kernel relies on several C++ structs and enums that configure
address modification patterns, MOP loop templates, and destination tile
geometry. These are defined in the LLK headers (not generated by the build
system) and instantiated by the matmul API functions at runtime.

All definitions come from `tt_llk_blackhole/common/inc/`.

---

### 5.3.1 `addr_mod_t` -- Address Modifier Configuration

**Source:** `ckernel_addrmod.h`

Address modifiers control how SrcA, SrcB, and Dest pointers auto-increment
after each MVMUL instruction. There are 8 slots (ADDR_MOD_0--7), each
independently programmable.

#### Struct Definition with `val()` Methods

```cpp
constexpr std::uint32_t SRC_INCR_MASK  = 0x3F;
constexpr std::uint32_t DEST_INCR_MASK = 0x3FF;

struct addr_mod_t
{
    struct addr_mod_src_t {            // SrcA and SrcB
        std::uint8_t incr = 0;         // 6-bit increment (masked to 0x3F)
        std::uint8_t clr  = 0;         // 1-bit: reset counter to 0
        std::uint8_t cr   = 0;         // 1-bit: carry-reset (reset carry chain)

        constexpr std::uint8_t val() const {
            return (incr & SRC_INCR_MASK) | ((cr & 0x1) << 6) | ((clr & 0x1) << 7);
        }
    };

    struct addr_mod_dest_t {           // Destination register
        std::int16_t incr    = 0;      // 10-bit signed increment
        std::uint8_t clr     = 0;      // 1-bit: clear
        std::uint8_t cr      = 0;      // 1-bit: carry-reset
        std::uint8_t c_to_cr = 0;      // 1-bit: carry-to-carry-reset

        constexpr std::uint16_t val() const {
            return (incr & DEST_INCR_MASK) | ((cr & 0x1) << 10)
                   | ((clr & 0x1) << 11) | ((c_to_cr & 0x1) << 12);
        }
    };

    struct addr_mod_fidelity_t {       // Fidelity phase counter
        std::uint8_t incr = 0;         // 2-bit phase increment
        std::uint8_t clr  = 0;         // 1-bit: clear phase counter

        constexpr std::uint16_t val() const {
            return (incr & 0x3) | ((clr & 0x1) << 2);
        }
    };

    struct addr_mod_bias_t {           // Bias counter
        std::uint8_t incr = 0;         // 4-bit increment
        std::uint8_t clr  = 0;

        constexpr std::uint16_t val() const {
            return (incr & 0xF) | ((clr & 0x1) << 4);
        }
    };

    addr_mod_src_t srca = {}, srcb = {};
    addr_mod_dest_t dest = {};
    addr_mod_fidelity_t fidelity = {};
    addr_mod_bias_t bias = {};
    addr_mod_src_t pack_ysrc = {}, pack_ydst = {};

    __attribute__((always_inline)) inline void set(const std::uint8_t mod_index) const
    {
        TTI_SETC16(addr_mod_src_reg_addr[mod_index],
                    srca.val() | (srcb.val() << 8));
        TTI_SETC16(addr_mod_dest_reg_addr[mod_index],
                    dest.val() | (fidelity.val() << 13));
        TTI_SETC16(addr_mod_bias_reg_addr[mod_index], bias.val());
    }
};
```

The `set()` method writes three 16-bit registers per slot: source (SRCA low
byte, SRCB high byte), dest+fidelity (dest in bits 0--12, fidelity in bits
13--15), and bias.

#### Field Reference Tables

**`addr_mod_src_t`** (SrcA / SrcB):

| Field | Width | Range | Purpose |
|---|---|---|---|
| `incr` | 6 bits | 0--63 | Post-operation address increment. Matmul uses 8 (row group), 16 (face), 32/48 (tile). |
| `clr` | 1 bit | 0/1 | Resets counter to zero. Used at phase/tile boundaries. |
| `cr` | 1 bit | 0/1 | Carry-reset: resets the carry chain so subsequent increments start from a new base. Used at face boundaries. |

**`addr_mod_dest_t`** (DEST):

| Field | Width | Range | Purpose |
|---|---|---|---|
| `incr` | 10 bits | -512 to +511 | Signed DEST increment. Typical matmul value: +8. |
| `clr` | 1 bit | 0/1 | Clears DEST counter. |
| `cr` | 1 bit | 0/1 | Carry-reset for DEST. |
| `c_to_cr` | 1 bit | 0/1 | Propagates carry overflow into carry-reset logic. |

**`addr_mod_fidelity_t`**: `incr` (2 bits, 0--3) and `clr` (1 bit).
**`addr_mod_bias_t`**: `incr` (4 bits, 0--15) and `clr` (1 bit).

#### Matmul Address Modifier Slot Configuration

For a standard non-transposed 32x32 tile matmul, programmed by
`matmul_configure_addrmod<math_fidelity, THROTTLE_LEVEL>()`:

```cpp
// ADDR_MOD_0: Inner loop -- advance SRCB and DEST by 8 rows
addr_mod_t{.srca={.incr=0}, .srcb={.incr=8}, .dest={.incr=8}}.set(ADDR_MOD_0);

// ADDR_MOD_1: Face boundary -- advance SRCA, carry-reset SRCB
addr_mod_t{.srca={.incr=16}, .srcb={.cr=1}, .dest={.incr=8}}.set(ADDR_MOD_1);

// ADDR_MOD_2: Half-tile -- carry-reset SRCA, advance SRCB to second half
addr_mod_t{.srca={.cr=1}, .srcb={.incr=32,.cr=1}, .dest={.incr=8}}.set(ADDR_MOD_2);

// ADDR_MOD_4: Tile boundary -- advance both, carry-reset DEST
addr_mod_t{.srca={.incr=32,.cr=1}, .srcb={.incr=48,.cr=1},
           .dest={.cr=1}}.set(ADDR_MOD_4);

// ADDR_MOD_5: Phase end -- clear all, advance fidelity phase
addr_mod_t{.srca={.clr=1,.cr=1}, .srcb={.clr=1,.cr=1},
           .dest={.clr=1,.cr=1},
           .fidelity={.incr=fidelity_increment}}.set(ADDR_MOD_5);
```

`fidelity_increment` is 1 for HiFi2--HiFi4 and 0 for LoFi. When
`THROTTLE_LEVEL > 0`, ADDR_MOD_6 is additionally configured with
`.fidelity={.clr=1}` to reset the fidelity counter.

> **Cross-reference:** Chapter 4, Section 2 walks through the complete address modifier programming sequence.

#### `addr_mod_pack_t`

```cpp
{
    addr_mod_vals_t y_src = {}, y_dst = {};       // 4-bit incr, clr, cr
    addr_mod_reduced_t z_src = {}, z_dst = {};    // 1-bit incr, clr
    void set(const std::uint8_t mod_index) const;
};
```

### 5.3.2 `ckernel_template` -- Math MOP Loop Template

**Source:** `ckernel_template.h`

The `ckernel_template` class programs the Tensix MOP (Macro Operation)
hardware loop engine -- a double-nested loop that executes Tensix instructions
without RISC-V intervention.

#### Loop Structure

```
LOOP_OUTER: <m_outer_loop_len>
  START_OP:    m_start_op0
  LOOP_INNER: <m_inner_loop_len>
    LOOP_OP0:  m_loop_op0
    LOOP_OP1:  m_loop_op1  (or replacement on last iteration)
  END_LOOP_INNER
  END_OP0:     m_end_op0
  END_OP1:     m_end_op1
END_LOOP_OUTER
```

#### Class Definition

```cpp
class ckernel_template
{
    const std::uint32_t m_outer_loop_len;
    const std::uint32_t m_inner_loop_len;
    std::uint32_t m_loop_op0;
    std::uint32_t m_loop_op1;
    std::uint32_t m_end_op0, m_end_op1, m_start_op0;
    std::uint32_t m_loop0_last_instr;
    std::uint32_t m_loop1_last_instr;

public:
    ckernel_template(uint32_t outer, uint32_t inner, uint32_t loop_op);
    ckernel_template(uint32_t outer, uint32_t inner, uint32_t op0, uint32_t op1);

    void set_loop_op0(uint32_t loop_op);
    void set_loop_op1(uint32_t loop_op);
    void set_end_ops(uint32_t end_op0, uint32_t end_op1);
    void set_end_op(uint32_t end_op0);
    void set_start_op(uint32_t start_op0);
    void set_last_inner_loop_instr(uint32_t op);
    void set_last_outer_loop_instr(uint32_t op);

    void program();
    static void run();
    void program_and_run();
};
```

#### Member Fields

| Field | Type | Matmul Meaning |
|---|---|---|
| `m_outer_loop_len` | `const uint32_t` | Always 1 for matmul. |
| `m_inner_loop_len` | `const uint32_t` | Fidelity phase count: 1 (LoFi), 2 (HiFi2), 3 (HiFi3), 4 (HiFi4). |
| `m_loop_op0` | `uint32_t` | Replay buffer instruction that plays the MVMUL sequence. |
| `m_loop_op1` | `uint32_t` | Often `TT_OP_NOP`. Replaceable on last iteration (see below). |
| `m_start_op0` | `uint32_t` | Pre-inner-loop instruction. Can set up source register positions. |
| `m_end_op0` | `uint32_t` | Post-inner-loop. HiFi: `SETRWC(CLR_A/CLR_B)` to clear reused source. |
| `m_end_op1` | `uint32_t` | Second post-inner-loop instruction. |
| `m_loop0_last_instr` | `uint32_t` | Replaces `m_loop_op1` on the final iteration of BOTH loops. |
| `m_loop1_last_instr` | `uint32_t` | Replaces `m_loop_op1` on the last inner iter (non-final outer). |

#### Last-Iteration Replacement

The MOP hardware dynamically replaces `m_loop_op1` on final iterations:

```
if (last_inner && last_outer):  use m_loop0_last_instr
else if (last_inner):           use m_loop1_last_instr
else:                           use m_loop_op1
```

This inserts source register clear operations (CLR_A/CLR_B) on the final
iteration without branch overhead. The outer-loop replacement takes priority
when both loops are on their last iteration.

#### Constructor Details

**Single-instruction constructor:**
```cpp
ckernel_template(uint32_t outer, uint32_t inner, uint32_t loop_op)
    : m_outer_loop_len(outer), m_inner_loop_len(inner),
      m_loop_op0(loop_op), m_loop_op1(TT_OP_NOP),
      m_end_op0(TT_OP_NOP), m_end_op1(TT_OP_NOP), m_start_op0(TT_OP_NOP)
{
    m_loop0_last_instr = loop_op;  // Default: no special last-iter behavior
    m_loop1_last_instr = loop_op;
}
```

**Two-instruction constructor:**
```cpp
ckernel_template(uint32_t outer, uint32_t inner, uint32_t op0, uint32_t op1)
    : m_outer_loop_len(outer), m_inner_loop_len(inner),
      m_loop_op0(op0), m_loop_op1(op1),
      m_end_op0(TT_OP_NOP), m_end_op1(TT_OP_NOP), m_start_op0(TT_OP_NOP)
{
    m_loop0_last_instr = op1;  // Default: last-iter uses same op
    m_loop1_last_instr = op1;
}
```

All optional slots default to `TT_OP_NOP`. Key setter methods:

- `set_start_op(op)` -- sets the instruction that executes once at the
  beginning of each outer loop iteration, before the inner loop starts. For
  matmul, this can be a `SETADCXY` instruction to position source registers.
- `set_end_op(op)` -- sets `m_end_op0`, the instruction after the inner loop
  completes. Typically used for address counter resets or source valid flag
  clearing.
- `set_end_ops(op0, op1)` -- sets both `m_end_op0` and `m_end_op1`.
- `set_last_inner_loop_instr(op)` -- overrides `m_loop1_last_instr`. On the
  last inner iteration (but not the final outer iteration), `m_loop_op1` is
  replaced by this instruction. In matmul, commonly a CLEARDVALID for the
  inner operand.
- `set_last_outer_loop_instr(op)` -- overrides `m_loop0_last_instr`. On the
  absolute last iteration (last inner of last outer), `m_loop_op1` is replaced
  by this instruction. Typically a CLEARDVALID that releases both source
  registers.

#### `program()` and `run()`

```cpp
void ckernel_template::program() {
    volatile uint32_t *mop_cfg = (volatile uint32_t *)(TENSIX_MOP_CFG_BASE);
    mop_sync();  // Wait for previous MOP
    mop_cfg[0] = m_outer_loop_len;  mop_cfg[1] = m_inner_loop_len;
    mop_cfg[2] = m_start_op0;       mop_cfg[3] = m_end_op0;
    mop_cfg[4] = m_end_op1;         mop_cfg[5] = m_loop_op0;
    mop_cfg[6] = m_loop_op1;        mop_cfg[7] = m_loop0_last_instr;
    mop_cfg[8] = m_loop1_last_instr;
}

static void ckernel_template::run() { TTI_MOP(1, 0, 0); }  // type=1: math MOP
```

#### Matmul MOP Configuration

In `matmul_configure_mop<math_fidelity>()`:

```cpp
constexpr uint32_t inner_loops = high_fidelity ? to_underlying(math_fidelity) : 1;

ckernel_template tmp(
    1,           // outer loop: always 1
    inner_loops, // inner loop: 1 (LoFi) or 2/3/4 (HiFi)
    lltt::replay_insn(replay_buf_offset, replay_buf_len)
);

if constexpr (high_fidelity) {
    if (reuse_a) tmp.set_end_op(TT_OP_SETRWC(p_setrwc::CLR_A, ...));
    else         tmp.set_end_op(TT_OP_SETRWC(p_setrwc::CLR_B, ...));
}
tmp.program();
```

The replay buffer (16 instructions for 32x32 tiles) contains the MVMUL
sequence with address-modifier references for the face-traversal pattern.

> **Cross-reference:** Chapter 4, Section 2 traces the complete MOP programming for LoFi and HiFi matmul paths.

### 5.3.3 `ckernel_unpack_template` -- Unpack MOP Loop Template

**Source:** `ckernel_template.h`

The unpack template uses a single loop with a zmask for conditional execution:

```
LOOP (count iterations):
  if zmask[iteration]: UNPACR_A0, [A1,A2,A3 if halo], [UNPACR_B if B enabled]
  else:                SKIP_A, [SKIP_B if B enabled]
```

#### Class Definition (Key Fields)

```cpp
class ckernel_unpack_template
{
    const bool m_unpackB;       // Enable source B unpacking
    const bool m_unpack_halo;   // Enable halo (multi-strip) unpacking

    const std::uint32_t m_A0_instr, m_A1_instr, m_A2_instr, m_A3_instr;
    const std::uint32_t m_B_instr;
    const std::uint32_t m_skipA_instr;
    const std::uint32_t m_skipB_instr;

public:
    void program() const;
    static void run(const std::uint8_t count, const std::uint32_t zmask);
    static void run(const std::uint8_t count);  // Without zmask
    void program_and_run(const std::uint8_t count, const std::uint32_t zmask = 0);
};
```

| Field | Purpose |
|---|---|
| `m_unpackB` | If true, the B unpack/skip instruction pair is included. For matmul, this is true since both A and B operands must be unpacked. |
| `m_unpack_halo` | If true, A0/A1/A2/A3 multi-strip instructions are all active (convolution halo). For standard matmul, this is false. |
| `m_A0_instr` | Primary A unpack instruction (UNPACR). |
| `m_A1_instr`--`m_A3_instr` | Additional A strip unpack instructions (halo mode only). |
| `m_B_instr` | B unpack instruction. |
| `m_skipA_instr` | Instruction when zmask says "skip A" -- typically `INCADCZW` to advance Z counter without unpacking. |
| `m_skipB_instr` | Skip instruction for B. |

#### Factory Methods

| Factory | Pattern | Matmul Usage |
|---|---|---|
| `lA(A, skipA)` | A only | Single-operand unpack |
| `lB(B, skipB)` | B only | Load SrcB |
| `lBA(A, skipA, B, skipB)` | Both A and B | **Matmul: load both operands** |
| `lzA(neginf, A, skipA)` | Zero SrcA, then unpack | Reduce ops |
| `lhA(halo)` | Halo A | Convolution |

#### Default Instruction Constants

```cpp
static constexpr uint32_t DEF_A_instr =
    TT_OP_UNPACR(0, 0b1, 0, 0, 0, 0, 1, p_unpacr::RAREFYB_DISABLE, 0, 0, 0, 0, 1);
static constexpr uint32_t DEF_B_instr =
    TT_OP_UNPACR(1, 0b01, 0, 0, 0, 0, 1, p_unpacr::RAREFYB_DISABLE, 0, 0, 0, 0, 1);
static constexpr uint32_t DEF_SKIP_A  = TT_OP_INCADCZW(0b001, 0, 0, 0, 1);
static constexpr uint32_t DEF_SKIP_B  = TT_OP_INCADCZW(0b010, 0, 0, 0, 1);
```

`DEF_A_instr` and `DEF_B_instr` are the default unpack instructions for
operands 0 (SrcA) and 1 (SrcB) respectively. `DEF_SKIP_A/B` increment the
Z-counter without unpacking data, advancing the tile pointer.

#### `program()` and `run()`

```cpp
void ckernel_unpack_template::program() const
{
    volatile uint32_t *mop_cfg = (volatile uint32_t *)(TENSIX_MOP_CFG_BASE);
    mop_sync();
    mop_cfg[1] = m_unpackB | (m_unpack_halo << 1);  // Control flags
    mop_cfg[2] = m_B_instr;
    mop_cfg[3] = m_A0_instr;
    mop_cfg[4] = m_A1_instr;
    mop_cfg[5] = m_A2_instr;
    mop_cfg[6] = m_A3_instr;
    mop_cfg[7] = m_skipA_instr;
    mop_cfg[8] = m_skipB_instr;
}

static void run(const uint8_t count, const uint32_t zmask)
{
    TT_MOP_CFG(zmask >> 16);               // Upper 16 bits of zmask
    TT_MOP(0, count - 1, zmask & 0xFFFF);  // Type 0 = unpack MOP
}
```

Note: `mop_cfg[1]` holds B-enable/halo flags (not loop count as in the math
template). The MOP engine uses the first argument of `TT_MOP` (0 vs. 1) to
distinguish unpack and math modes. The zmask is split because the `TT_MOP`
instruction can only encode 16 bits directly.

A zmask-free variant exists for when all iterations should unpack:

```cpp
static void run(const std::uint8_t count)
{
    TT_MOP(0, count - 1, 0);  // zmask=0 means "unpack all"
}
```

### 5.3.4 The `HF` Define -- Half-Face Mode

**Source:** `llk_math_matmul.h`

```cpp
#ifndef HF
#define HF 0
#endif
```

`HF` (Half Face) controls whether the matmul processes half-face (8 rows)
instead of full-face (16 rows) per MVMUL step. Default is 0 (full face). Can
be overridden via `-DHF=1`. For the standard P150 32x32 matmul, `HF = 0` is
always correct.

### 5.3.5 `DstTileShape` -- Destination Tile Geometry

**Source:** `DstTileShape` enum is in `ckernel_defs.h`; `DstTileSizeLog2[]` is in `ckernel.h`.

```cpp
enum DstTileShape { Tile32x32 = 0, Tile32x16 = 1, Tile16x16 = 2 };

constexpr uint32_t DstTileSizeLog2[] = {
    6,  // Tile32x32: 1024 datum positions (4 faces x 256 elements, 2^6 row-groups)
    5,  // Tile32x16: 512 datum positions (2 faces, 2^5 row-groups)
    4   // Tile16x16: 256 datum positions (1 face, 2^4 row-groups)
};
```

`set_dst_write_addr()` converts tile index to register address:
`dst_index << DstTileSizeLog2[tile_shape]`. Matmul always uses `Tile32x32`.

The maximum tiles in DEST depends on tile shape and `is_fp32_dest_acc_en`
(which halves capacity by doubling per-datum width).

> **Cross-reference:** Chapter 2 covers `is_fp32_dest_acc_en` and DEST capacity.

### 5.3.6 Non-Standard Tile Dimension Handling

The matmul supports non-32x32 input tiles, detected via boolean flags:

```cpp
const bool is_in0_16x32 = (in0_tile_r_dim <= FACE_R_DIM) && (in0_tile_c_dim > FACE_C_DIM);
const bool is_in0_32x16 = (in0_tile_r_dim > FACE_R_DIM)  && (in0_tile_c_dim <= FACE_C_DIM);
```

These select different replay buffer lengths: 4 instructions (both inputs
narrow), 8 (one narrow), or 16 (standard 32x32).

Two cases are explicitly unsupported:

```cpp
LLK_ASSERT(!((in0_r == FACE_R_DIM) && (in0_c == FACE_C_DIM) &&
             (in1_r == FACE_R_DIM) && (in1_c == FACE_C_DIM)),
            "16x16 by 16x16 matmul is not supported");

LLK_ASSERT(!(transpose && (in1_r == TILE_R_DIM) && (in1_c == FACE_C_DIM)),
            "Transpose with input 1 dimensions 32x16 not supported");
```

Tile dimensions flow into `build.h` via the `IN_TILE_DIMS` RuntimeParameter
(fields: `in0_r_dim`, `in0_c_dim`, `in1_r_dim`, `in1_c_dim`, all defaulting
to 32).

### 5.3.7 `MathFidelity` -- FPU Precision Control

**Source:** `llk_defs.h`

```cpp
enum class MathFidelity : std::uint8_t
{
    LoFi  = 0,   // Note: value 1 is intentionally unused
    HiFi2 = 2,
    HiFi3 = 3,
    HiFi4 = 4
};
```

| Fidelity | Phases | Accuracy | Throughput |
|----------|--------|----------|-----------|
| `LoFi` | 1 | Lowest (sufficient for inference) | Maximum |
| `HiFi2` | 2 | Good for most training | 50% of LoFi |
| `HiFi3` | 3 | High accuracy | 33% of LoFi |
| `HiFi4` | 4 | Maximum (closest to IEEE float) | 25% of LoFi |

The helper `is_high_fidelity()` returns `true` for any fidelity above LoFi.
When high fidelity is enabled: the MOP inner loop count equals
`to_underlying(math_fidelity)` (2/3/4), ADDR_MOD_5 increments the fidelity
phase counter, and the end-op clears the source register after all phases.

> **Cross-reference:** Chapter 2 covers MathFidelity in the context of data formats; Chapter 4 covers how fidelity phases map to MOP inner loop counts.

### 5.3.9 How These Structs Connect to `build.h`

These structs are not generated by Python but are parameterized by `build.h`
values through template instantiation:

| `build.h` Value | Struct / Function It Configures |
|---|---|
| `MATH_FIDELITY` | `addr_mod_t` fidelity increment (ADDR_MOD_5); `ckernel_template` inner loop count |
| `THROTTLE_LEVEL` | Standard vs. throttled MOP config (adds ADDR_MOD_6) |
| `is_fp32_dest_acc_en` | `get_dest_max_tiles<>()` DEST capacity calculation |
| `dest_sync` | DEST buffer half/full mode for all three threads |
| `CT_DIM`, `RT_DIM`, `KT_DIM` | Tile loop bounds in the matmul execution function |

The complete configuration sequence for a matmul tile operation ties these
structs together:

1. **`_llk_math_matmul_init_<MathFidelity, THROTTLE_LEVEL>()`**
   - Calls `matmul_configure_addrmod()` to program `addr_mod_t` slots 0--6
   - Calls `matmul_configure_mop()` which loads the replay buffer with 16
     `TTI_MVMUL` instructions, constructs a `ckernel_template`, and calls
     `tmp.program()` to write MOP config registers
   - Resets hardware counters via `math::reset_counters()`

2. **`_llk_math_matmul_<MathFidelity>(dst_index, ct_dim, rt_dim)`**
   - Loops over the tile block dimensions (`rt_dim` x `ct_dim`)
   - For each tile: sets `dst_write_addr` using `DstTileShape::Tile32x32`
   - Calls `ckernel_template::run()` which triggers `TTI_MOP(1, 0, 0)`
   - The hardware autonomously executes the double-nested loop, replaying the
     MVMUL sequence with address modifiers advancing pointers through all four
     face pairs
   - Clears the appropriate source register at block boundaries

The entire tile multiply (8 face-pair operations x 2 MVMUL each x up to 4
fidelity phases = up to 64 MVMUL instructions, with `replay_buf_len = 16`
per phase for 32x32 tiles) executes as a single MOP trigger with zero
software overhead after the initial setup.

Python generates the *parameters*; the C++ structs consume them at compile time.
This keeps the generated header small while the hardware programming logic
lives in the LLK library.
