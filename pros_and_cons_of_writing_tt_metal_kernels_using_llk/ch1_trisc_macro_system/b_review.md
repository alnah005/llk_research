# Agent B Review: Chapter 1 — TRISC Macro System — Pass 1

1. **pitfalls.md, Section 2 ("Debugging Confusion: Line Numbers Do Not Match") — incorrect claim about preprocessor line-number behavior.** The text states: "After preprocessing, `MATH(x)` expands to just `x`, but multi-line macro arguments or surrounding eliminated code change the line count. The preprocessed output for TRISC1 of `bmm_large_block_zm_fused_bias_activation.cpp` looks nothing like the original 500-line source -- all `PACK(...)` and `UNPACK(...)` lines have vanished, and the remaining code has different line numbers." This is factually wrong about how the C preprocessor works. Standard C/C++ preprocessors emit `#line` directives (or equivalent blank lines) to maintain exact line-number correspondence between the original source and the preprocessed output. When `PACK(x)` expands to nothing, the preprocessor does NOT collapse the line — it emits an empty line to preserve the line count. Therefore, line numbers in preprocessed output match the original source file. The actual debugging confusion comes from (a) the JIT-generated wrapper files (`chlkc_math.cpp` etc.) that `#include` the user's kernel, adding line-number offsets from the wrapper preamble, and (b) template instantiation errors pointing into LLK internals. Both of these are real issues (and the chapter does mention them), but the stated mechanism — that macro expansion shifts line numbers — is incorrect and would give developers a wrong mental model when debugging. Suggested fix: replace the "macro expansion shifts line numbers" paragraph with an explanation of the generated wrapper file mechanism (`chlkc_math.cpp` includes the kernel source, so compiler errors report line numbers relative to the generated file unless `#line` directives are used).

1. **index.md, code snippet comment "lines 19-58" is off-by-one.** The snippet includes the final `#endif` that closes the `TRISC_UNPACK` block, which is on line 59 of `compute_kernel_api.h` (not line 58). The correct range is "lines 19-59". Verified against source: `#ifdef TRISC_MATH` starts at line 19, `#define UNPACK(x)` is line 58, `#endif` is line 59.

2. **pitfalls.md, Section 2 — misleading claim that the generated wrapper file "#includes the kernel after a preamble."** The text states: `simple_kernel_syntax::transform_to_legacy_syntax()` wraps the user's kernel source into a generated file "that `#include`s the kernel after a preamble of namespace declarations, `using` directives, and other boilerplate." Two factual problems: (a) The function does NOT use `#include`. It reads the kernel source, splits it at `kernel_main()`, renames the function (e.g., to `math_main`), wraps the body in a `namespace chlkc_math { }` block, and writes the transformed source directly into the generated `.cpp` file (see `genfiles.cpp` lines 112-133). (b) The preamble consists of `#define TRISC_X` and `#include "defines_generated.h"` (from `build_trisc_prolog`, line 137-141) plus the namespace wrapper — there are no `using` directives in the preamble.

3. **pitfalls.md, Section 2 — overstates the line-number offset problem for FILE_PATH kernels.** The text claims "a reported 'line 142' may correspond to line 95 in the actual kernel source." However, the JIT pipeline adds a `#line 1 "original_file.cpp"` directive for FILE_PATH source kernels (see `genfiles.cpp` lines 212-214), which instructs the compiler to report errors relative to the original file. The line-number offset issue therefore primarily affects SOURCE_CODE type kernels (where no `#line` directive is emitted), not the common FILE_PATH case. The chapter should clarify that the offset is mitigated for file-based kernels by the `#line` directive but remains a problem for inline source-code kernels.
