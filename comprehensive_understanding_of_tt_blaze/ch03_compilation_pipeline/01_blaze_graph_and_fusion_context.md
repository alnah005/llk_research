# 01 -- BlazeGraph and Fusion Context

This section covers the two foundational data structures of the compilation pipeline: the `BlazeGraph` DAG that encodes a fused operation, and the `FusionContext` context manager that builds it from user-level Python code. It also covers `ExternalTensor` placeholders, `FusionResult` handles, topological ordering, graph validation, and the `compile_engines()` orchestrator that drives the three downstream engines.

**Source files:** `blaze/graph.py`, `blaze/context.py`

---

## 1. The BlazeGraph Data Structure

`BlazeGraph` is the central intermediate representation. Every compilation path -- whether the user writes `with blaze.fuse() as ctx:` (graph API) or wires ops through `FusedProgram.emit()` (composition API, see [Chapter 1 -- Two-API Design](../ch01_architecture/03_two_api_design.md)) -- ultimately produces a `BlazeGraph`.

### 1.1 Core Fields

```python
class BlazeGraph:
    def __init__(self):
        self.nodes: list[OpNode] = []
        self.edges: list[Edge] = []
        self.input_tensors: dict[str, Any] = {}
        self.output_nodes: list[OpNode] = []
        self.external_input_ports: set[tuple[str, str]] = set()
        self.tensor_to_ports: dict[str, list[tuple[str, str]]] = {}
```

| Field | Type | Purpose |
|-------|------|---------|
| `nodes` | `list[OpNode]` | All op instances in **insertion order** -- significant because the kernel codegen module uses this order for phase sequencing, not topological order |
| `edges` | `list[Edge]` | Data dependency edges between op ports |
| `input_tensors` | `dict[str, Any]` | Named external tensors feeding the graph (name to `ExternalTensor`) |
| `output_nodes` | `list[OpNode]` | Terminal nodes (no outgoing edges), computed at graph-build time |
| `external_input_ports` | `set[tuple[str, str]]` | `(node_id, port_name)` pairs connected to external tensors |
| `tensor_to_ports` | `dict[str, list[tuple[str, str]]]` | Reverse mapping: `ExternalTensor.name` to all `(node_id, port_name)` pairs it feeds |

The `tensor_to_ports` mapping enables the CBEngine and BlazeCompiler to trace which CB IDs belong to which external tensors -- critical for shared tensor optimization (see [Section 02 -- CB Engine](./02_cb_engine.md)).

The `external_input_ports` set records which `(node_id, port_name)` pairs are fed by external tensors rather than by graph-internal edges. During CB assignment, a port in this set but *not* in any edge's consumer side receives its own `ext_in` CB allocation.

### 1.2 Edge Indexing

For efficient traversal, `BlazeGraph` lazily builds adjacency indices:

```python
def _build_edge_index(self) -> None:
    self._edges_to_idx: dict[str, list[Edge]] = {}    # consumer.id -> edges
    self._edges_from_idx: dict[str, list[Edge]] = {}   # producer.id -> edges
```

These indices are invalidated on `add_edge()` and rebuilt on-demand. Convenience methods `get_edges_to(node)`, `get_edges_from(node)`, `predecessors(node)`, and `successors(node)` delegate to these indices, providing $O(1)$ lookups after the initial build.

## 2. OpNode

Each node in the graph is an `OpNode`:

```python
@dataclass
class OpNode:
    id: str                    # e.g. "matmul_1", "mcast_2"
    spec: OpSpec               # registered op specification
    grid: Any = None           # CoreRangeSet or grid specifier
    kwargs: dict = field(...)  # user config + runtime annotations
    ct_values: dict = field(...)  # {ct_arg_name: value} from emit()
```

**`id`** is generated by `FusionContext._next_id()`, which appends a monotonically increasing counter to the op name: `"matmul_1"`, `"mcast_1"`, `"gather_1"`, etc.

**`spec`** references an `OpSpec` (see [Chapter 2 -- BlazeOp Base](../ch02_blazeop_hierarchy/01_blazeop_base.md)) that carries the op's port declarations, kernel path, RISC role assignments, and behavioral flags. The fields most relevant to the compilation pipeline are:

| OpSpec field | Consumed by |
|---|---|
| `input_ports`, `output_ports` | CBEngine (port-to-CB mapping), graph validation |
| `internal_cbs` | CBEngine (scratch buffer allocation) |
| `is_inter_core` | SemEngine (determines if semaphores are needed) |
| `preferred_math_fidelity`, `math_approx_mode` | BlazeCompiler (`_get_compute_config()`) |
| `has_init_teardown` | Kernel codegen (lifecycle call emission) |

**`kwargs`** is the workhorse dictionary carrying user-supplied configuration plus pipeline-significant keys:

| Key | Purpose |
|-----|---------|
| `ct_prefix` | Overrides the default CT arg prefix (the op's spec name). Essential for multi-instance ops. |
| `ct_values` | Compile-time arg values captured during `emit()` calls in the composition API. |
| `noop_kernel` | When `True`, kernel codegen emits an empty `DeviceZoneScopedN` block instead of a full phase. |
| `num_tiles_<port>` | CB page count override for double-buffering. |
| `dual_noc` | Controls whether the semaphore engine allocates NOC1 semaphores (for gather ops). |
| `sender` | For mcast ops, the logical core coordinate of the sender (drives mcast grouping). |
| `branch` | For branched ops like `kn_sliced_matmul` with gate/up variants. |
| `_op_flags` | Per-core compile-time flag overrides (e.g., `{flag_name: CoreRangeSet}`). |

## 3. Edge

```python
@dataclass
class Edge:
    producer: OpNode
    producer_port: str      # output port name
    consumer: OpNode
    consumer_port: str      # input port name
    cb_id: Optional[int] = None  # filled in by CB engine later
```

Each edge connects an output port on a producer to an input port on a consumer. The `cb_id` field starts as `None` and is the only mutable state on an edge -- the CBEngine writes it during assignment, and downstream stages (CTArgEngine, BlazeCompiler) read it.

When one output port feeds multiple consumers (fan-out), multiple edges share the same `producer` and `producer_port` but differ in `consumer`. The CB engine detects this and assigns a single CB ID to all edges from that port.

## 4. ExternalTensor and FusionResult

These are the two handle types that flow through the fusion context during graph construction.

### 4.1 ExternalTensor

```python
@dataclass
class ExternalTensor:
    name: str
    tensor: Any = None       # the actual ttnn tensor, if available
    metadata: dict = field(default_factory=dict)
```

`ExternalTensor` is a placeholder for data from outside the graph (weights, activations, etc.). At graph construction time, the actual tensor may or may not be available -- the `name` is what matters for wiring. At compile time, the `BlazeCompiler.compile()` method receives a `tensors: dict` that maps these names to real `ttnn.Tensor` objects.

### 4.2 FusionResult

```python
@dataclass
class FusionResult:
    node: OpNode
    port_name: str  # which output port this result comes from
```

`FusionResult` is the handle returned by every op call inside a `fuse()` block. Passing a `FusionResult` as an input to a subsequent op call creates an `Edge` in the graph:

```python
with blaze.fuse() as ctx:
    x = blaze.matmul({"in0": ext_act, "in1": ext_wt}, grid=grid)
    y = blaze.gelu({"input": x}, grid=grid)
    # Edge: matmul.out -> gelu.input
```

## 5. The FusionContext

`FusionContext` (defined in `blaze/context.py`) is the context manager behind `blaze.fuse()`. It captures op calls and assembles them into a `BlazeGraph`.

### 5.1 Module-Level Active Context

```python
_active_context: Optional[FusionContext] = None
```

A module-level variable holds the currently active context. This is a simple global (not `threading.local`) -- single-threaded fusion is the design assumption. When user code calls `blaze.matmul(...)` inside a `with blaze.fuse():` block, the dispatched `_op_call()` function retrieves `_active_context`. If no context is active, a `RuntimeError` is raised.

### 5.2 Context Manager Protocol

```python
class FusionContext:
    def __enter__(self) -> FusionContext:
        _set_active_context(self)
        return self

    def __exit__(self, exc_type, exc_val, exc_tb) -> None:
        _clear_active_context()
        if exc_type is None:
            self.graph = self._build_graph()
            errors = self.graph.validate()
            if errors:
                raise ValueError(...)
        return None  # don't suppress exceptions
```

On `__enter__`, the context installs itself. On `__exit__`, it:

1. Clears itself from the module-level slot.
2. If no exception occurred, builds the graph via `_build_graph()`.
3. Validates the graph and raises if errors are found.
4. Stores the graph as `self.graph` for downstream compilation.

### 5.3 `add_op()` -- Recording Operations

```python
def add_op(self, op_name, inputs, grid=None, **kwargs) -> FusionResult:
    spec = get_op_spec(op_name)
    node_id = self._next_id(op_name)  # e.g. "matmul_1", "matmul_2"
    node = OpNode(id=node_id, spec=spec, grid=grid, kwargs=kwargs)
    self._nodes.append(node)

    for port_name, source in inputs.items():
        if isinstance(source, FusionResult):
            edge = Edge(producer=source.node, producer_port=source.port_name,
                        consumer=node, consumer_port=port_name)
            self._edges.append(edge)
        elif isinstance(source, ExternalTensor):
            self._input_tensors[source.name] = source
            self._external_input_ports.add((node_id, port_name))
            self._tensor_to_ports.setdefault(source.name, []).append((node_id, port_name))

    return FusionResult(node=node, port_name=spec.output_ports[0].name)
```

The method:

1. Looks up the `OpSpec` from the global registry (see [Chapter 2 -- Auto-Discovery](../ch02_blazeop_hierarchy/04_auto_discovery_and_registration.md)).
2. Generates a unique node ID via `_next_id()`.
3. Creates an `OpNode` and appends it to the internal node list.
4. For each input, either creates an `Edge` (when the source is a `FusionResult`) or registers an external tensor input.
5. Returns a `FusionResult` pointing to the first output port, enabling chaining.

### 5.4 `_build_graph()` -- Assembly

```python
def _build_graph(self) -> BlazeGraph:
    graph = BlazeGraph()
    for node in self._nodes:
        graph.add_node(node)
    for edge in self._edges:
        graph.add_edge(edge)
    graph.input_tensors = dict(self._input_tensors)
    graph.external_input_ports = set(self._external_input_ports)
    graph.tensor_to_ports = {k: list(v) for k, v in self._tensor_to_ports.items()}
    # Terminal nodes: nodes with no outgoing edges
    producer_ids = {e.producer.id for e in self._edges}
    all_ids = {n.id for n in self._nodes}
    terminal_ids = all_ids - producer_ids
    graph.output_nodes = [n for n in self._nodes if n.id in terminal_ids]
    return graph
```

Terminal nodes (those that never appear as a producer in any edge) are identified as output nodes. These will receive `ext_out` CB assignments during the CB engine pass.

## 6. Topological Ordering

`BlazeGraph.topological_order()` implements Kahn's algorithm:

```python
def topological_order(self) -> list[OpNode]:
    in_degree = {n.id: 0 for n in self.nodes}
    adj = {n.id: [] for n in self.nodes}
    for e in self.edges:
        adj[e.producer.id].append(e.consumer.id)
        in_degree[e.consumer.id] += 1

    queue = [nid for nid, deg in in_degree.items() if deg == 0]
    queue.sort()
    result = []
    while queue:
        nid = queue.pop(0)
        result.append(node_map[nid])
        for succ_id in sorted(adj[nid]):
            in_degree[succ_id] -= 1
            if in_degree[succ_id] == 0:
                queue.append(succ_id)
        queue.sort()

    if len(result) != len(self.nodes):
        raise ValueError("Cycle detected in graph")
    return result
```

Key design choices:

1. **Determinism**: The queue is sorted at each step, ensuring identical ordering across runs for the same graph structure. This is critical because CB IDs, semaphore IDs, and CT arg tuples all depend on traversal order.
2. **Cycle detection**: If the result length differs from the node count, a cycle exists and a `ValueError` is raised.
3. **Complexity**: $O(V^2 \log V + E)$ due to the per-pop sorting of the queue. This is intentional -- determinism is prioritized over asymptotic efficiency, and graph sizes are small (typically 3--15 nodes).

Topological order is used by:
- **CBEngine**: walks nodes to assign sequential CB IDs.
- **SemEngine**: uses topological position to identify sync points.
- **CTArgEngine**: iterates in topological order for deterministic arg generation.
- **BlazeCompiler**: uses topological order for IO tensor ordering.

Note: **kernel codegen** uses **insertion order** (not topological) for phase emission -- the user's emit sequence defines the execution flow.

## 7. Graph Validation

`BlazeGraph.validate()` checks three invariants and returns a list of error strings:

1. **No duplicate node IDs**: Each `OpNode.id` must be unique.
2. **Valid port references**: Every edge's `producer_port` must exist in the producer's `spec.output_ports`, and every edge's `consumer_port` must exist in the consumer's `spec.input_ports`.
3. **No cycles**: Validated via `topological_order()` (which raises on cycle detection).

Validation is called automatically by `FusionContext.__exit__()` and will raise a `ValueError` if any errors are found.

## 8. The `compile_engines()` Orchestrator

After graph construction, the `compile_engines()` function drives the three-engine pass:

```python
def compile_engines(graph: BlazeGraph, user_args=None, grid_config=None) -> EngineResult:
    result = EngineResult()

    # 1. CB Engine
    cb_engine = CBEngine()
    result.cb_assignments = cb_engine.assign(graph)
    for key, a in result.cb_assignments.items():
        if a.node_id:
            result.cb_by_node.setdefault(a.node_id, {})[a.port_name] = a.cb_id
    # Also populate consumer-side CB mappings from edges
    for edge in graph.edges:
        if edge.cb_id is not None:
            result.cb_by_node.setdefault(edge.consumer.id, {})[edge.consumer_port] = edge.cb_id

    # 2. Sem Engine
    sem_engine = SemEngine()
    result.sem_assignments = sem_engine.assign(graph)
    for sa in result.sem_assignments:
        result.sem_by_node.setdefault(sa.node_id, {})[sa.protocol] = sa.sem_id

    # 3. CT Arg Engine (best-effort)
    ct_engine = CTArgEngine()
    try:
        result.ct_tuples = ct_engine.generate_tuples(
            graph, result.cb_by_node, result.sem_by_node,
            user_overrides=user_args or {},
        )
    except ValueError:
        pass  # Name collisions expected for multi-instance ops without ct_prefix

    return result
```

### 8.1 Engine Execution Order

The engines execute in strict dependency order:

$$\text{CBEngine} \rightarrow \text{SemEngine} \rightarrow \text{CTArgEngine}$$

- **CBEngine** runs first because it assigns `cb_id` values that the CTArgEngine needs for `source="cb"` args.
- **SemEngine** runs second because it assigns `sem_id` values that the CTArgEngine needs for `source="sem"` args.
- **CTArgEngine** runs last, consuming both CB and semaphore mappings.

### 8.2 EngineResult

```python
@dataclass
class EngineResult:
    cb_assignments: dict = field(default_factory=dict)   # key -> CBAssignment
    cb_by_node: dict = field(default_factory=dict)       # {node_id: {port: cb_id}}
    sem_assignments: list = field(default_factory=list)   # List[SemAssignment]
    sem_by_node: dict = field(default_factory=dict)       # {node_id: {source_key: sem_id}}
    ct_tuples: dict = field(default_factory=dict)         # {risc: [(name, value), ...]}
```

The `cb_by_node` and `sem_by_node` dictionaries are convenience views: they map `node_id` to per-port CB IDs and per-protocol semaphore IDs, making it easy for the CTArgEngine to resolve `source="cb"` and `source="sem"` args.

### 8.3 Best-Effort CT Arg Generation

The CT arg generation is wrapped in a `try/except ValueError`. This is intentional: graphs with duplicate op types that have not been given unique `ct_prefix` values will produce name collisions. The orchestrator handles this gracefully by leaving `ct_tuples` empty. Callers that need CT args should set `ct_prefix` per node.

Note: this function is the standalone orchestrator for the graph-API path. The `BlazeCompiler` runs these engines individually with additional device-specific context, as detailed in [Section 06](./06_blaze_compiler.md).

### 8.4 Node Prefix Utilities

Two utility functions support CT arg prefixing:

```python
def _node_prefix(node: OpNode) -> str:
    return node.kwargs.get("ct_prefix", node.spec.name)

def _prefix_matches(node_prefix: str, target: str) -> bool:
    return (
        node_prefix == target
        or node_prefix.startswith(target + "__")
        or node_prefix.startswith(target + ".")
    )
```

`_prefix_matches` handles nested child-op prefixing (`__` delimiter, see [Chapter 2 -- Writing a FusedOp](../ch02_blazeop_hierarchy/06_writing_a_fused_op.md)) and branch suffixes (`.` delimiter, e.g., `gu.gate`/`gu.up` for `kn_sliced_matmul`). These are used by `BlazeCompiler._tag_noop_nodes()` and other internal machinery.

## 9. Relationship to the Two Compilation Paths

The `BlazeGraph` produced by either API is consumed by the `BlazeCompiler` (see [Section 06](./06_blaze_compiler.md)), which dispatches to one of two paths:

1. **Engine path** (`_compile_for_device()`): For multi-node graphs. Runs CBEngine, SemEngine, CTArgEngine, then builds `UnifiedKernelDescriptor` and `ProgramDescriptor`.
2. **Fused op path** (`_compile_fused_op()`): For single-node graphs whose op has a registered `compose_fn`. Delegates to the `FusedProgram` composition API.

Both paths produce a `CompiledProgram` that wraps a `ProgramDescriptor` and an ordered `io_tensors` list for `ttnn.generic_op()`.

## 10. Worked Example

```python
import blaze

weights = blaze.ExternalTensor("weights")
activations = blaze.ExternalTensor("activations")

with blaze.fuse() as ctx:
    mm_out = blaze.matmul({"in0": activations, "in1": weights}, grid=[(0,0)])
    final = blaze.mcast({"input": mm_out}, grid=[(0,0), (0,1)])

graph = ctx.graph
```

After the `with` block exits:

1. `graph.nodes` contains `[OpNode(id="matmul_1", ...), OpNode(id="mcast_1", ...)]` in insertion order.
2. `graph.edges` contains one `Edge(producer=matmul_1, producer_port="out", consumer=mcast_1, consumer_port="input")`.
3. `graph.external_input_ports` = `{("matmul_1", "in0"), ("matmul_1", "in1")}`.
4. `graph.tensor_to_ports` = `{"activations": [("matmul_1", "in0")], "weights": [("matmul_1", "in1")]}`.
5. `graph.output_nodes` = `[mcast_1]` (only node not appearing as a producer).
6. `graph.topological_order()` returns `[matmul_1, mcast_1]` (matmul has in-degree 0, mcast depends on it).

This graph is now ready to be consumed by the CB engine (next section).
