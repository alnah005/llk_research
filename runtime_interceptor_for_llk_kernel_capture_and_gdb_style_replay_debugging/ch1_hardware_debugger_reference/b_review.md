# Agent B Review: Chapter 1 — Pass 1

1. **File:** `01_gpu_debugger_architectures.md`, **~line 272 and surrounding context (Section 5.1).** The RISC_DBG_CNTL_0 description states `risc_sel` maps as "0=BRISC, 1-3=TRISC0-2, 4=NCRISC" (matching the source comment in `ckernel_riscv_debug.h`), but the `rvdbg_risc_sel` enum presented immediately after encodes TRISC0=0x00000000 (risc_sel=0), TRISC1=0x00020000 (risc_sel=1), TRISC2=0x00040000 (risc_sel=2), TRISC3=0x00060000 (risc_sel=3) -- with no BRISC or NCRISC entries. This means enum value TRISC0 occupies the same bit pattern the comment assigns to BRISC, and the enum has no way to target BRISC or NCRISC at all. The chapter presents both without acknowledging the conflict. An implementer building a GDB stub to target BRISC would find no `rvdbg_risc_sel` value for it and would be uncertain whether `TRISC0` actually targets TRISC0 or BRISC. **Fix:** Add an explicit note that the `rvdbg_risc_sel` enum covers only the four compute processors (TRISC0-3) in Quasar's NEO model, that the RISC_DBG_CNTL_0 comment describing "0=BRISC...4=NCRISC" may be inherited from a different architecture or apply to a different access mode, and that targeting BRISC/NCRISC via this interface is an open question requiring hardware validation.

2. **File:** `01_gpu_debugger_architectures.md`, **~line 338 (Section 5.3).** The text states the debug array is read via `dbg_dump_read(thread, array_id, addr)`. This function does not exist in the codebase. The actual functions are `dbg_dump_array_rd_cmd(thread, array_id, addr)` in `tensix_functions.h` (low-level register-poke API) and `dbg_get_array_row(array_id, row_addr, rd_data)` in `ckernel_debug.h` (higher-level function that handles the SRCA-via-DEST workaround). An implementer searching for `dbg_dump_read` would find nothing. **Fix:** Replace `dbg_dump_read(thread, array_id, addr)` with `dbg_dump_array_rd_cmd(thread, array_id, addr)` (from `tensix_functions.h`) and note that `ckernel_debug.h` provides the higher-level `dbg_get_array_row(array_id, row_addr, rd_data)` which handles the SRCA indirection.

3. **File:** `01_gpu_debugger_architectures.md`, **~line 292 (Section 5.1).** The text states "Hardware watchpoints are available via `HW_WCHPT0` through `HW_WCHPT7` (8 per core)." The watchpoints are set via `riscv_dbg_wr(risc_sel, rvdbg_reg::HW_WCHPT0, val)`, meaning each RISC-V processor has its own independent set of 8 watchpoints. The correct unit is "8 per RISC," not "8 per core." File `02_lessons_and_requirements_for_tenstorrent.md` line 178 correctly says "8 per RISC," creating an internal contradiction. An implementer reading "8 per core" would incorrectly assume all five RISCs share a single pool of 8 watchpoints. **Fix:** Change "8 per core" to "8 per RISC" on line 292.

# Agent B Review: Chapter 1 — Pass 2

1. **File:** `01_gpu_debugger_architectures.md`, **line 135 (Section 3.1).** The text states AMD GPUs execute in wavefronts of "64 threads on RDNA, 32 or 64 on CDNA." This is reversed. RDNA introduced wave32 as its native execution mode (with wave64 available as an emulated mode using two wave32 passes), while CDNA architectures (MI100/MI200/MI300) use wave64 natively. The correct statement is "32 or 64 on RDNA, 64 on CDNA." An implementer mapping AMD wavefront concepts to Tenstorrent execution units would get the wrong size association for each architecture family. **Fix:** Change "64 threads on RDNA, 32 or 64 on CDNA" to "32 or 64 threads on RDNA (wave32 native), 64 on CDNA."

2. **File:** `01_gpu_debugger_architectures.md`, **line 117 and line 366 (Sections 2 and 6.1).** The Tensix focus formula lists exactly five processors: BRISC, NCRISC, TRISC0, TRISC1, TRISC2. Section 6.1 states "A Tensix core runs five independent RISC-V processors (BRISC, NCRISC, TRISC0, TRISC1, TRISC2)." However, Quasar has six RISC-V processors per core (BRISC, NCRISC, TRISC0-TRISC3), as confirmed by `NUM_TRISC_CORES 4` in `tt-metal/tt_metal/hw/inc/internal/tt-2xx/quasar/dev_mem_map.h` and the `rvdbg_risc_sel` enum listing TRISC0-TRISC3. An implementer building a Quasar debug focus model from the formula would omit TRISC3 entirely. **Fix:** Qualify the formula and the "five RISC" statement as Wormhole/Blackhole-specific, and note that Quasar adds TRISC3 for six processors total, or parameterize the formula (e.g., "TRISC0..TRISC_{N-1}" where N=3 on WH/BH, N=4 on Quasar).
