# Wrapping vs. Extending vs. Replacing: A Per-Component Integration Analysis

This section performs a systematic analysis of every component in TT-Blaze's infrastructure, classifying its relationship to TensorAdapter as WRAP, EXTEND, or UNCHANGED. The classification is not arbitrary -- each decision is grounded in the component's API stability, the nature of TensorAdapter's interaction with it, and the "compose, do not replace" constraint established in [Chapter 1, File 04](../ch01_the_gap/04_abstraction_goals_and_scope.md). The section concludes with the "adapter as emit() caller" pattern that unifies all integration points.

**What you will learn:**

- The precise WRAP / EXTEND / UNCHANGED classification for each Blaze component, with the rationale for each decision and rejected alternatives
- What "WRAP" means concretely: TensorAdapter creates input parameters for the component, then calls the component's existing API unchanged
- What "EXTEND" means concretely: the component gains new metadata fields or optional parameters, but its core algorithm is unmodified
- Why "REPLACE" is never the correct answer for any component (the cost/benefit analysis)
- The `[A]`/`[B]` annotation pattern showing which code belongs to the adapter vs. existing Blaze (cherry-picked from V3)
- The four-level override hierarchy for user_args (cherry-picked from V4)
- The "adapter as emit() caller" pattern and why it works

---

## Classification Framework

Before examining each component, we define the three classifications precisely:

| Classification | Definition | API Impact | Testing Impact |
|---------------|-----------|-----------|---------------|
| **WRAP** | TensorAdapter calls the component's existing public API with computed parameters. The component's source code is not modified. | Zero. Existing tests pass unchanged. | New tests verify adapter's parameter computation. |
| **EXTEND** | The component gains new optional fields or methods that TensorAdapter populates. Existing behavior is unaffected when the new fields are absent. | Additive only. All existing code paths unaffected. | New tests for extended fields; existing tests unchanged. |
| **UNCHANGED** | TensorAdapter does not interact with this component. It operates on a different concern axis (e.g., inter-core topology vs. tensor shapes). | None. | None. |

The critical constraint: **no classification is REPLACE.** Replacing any component would mean reimplementing its behavior, which violates the compose-not-replace principle and risks breaking production models that depend on the component's exact semantics.

## Component-by-Component Analysis

### CBHandle (blaze/cb_handle.py) -- WRAP

**Current state:** `CBHandle` is a dataclass with 11 fields, `eq=False` (identity-based equality), a `buffer_address()` method, and graph metadata fields (`_node_id`, `_port_name`).

**TensorAdapter interaction:** The CBBackend layer creates `CBHandle` instances by calling `FusedProgram.cb_from_tensor()` or `FusedProgram.cb_scratch()`. TensorAdapter computes the parameters passed to these methods; it does not construct `CBHandle` directly.

**Why WRAP, not EXTEND:** Adding a `shape_descriptor` field to CBHandle was considered and rejected for three reasons: (1) lifetime mismatch -- CBHandle lives through execution, ShapeDescriptor only during graph construction; (2) circular dependency -- both carry `page_size` and `data_format`; (3) API contract -- every existing `emit()` method constructs CBHandles and would need updating. Instead, `TypedHandle` carries both as siblings (see [File 02](./02_shape_metadata_system.md)).

**What TensorAdapter computes for CBHandle:**

| CBHandle Field | TensorAdapter Derivation |
|---------------|------------------------|
| `num_pages` | From `ShapeDescriptor.tile_grid_shape` and op blocking requirements |
| `page_size` | From `ShapeDescriptor.page_size` (using `tile.get_tile_size()` for BFP) |
| `data_format` | From `ShapeDescriptor.data_format` (after format negotiation) |
| `tile_desc` | Constructed from `ShapeDescriptor.tile_shape` |
| `core_ranges` | From weight shard spec or grid planner |
| `access_mode` | Inferred from op_hint: `"matmul.in1"` -> DIRECT_ADDRESS, others -> FIFO |
| `backing_tensor` | The original TTNN tensor (for tensor-backed CBs) |

### FusedProgram (blaze/fused_program.py) -- WRAP

**Current state:** FusedProgram is the composition context (>800 lines) managing CB allocation, CT arg wiring, semaphore creation, multi-phase reconfig, shadow graph recording, and output wiring. It provides methods including `cb_from_tensor()`, `cb_scratch()`, `cb_alias()`, `ncrisc_ct_args()`, `trisc_ct_args()`, `output()`, `wire_output()`, `build()`, and `reconfig()`.

**TensorAdapter interaction:** The CBBackend layer calls FusedProgram's existing methods. The call sequence generated by TensorAdapter is identical to a hand-written composition.

**Why WRAP, not EXTEND:** Extending FusedProgram with shape-aware methods (e.g., `f.cb_from_shape(desc)`) was rejected because: (1) Single responsibility -- shape reasoning is ShapeEngine's domain; (2) Production stability -- FusedProgram is refined across hundreds of production kernels (DeepSeek V3, Llama, Qwen); (3) The methods already accept the right parameters -- `cb_scratch(name, num_pages, core_ranges, data_format, tile, page_size)` already accepts every field TensorAdapter needs.

**Why WRAP, not REPLACE:** Replacing FusedProgram would mean reimplementing disjoint-core CB sharing (`_format_key()` dedup), scratch-mapped allocation, multi-phase reconfiguration via `CircularBufferIdManager`, `OverlappedView` management for shared L1 tensors, shadow graph recording, and `BlazeProgram` integration. Each is complex, hardware-specific, and battle-tested.

> **Note:** The plan spec (line 136) labels FusedProgram as "EXTEND" but describes it as "gains no new methods." Since the adapter calls existing methods with computed parameters and adds no new methods or fields, WRAP more accurately reflects the described behavior. The EXTEND label in the plan appears to be a misnomer.

**Concrete call pattern using `[A]`/`[B]` annotations:**

```python
# [A] = adapter-generated code    [B] = unchanged Blaze code
def allocate_scratch_cb(self, f, desc, name):
    return f.cb_scratch(                  # [B] existing FusedProgram method
        name=name,                         # [A] adapter-computed
        num_pages=self._num_pages(desc),   # [A] from ShapeDescriptor
        core_ranges=self._cores(desc),     # [A] from grid planner
        data_format=desc.data_format,      # [A] from ShapeDescriptor
        tile=self._tile(desc.tile_shape),  # [A] from ShapeDescriptor
        page_size=desc.page_size,          # [A] from ShapeDescriptor
    )
```

Every `[B]` line is an existing FusedProgram method. Every `[A]` line is a parameter computed by TensorAdapter from ShapeDescriptor fields.

### CBEngine (blaze/cb_engine.py) -- WRAP

**Current state:** `CBEngine.assign()` walks a `BlazeGraph` in topological order, allocates sequential CB IDs, handles fan-out via `_union_grids()`, tracks lifetimes, and supports `compact_cb_ids()` for interval coloring. It reads `TensorPort.dtype` and `TensorPort.tile_shape` from the graph.

**TensorAdapter interaction:** TensorAdapter enriches the `TensorPort` metadata on graph `OpSpec` entries with the correct `dtype` and `tile_shape` values before the graph is passed to CBEngine. The engine then computes page sizes from these fields using its existing `_tile_page_size()` logic.

**Why WRAP, not REPLACE:** CBEngine handles: shared tensor CB reuse, fan-out grid union, CircularBufferIdManager integration for multi-phase programs, CB limit enforcement (warnings at 56/64, hard errors above 64), and lifetime tracking for compaction. All of this continues unchanged. TensorAdapter's enrichment of TensorPort metadata means CBEngine makes better page_size decisions, but the engine's code does not change.

---

### SemEngine (blaze/sem_engine.py) -- UNCHANGED

Semaphore assignment depends entirely on inter-core topology (which ops are gather/mcast, how they are connected, which cores are senders/receivers). Tensor shapes have no influence on semaphore protocols or indices. A `[1, 4096]` tensor and a `[32, 4096]` tensor require the same semaphore protocol for a mcast operation -- the semaphore coordinates data movement timing, not data shape.

---

### CTArgEngine (blaze/ct_args.py) -- WRAP

**Current state:** `CTArgEngine.generate_tuples()` walks the graph, looks up each op's `CTArgSchema`, resolves values from `cb_assignments`, `sem_assignments`, `grid_context`, or `user_overrides`, validates collision-freedom, and groups results by RISC.

**TensorAdapter interaction:** TensorAdapter computes the `user_overrides` dict that CTArgEngine consumes. Specifically, it derives shape-dependent values from `ShapeDescriptor`:

```python
# PROPOSED (illustrative, not prescriptive) -- CT arg derivation
def derive_user_args(self, graph, shape_map):
    """Compute shape-dependent CT args from ShapeDescriptors."""
    args = {}
    for node in graph.topological_order():
        node_args = {}
        if node.spec.name in ("matmul", "matmul_cb"):
            in0_desc = shape_map.get(f"{node.id}.in0")
            in1_desc = shape_map.get(f"{node.id}.in1")
            if in0_desc and in1_desc:
                k = in0_desc.tile_grid_shape[-1]         # [A] from ShapeDescriptor
                node_args["k_num_tiles"] = k              # [A] adapter-computed
                node_args["out_w_per_core"] = in1_desc.total_tiles // k  # [A]
        elif node.spec.name in ("rmsnorm", "padded_rmsnorm"):
            input_desc = shape_map.get(f"{node.id}.input")
            if input_desc:
                node_args["width"] = input_desc.tile_grid_shape[-1]  # [A]
        args[node.id] = node_args
    return args   # -> passed to CTArgEngine.generate_tuples(user_overrides=...)  [B]
```

**Why WRAP, not EXTEND:** CTArgEngine already supports `user_overrides` as a first-class parameter. TensorAdapter simply populates it with shape-derived values instead of requiring the developer to compute them manually.

**Why WRAP, not REPLACE:** CTArgEngine handles prefix computation with `ct_prefix` kwargs, shared-prefix deduplication, multi-source resolution (`"cb"`, `"sem"`, `"grid"`, `"derived"`), per-RISC grouping, and `per_core` arg types. All orthogonal to shape inference.

### BlazeGraph / Graph IR (blaze/graph.py) -- EXTEND

**Current state:** `BlazeGraph` is a DAG of `OpNode`s connected by `Edge`s. `Edge` has fields `producer`, `producer_port`, `consumer`, `consumer_port`, and `cb_id`.

**TensorAdapter interaction:** TensorAdapter attaches `ShapeDescriptor` metadata to graph edges. This is the one component where EXTEND is the correct classification.

```python
# PROPOSED (illustrative, not prescriptive) -- Edge extension
@dataclass
class Edge:
    producer: OpNode
    producer_port: str
    consumer: OpNode
    consumer_port: str
    cb_id: Optional[int] = None
    # --- NEW: TensorAdapter metadata (optional, None when not used) ---
    shape_descriptor: Optional[ShapeDescriptor] = None
```

**Why EXTEND, not WRAP:** TensorAdapter needs to *attach data* to the graph so that shape information propagates through the graph structure, visualization tools display it, and the L1 budget validator can compute per-node memory usage.

**Why not a separate metadata dict?** A `dict[edge_key, ShapeDescriptor]` was considered. Rejected because: (1) Sync risk -- if edges are added/removed, the dict must be updated in lockstep; (2) Serialization -- graph export naturally includes edge fields; (3) Ergonomics -- `edge.shape_descriptor.logical_shape` is clearer than `shape_map[(edge.producer.id, edge.producer_port)].logical_shape`.

---

### BlazeCompiler (blaze/compiler.py) -- WRAP

**Current state:** `BlazeCompiler.compile()` accepts a `BlazeGraph`, a tensor dict, an output tensor, `user_args`, and optional parameters.

**TensorAdapter interaction:** TensorAdapter calls `compile()` with pre-computed `user_args`. The compiler's internal logic -- per-device iteration, `MeshProgramDescriptor` assembly, global semaphore creation, fused op compilation, auto-export -- is unchanged.

---

### BlazeOp Hierarchy (blaze/blaze_op.py) -- UNCHANGED

TensorAdapter calls `emit()` on op classes with pre-computed handles. The op's `emit()` executes exactly as if called by hand. The `BlazeOp` base class, port descriptors, `CompileTimeArg` declarations, and `register()` mechanism are all untouched.

**Why not add adapt() to BlazeOp?** (Rejected alternative from V3.) Adding a shape-aware method to the op base class was rejected because: (1) shape expectations vary by invocation, not by op type (a matmul operates on any shape); (2) it would modify the registration mechanism affecting every op; (3) per-invocation metadata belongs on graph edges, not on op specs.

---

### CircularBufferIdManager (blaze/cb_reconfig.py) -- WRAP

**Current state:** `CircularBufferIdManager` manages CB ID allocation for multi-phase programs, enabling format-aware reuse of CB IDs across phases via `create_context()` and `_allocate_id()`.

**TensorAdapter interaction:** TensorAdapter passes the manager to CBEngine/FusedProgram for multi-phase programs. The manager's allocation algorithm is unchanged.

---

### BlazeProgram (blaze/program.py) -- UNCHANGED

FusedProgram mediates all interactions with `BlazeProgram` for `ProgramDescriptor` construction. TensorAdapter has no direct contact with this component.

## The Four-Level Override Hierarchy

When TensorAdapter derives user_args from ShapeDescriptor, the values interact with existing override mechanisms. The precedence hierarchy (highest to lowest):

| Priority | Source | Example | Who Sets It |
|----------|--------|---------|-------------|
| **1 (highest)** | Explicit `user_args` on `compile()` | `user_args={"k_num_tiles": 64}` | Developer (override escape hatch) |
| **2** | `adapt()` Level 1 overrides | `adapt(tensor, tile_shape=(1, 32))` | Developer (performance tuning) |
| **3** | Adapter defaults from ShapeDescriptor | `k_num_tiles = desc.tile_grid_shape[-1]` | TensorAdapter (automatic) |
| **4 (lowest)** | Blaze op defaults | `DEFAULT_TILE_SHAPE = (32, 32)` | Blaze infrastructure (built-in) |

This implements Principle P4 (Defaults with Per-Decision Overrides): every automated decision can be overridden at any level without disabling the others. A developer who disagrees with the auto-derived `k_num_tiles` can pass it explicitly via `user_args` without also needing to manually derive `out_w_per_core`.

## The Adapter as emit() Caller: The Unifying Pattern

The integration analysis reveals a consistent pattern across all WRAP components:

```
TensorAdapter                              Existing Blaze
--------------                              ---------------

  For each op in the graph:
  1. [A] Compute ShapeDescriptor(s)        (ShapeEngine)
     from input tensors + op_hint

  2. [A] Derive CB parameters:             (CBBackend)
     - num_pages from shape + blocking
     - page_size from tile + format
     - core_ranges from grid + shard spec

  3. [A] Derive CT arg values:
     - k_num_tiles from activation grid
     - out_w_per_core from weight grid
     - width from input grid

  4. [B] Call the op's emit():             --> Matmul.emit(f, in0, in1, prefix="mm")
                                           --> EltwiseMul.emit(f, gate, up, prefix="mul")
                                           --> Mcast.emit(f, data, prefix="mcast")

  5. [A] Record output ShapeDescriptor     (ShapeTracker)
     for downstream ops
```

Steps 1-3 and 5 are `[A]` (adapter-generated). Step 4 is `[B]` (existing Blaze code, unchanged).

### Why This Pattern Works

This pattern works because of a fortunate property of the current Blaze architecture: `emit()` methods are **pure functions of their parameters**. Given the same `FusedProgram`, the same input `CBHandle`s, and the same keyword arguments, `emit()` produces the same output `CBHandle` and the same CT args. There is no hidden state, no global configuration, and no side channels.

This means TensorAdapter can compute the parameters offline (during graph analysis), validate them (during the validation step), and then feed them to `emit()` with confidence that the result matches what a human developer would have produced.

### What If emit() Needs to Change?

If a future op requires information that TensorAdapter has but `emit()` cannot accept:

1. Add the information to `TypedHandle` (which the adapter already returns).
2. If the op needs it during emission, add an optional keyword argument to `emit()` (backward-compatible extension of the op, not of TensorAdapter).
3. If the op needs shape metadata on the graph, use the `Edge.shape_descriptor` extension.

The adapter never modifies `emit()` signatures. It computes inputs and reads outputs.

### Concrete MatmulAdapter Example

```python
# PROPOSED (illustrative, not prescriptive) -- MatmulAdapter delegation
class MatmulAdapter:
    """Wraps Matmul.emit() with automatic shape tracking."""

    def emit(self, f, activation, weights, prefix="matmul"):
        # [A] Compute shape descriptors
        act_desc = self.shape_engine.full_analysis(activation, "matmul.in0")
        wt_desc = self.shape_engine.full_analysis(weights, "matmul.in1")

        # [A] Allocate CBs via CBBackend
        in0 = self.cb_backend.allocate(f, act_desc)
        in1 = self.cb_backend.allocate_direct_address(f, wt_desc)

        # [A] Derive CT args
        k = act_desc.tile_grid_shape[-1]
        out_w = wt_desc.total_tiles // k

        # [B] Delegate to existing Matmul.emit() -- UNCHANGED
        out_handle = Matmul.emit(f, in0, in1, prefix=prefix)

        # [A] Track output shape for downstream ops
        out_desc = self.shape_engine.infer_matmul_output(act_desc, wt_desc)
        self.tracker.record(out_handle, out_desc)
        return TypedHandle(handle=out_handle, shape=out_desc)
```

## Summary: Integration Classification Table

| Component | File | Classification | TensorAdapter's Role | Rationale |
|-----------|------|---------------|---------------------|-----------|
| **CBHandle** | `cb_handle.py` | **WRAP** | Creates via FusedProgram methods with computed parameters | Wrapping preserves API; TypedHandle adds metadata externally |
| **FusedProgram** | `fused_program.py` | **WRAP** | Calls `cb_from_tensor()`, `cb_scratch()`, CT arg methods, `output()`, `build()` | All needed parameters are already method arguments |
| **CBEngine** | `cb_engine.py` | **WRAP** | Enriches TensorPort metadata on graph before engine runs | Engine algorithm unchanged; inputs more complete |
| **SemEngine** | `sem_engine.py` | **UNCHANGED** | No interaction | Semaphores depend on topology, not tensor shapes |
| **CTArgEngine** | `ct_args.py` | **WRAP** | Computes `user_overrides` dict with shape-derived values | Engine already accepts `user_overrides` parameter |
| **BlazeGraph** | `graph.py` | **EXTEND** | Attaches ShapeDescriptor to Edge (optional, additive field) | Metadata must live on graph for propagation + visualization |
| **BlazeCompiler** | `compiler.py` | **WRAP** | Calls `compile()` with pre-computed `user_args` | `user_args` parameter is the designed injection point |
| **BlazeOp hierarchy** | `blaze_op.py` | **UNCHANGED** | Calls `emit()` with computed parameters | Ops are unchanged; adapter computes their inputs |
| **CircularBufferIdManager** | `cb_reconfig.py` | **WRAP** | Passes to CBEngine/FusedProgram for multi-phase programs | Manager's allocation algorithm unchanged |
| **BlazeProgram** | `program.py` | **UNCHANGED** | No direct interaction | FusedProgram mediates all BlazeProgram interactions |

**Score: 6 WRAP, 1 EXTEND, 3 UNCHANGED, 0 REPLACE.**

The overwhelmingly WRAP classification confirms that TT-Blaze's existing architecture has good separation of concerns and well-designed injection points (`user_args`, `user_overrides`, method parameters). TensorAdapter is a builder that fills in the blanks, not a replacement that rewrites the plumbing.

## Key Takeaways

- **WRAP is the dominant pattern** (6 of 10 components): TensorAdapter computes parameters and calls existing APIs. This is the lowest-risk integration strategy and enables incremental adoption without modifying production kernels.
- **EXTEND applies only to BlazeGraph** (Edge gains an optional `shape_descriptor` field). This is the minimum modification needed to propagate shape metadata through the graph structure. The extension is additive and backward-compatible.
- **UNCHANGED components** (SemEngine, BlazeOp hierarchy, BlazeProgram) operate on concern axes orthogonal to tensor shapes.
- **REPLACE is never correct** for any component. The existing infrastructure represents complex, hardware-specific, battle-tested logic that would be wasteful and error-prone to reimplement.
- The **`[A]`/`[B]` annotation pattern** makes the boundary between adapter code and existing Blaze code visible at a glance in every integration example.
- The **four-level override hierarchy** (user_args > adapt() overrides > adapter defaults > Blaze defaults) implements Principle P4: every automated decision can be overridden without disabling the others.
- The **"adapter as emit() caller"** pattern unifies all integration points: TensorAdapter computes parameters (steps 1-3), delegates to existing emit() methods (step 4), and records output metadata (step 5). The kernel binary produced is identical to hand-written code.
