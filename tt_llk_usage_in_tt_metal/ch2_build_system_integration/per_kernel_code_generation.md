# Per-Kernel Code Generation

When a compute kernel is submitted for JIT compilation, TT-Metal does not compile the user's `.cpp` file directly. Instead, it generates four separate source files -- one for each TRISC processor plus an SFPU isolation file -- and a descriptor header that configures LLK with the correct data formats and tile dimensions. This section traces that code generation pipeline.

## Splitting a Kernel into Four TRISC Source Files

The function `jit_build_genfiles_triscs_src()` in `tt_metal/jit_build/genfiles.cpp` (lines 171-247) is responsible for generating the per-TRISC source files:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 171-181
void jit_build_genfiles_triscs_src(
    const JitBuildEnv& env, const JitBuildSettings& settings, const KernelSource& kernel_src) {
    log_trace(tt::LogBuildKernels, "Generating defines for TRISCs");

    const string out_dir = env.get_out_kernel_root_path() + settings.get_full_kernel_name() + "/";
    const string unpack_cpp = out_dir + "chlkc_unpack.cpp";
    const string math_cpp = out_dir + "chlkc_math.cpp";
    const string pack_cpp = out_dir + "chlkc_pack.cpp";
    const string isolate_sfpu_cpp = out_dir + "chlkc_isolate_sfpu.cpp";
```

Each generated file is a thin wrapper that:

1. **Sets a TRISC guard define** (`TRISC_UNPACK`, `TRISC_MATH`, `TRISC_PACK`, or `TRISC_ISOLATE_SFPU`) and includes `defines_generated.h`.
2. **Includes or inlines the user's kernel source**, potentially transformed from simplified to legacy syntax.

The TRISC prolog is generated by `build_trisc_prolog()`:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 137-142
string build_trisc_prolog(const char* trisc_define) {
    ostringstream prolog;
    prolog << "#define " << trisc_define << "\n";
    prolog << "#include \"defines_generated.h\"\n";
    return prolog.str();
}
```

So `chlkc_unpack.cpp` begins with:
```cpp
#define TRISC_UNPACK
#include "defines_generated.h"
// ... user kernel source follows ...
```

And `chlkc_math.cpp` begins with:
```cpp
#define TRISC_MATH
#include "defines_generated.h"
// ... user kernel source follows ...
```

These guards control which LLK code paths are compiled. Throughout the LLK library, headers use `#ifdef TRISC_UNPACK`, `#ifdef TRISC_MATH`, and `#ifdef TRISC_PACK` to expose only the relevant API functions to each TRISC processor. The fourth file, `chlkc_isolate_sfpu.cpp`, uses `TRISC_ISOLATE_SFPU` and is used on Quasar to separate SFPU code for an additional processing unit.

## Simplified Kernel Syntax Transformation

Modern TT-Metal compute kernels use simplified syntax where the entry point is declared as `void kernel_main()`. The code generation pipeline detects this and transforms it into the legacy namespace-per-TRISC format that the LLK infrastructure expects.

Detection happens via regex matching:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 64-106
namespace simple_kernel_syntax {

const std::regex kernel_main_pattern(R"(\bvoid\s+kernel_main\s*\(\s*\)\s*\{)");

// ...

bool is_used_in_source(const string& source) {
    // Use simplified syntax only if kernel_main is found AND no legacy markers present.
    if (find_kernel_main_definition(source) == string::npos) {
        return false;
    }
    if (has_legacy_syntax_markers(source)) {
        return false;
    }
    // Multiple kernel_main() with simplified syntax for compute is not supported.
    if (count_kernel_main_definitions(source) > 1) {
        throw std::runtime_error(
            "Multiple kernel_main() definitions found. Kernels with multiple entrypoints must use "
            "legacy syntax (namespace NAMESPACE { void MAIN { } }) for the compute path.");
    }
    return true;
}
```

The `transform_to_legacy_syntax()` function (lines 112-133) performs the actual transformation. It splits the source at the `void kernel_main()` boundary, places the preamble (includes, type definitions) outside the namespace, and wraps the function body inside the appropriate namespace with the correct entry-point name:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 112-133
string transform_to_legacy_syntax(const string& source, const char* ns_name, const char* func_name) {
    size_t func_pos = find_kernel_main_definition(source);
    if (func_pos == string::npos) {
        throw std::runtime_error("Could not find 'void kernel_main() {' in source");
    }

    string preamble = source.substr(0, func_pos);
    string function_part = source.substr(func_pos);

    // Rename kernel_main -> func_name
    size_t name_pos = function_part.find("kernel_main");
    if (name_pos != string::npos) {
        function_part.replace(name_pos, strlen("kernel_main"), func_name);
    }

    ostringstream result;
    result << preamble;
    result << "namespace " << ns_name << " {\n";
    result << function_part;
    result << "\n}  // namespace " << ns_name << "\n";
    return result.str();
}
```

This is called four times with different namespace/function pairs (lines 216-221):

| Generated file | Namespace | Function name |
|---|---|---|
| `chlkc_unpack.cpp` | `chlkc_unpack` | `unpack_main` |
| `chlkc_math.cpp` | `chlkc_math` | `math_main` |
| `chlkc_pack.cpp` | `chlkc_pack` | `pack_main` |
| `chlkc_isolate_sfpu.cpp` | `chlkc_isolate_sfpu` | `isolate_sfpu_main` |

For `FILE_PATH` kernel sources, a `#line` directive is emitted before the transformed source so that compiler error messages reference the original kernel file rather than the generated file:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 213-215
if (kernel_src.source_type_ == KernelSource::FILE_PATH) {
    line_directive = "#line 1 \"" + kernel_src.path_.string() + "\"\n";
}
```

Kernels using legacy syntax (`namespace NAMESPACE { void MAIN { } }`) skip the transformation and are included as-is. A deprecation warning is emitted for these:

```cpp
// tt_metal/jit_build/genfiles.cpp, lines 188-193
log_warning(
    tt::LogBuildKernels,
    "Compute kernel '{}' uses deprecated 'namespace NAMESPACE {{ void MAIN {{ }} }}' syntax. "
    "Please migrate to simplified 'void kernel_main() {{ }}' syntax.",
    settings.get_full_kernel_name());
```

## HAL Common Defines: Origin of `UCK_CHLKC_*` Guards

The preprocessor guards `UCK_CHLKC_UNPACK`, `UCK_CHLKC_MATH`, `UCK_CHLKC_PACK`, and `UCK_CHLKC_ISOLATE_SFPU` used throughout the generated headers (including `chlkc_descriptors.h` below) are **not** set by the generated source files. They originate from the HAL common defines layer, which injects them as `-D` flags during JIT compilation based on which TRISC processor the code is being compiled for.

For tt-1xx architectures (Wormhole, Blackhole), the function `HalJitBuildQueryBase::defines()` in `tt_metal/llrt/hal/tt-1xx/hal_1xx_common.cpp` maps each compute processor ID to its corresponding define and namespace:

```cpp
// tt_metal/llrt/hal/tt-1xx/hal_1xx_common.cpp, lines 34-51
case HalProcessorClassType::COMPUTE: {
    switch (params.processor_id) {
        case 0:
            defines.push_back("UCK_CHLKC_UNPACK");
            defines.push_back("NAMESPACE=chlkc_unpack");
            break;
        case 1:
            defines.push_back("UCK_CHLKC_MATH");
            defines.push_back("NAMESPACE=chlkc_math");
            break;
        case 2:
            defines.push_back("UCK_CHLKC_PACK");
            defines.push_back("NAMESPACE=chlkc_pack");
            break;
        default: TT_THROW("Invalid processor id {}", params.processor_id);
    }
    defines.push_back(fmt::format("COMPILE_FOR_TRISC={}", params.processor_id));
    break;
}
```

For tt-2xx architectures (Quasar), the equivalent logic in `tt_metal/llrt/hal/tt-2xx/hal_2xx_common.cpp` performs the same mapping but accounts for Quasar's multi-NEO topology, where each of four NEO engines has its own set of compute processors. The fourth processor in each NEO engine receives `UCK_CHLKC_ISOLATE_SFPU` and `NAMESPACE=chlkc_isolate_sfpu`.

The full set of defines injected by the HAL common layer for a TENSIX compute processor includes:

| Define | Purpose |
|---|---|
| `UCK_CHLKC_UNPACK` / `UCK_CHLKC_MATH` / `UCK_CHLKC_PACK` / `UCK_CHLKC_ISOLATE_SFPU` | Controls which LLK code paths are compiled in shared headers like `chlkc_descriptors.h` |
| `NAMESPACE=chlkc_*` | Sets the namespace that the compiled kernel code lives in, matching the generated source file's namespace |
| `COMPILE_FOR_TRISC=N` | Numeric processor ID (0=unpack, 1=math, 2=pack; extended on Quasar) |

Additionally, each architecture's HAL appends an architecture define (e.g., `ARCH_WORMHOLE`, `ARCH_BLACKHOLE`) from its own `defines()` override. Data-movement processors receive `COMPILE_FOR_BRISC` or `COMPILE_FOR_NCRISC` (tt-1xx) or `COMPILE_FOR_DM=N` (tt-2xx) instead of the compute-specific defines above.

These HAL-injected defines work in concert with the `TRISC_UNPACK` / `TRISC_MATH` / `TRISC_PACK` guards set in the generated source files (see "Splitting a Kernel into Four TRISC Source Files" above). The `TRISC_*` guards control code paths within the user's kernel source, while the `UCK_CHLKC_*` guards control code paths within the LLK library and generated descriptor headers.

## The `chlkc_descriptors.h` Header

The `generate_all_descriptors()` function (lines 488-525) creates a `chlkc_descriptors.h` header that configures the LLK library for the specific kernel invocation. This header is included by each TRISC source file via `defines_generated.h` and the TRISC guard defines control which sections are visible.

The generated header has this structure:

```cpp
// Generated chlkc_descriptors.h (reconstructed from genfiles.cpp lines 501-520)
#pragma once

#if defined(UCK_CHLKC_MATH)
#include "llk_defs.h"
constexpr ckernel::MathFidelity MATH_FIDELITY = static_cast<ckernel::MathFidelity>(...);
constexpr bool APPROX = ...;
#endif

#if !defined(UCK_CHLKC_PACK)
constexpr std::int32_t unpack_src_format[MAX_CBS] = { ... };
constexpr std::int32_t unpack_dst_format[MAX_CBS] = { ... };
constexpr uint8_t unpack_tile_num_faces[MAX_CBS] = { ... };
// ... additional unpack tile dimension arrays ...
#endif

#if !defined(UCK_CHLKC_MATH) && !defined(UCK_CHLKC_UNPACK)
constexpr unsigned char pack_src_format[MAX_CBS] = { ... };
constexpr unsigned char pack_dst_format[MAX_CBS] = { ... };
constexpr uint8_t pack_tile_num_faces[MAX_CBS] = { ... };
// ... additional pack tile dimension arrays ...
#endif

#if defined(UCK_CHLKC_MATH) || defined(UCK_CHLKC_PACK) || defined(UCK_CHLKC_UNPACK) || defined(UCK_CHLKC_ISOLATE_SFPU)
constexpr bool DST_ACCUM_MODE = ...;
#define DST_SYNC_MODE DstSync::SyncHalf  // or SyncFull
#endif
```

Key observations:

- **`#include "llk_defs.h"`** is only emitted under the `UCK_CHLKC_MATH` guard. This header comes from the LLK submodule and provides math-related definitions (fidelity enums, register constants) needed by the math TRISC.
- **Unpack format arrays** are visible to both unpack and math TRISCs (guarded by `!defined(UCK_CHLKC_PACK)`).
- **Pack format arrays** are only visible to the pack TRISC (guarded by `!defined(UCK_CHLKC_MATH) && !defined(UCK_CHLKC_UNPACK)`).
- **`DST_ACCUM_MODE` and `DST_SYNC_MODE`** are visible to all compute TRISCs and control accumulation and synchronization behavior in the destination register.

The data format arrays are populated at JIT time based on the `tt_hlk_desc` descriptor, which captures the data formats of all circular buffers configured for the kernel. The `compute_data_formats()` function (line 366) handles format selection, including automatic promotion to `Tf32` when FP32 destination accumulation is enabled.

## The Fake Kernels Target

TT-Metal includes a build-time validation mechanism: the `jit_kernels_index` CMake target (defined in `tt_metal/jit_build/fake_kernels_target/CMakeLists.txt`) compiles all in-tree kernel files to catch syntax errors and missing includes without waiting for runtime JIT compilation.

This target gathers all kernel `.cpp` files from the repository:

```cmake
# tt_metal/jit_build/fake_kernels_target/CMakeLists.txt, lines 1-8
set(ALL_KERNEL_FILES "")
file(
    GLOB_RECURSE TT_METAL_KERNELS
    "${CMAKE_SOURCE_DIR}/tt_metal/kernels/*.cpp"
    "${CMAKE_SOURCE_DIR}/tt_metal/kernels/*.cc"
)
```

It also collects dispatch kernels, fabric kernels, and TTNN operation kernels (lines 9-42).

The target replicates the same LLK include paths that the JIT compiler would use:

```cmake
# tt_metal/jit_build/fake_kernels_target/CMakeLists.txt, lines 62-88
add_library(jit_kernels_index OBJECT EXCLUDE_FROM_ALL ${ALL_KERNEL_FILES})
target_include_directories(
    jit_kernels_index
    PRIVATE
        # ... standard includes ...
        "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/common"
        "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/common/inc"
        "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/llk_lib"
        "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/common"
        "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/llk_api"
        "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/llk_api/llk_sfpu"
        "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/llk_io"
        "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/common/inc/sfpu"
)
```

Because JIT compilation normally generates `chlkc_descriptors.h` with kernel-specific values, the fake-kernel build cannot produce one. Instead, it force-includes a `fake_jit_prelude.h` header (line 113) that provides hardcoded placeholder values for all the arrays and constants that `chlkc_descriptors.h` would normally supply:

```cmake
# tt_metal/jit_build/fake_kernels_target/CMakeLists.txt, lines 112-113
-include
"${CMAKE_SOURCE_DIR}/tt_metal/jit_build/fake_kernels_target/fake_jit_prelude.h"
```

The fake prelude (`tt_metal/jit_build/fake_kernels_target/fake_jit_prelude.h`) provides:

- Scalar configuration: `DST_ACCUM_MODE = false`, `DST_SYNC_MODE = DstSync::SyncHalf`, `APPROX = true`, `MATH_FIDELITY = 255`
- Data format arrays: all 32 entries set to `255` (representing `DataFormat::Invalid`)
- Tile dimension arrays: default 32x32 tile dimensions with 4 faces

The target also defines `TRISC_MATH=1` and `NAMESPACE=chlkc_math` (lines 124-125), meaning it compiles all kernels as if they were running on the math TRISC. This is a practical choice since the math path exercises the most LLK API surface area.

The target is marked `EXCLUDE_FROM_ALL`, so it does not build by default -- developers must explicitly request it. This keeps normal build times fast while providing an opt-in validation tool.
