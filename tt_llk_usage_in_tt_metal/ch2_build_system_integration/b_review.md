In the Wormhole HAL, the equivalent line (line 123) is INSIDE the TENSIX case. This is a real architectural difference between the two HALs that the chapter obscures by truncating both snippets to look structurally identical. A developer porting firmware includes between architectures would get this wrong.

There is a **second hash** computed in the `JitBuildState` constructor at `build.cpp` lines 419-434 (`build_state_hash_`), which explicitly hashes `includes_` (line 424) along with `cflags_`, `defines_`, `lflags_`, `linker_script_`, `extra_link_objs_`, `srcs_`, and optimization levels. Changes to include paths DO invalidate the build cache through this mechanism. The chapter's claim would mislead someone debugging stale-binary issues. Fix: replace the "subtle gap" paragraph with an accurate description of the two-level hashing scheme (`build_key_` at `JitBuildEnv` level for output directory routing, `build_state_hash_` at `JitBuildState` level for actual cache invalidation including includes).
