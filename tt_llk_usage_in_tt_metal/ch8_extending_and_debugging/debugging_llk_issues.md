# Debugging LLK Issues

When a compute kernel misbehaves, the root cause often lies in one of several areas: incorrect data formats, wrong preprocessor defines, assertion failures in the LLK layer, or outright compile errors masked by the JIT system. This section covers the debugging tools and techniques available at each level.

## Enabling LLK Assertions

The LLK library contains assertions throughout its unpack, math, and pack code. These are gated behind the `ENABLE_LLK_ASSERT` preprocessor define and are **disabled by default** for performance.

### How to Enable

Set the environment variable before running your program:

```bash
export TT_METAL_LLK_ASSERTS=1
```

This is read by `rtoptions.cpp` (line 1267):

```cpp
case EnvVarID::TT_METAL_LLK_ASSERTS: this->enable_llk_asserts = true; break;
```

During JIT compilation, `build.cpp` (line 259-260) checks this flag and adds the define to the compiler command line:

```cpp
if (rtoptions.get_llk_asserts()) {
    this->defines_ += "-DENABLE_LLK_ASSERT ";
}
```

### How LLK_ASSERT Works

The macro is defined in [`common/llk_assert.h`](https://github.com/tenstorrent/tt-llk/blob/main/common/llk_assert.h):

```cpp
// When ENABLE_LLK_ASSERT is defined and running inside TT-Metal:
#include "api/debug/assert.h"
#define LLK_ASSERT(condition, message) ASSERT(condition)

// When ENABLE_LLK_ASSERT is defined and running in the standalone LLK test infra:
#define LLK_ASSERT(condition, message) \
    do {                               \
        if (UNLIKELY(!(condition)))    \
        {                              \
            asm volatile("ebreak");    \
        }                              \
    } while (0)

// When ENABLE_LLK_ASSERT is NOT defined (production):
#define LLK_ASSERT(condition, message) ((void)sizeof((condition)))
```

The production path uses `sizeof` to ensure the condition is still type-checked and names are resolved at compile time, but generates no runtime code. This is intentional -- it catches typos even in production builds without paying the performance cost of evaluation.

### What Gets Asserted

LLK assertions validate critical invariants. Some examples from the codebase:

- **Destination index bounds** (`llk_math_eltwise_unary_sfpu_params.h` line 17):
  ```cpp
  LLK_ASSERT((dst_index < get_dest_max_tiles<DST_SYNC_MODE, DST_ACCUM_MODE, DstTileShape::Tile32x32>()),
             "dst_index exceeds max dest tiles");
  ```

- **L1 address validity** (`llk_unpack_common.h` line 204):
  ```cpp
  LLK_ASSERT(is_valid_L1_address(address_a), "L1 address_a must be in valid L1 memory region");
  ```

- **Face count constraints** (`llk_unpack_tilize.h` line 191):
  ```cpp
  LLK_ASSERT(num_faces == 1 || num_faces == 2 || num_faces == 4,
             "num_faces must be 1, 2, or 4");
  ```

When an assertion fires on device, it triggers the Watcher debug infrastructure in TT-Metal, which reports the failing core coordinates and a stack trace.

## Inspecting Generated Files

The JIT build system generates several files per kernel compilation. These are the first place to look when debugging unexpected behavior.

### Where to Find Them

Generated files are written to the JIT output directory. The path is:

```
<build_root>/<build_key>/kernels/<kernel_name>/
```

The build root is typically `built/<build_key>/kernels/` under TT-Metal's working directory. To find the exact path, enable kernel compilation logging:

```bash
export TT_METAL_LOG_KERNELS_COMPILE_COMMANDS=1
```

This causes `build.cpp` (line 526-528) to print every `g++` compile command, which includes the full output directory path.

### `chlkc_descriptors.h`

Generated by `generate_all_descriptors()` in [`genfiles.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/genfiles.cpp) (lines 488-536). Contains:

- **Math scalars** (guarded by `UCK_CHLKC_MATH`):
  ```cpp
  constexpr ckernel::MathFidelity MATH_FIDELITY = static_cast<ckernel::MathFidelity>(4);
  constexpr bool APPROX = false;
  ```

- **Compute scalars** (guarded by `UCK_CHLKC_MATH || UCK_CHLKC_PACK || UCK_CHLKC_UNPACK || UCK_CHLKC_ISOLATE_SFPU`):
  ```cpp
  constexpr bool DST_ACCUM_MODE = false;
  #define DST_SYNC_MODE DstSync::SyncHalf
  ```

- **Unpack data format arrays** (guarded by `!UCK_CHLKC_PACK`):
  ```cpp
  constexpr uint32_t unpack_src_format[NUM_CBS] = { ... };
  constexpr uint32_t unpack_dst_format[NUM_CBS] = { ... };
  ```

- **Pack data format arrays** (guarded by `!UCK_CHLKC_MATH && !UCK_CHLKC_UNPACK`):
  ```cpp
  constexpr uint32_t pack_src_format[NUM_CBS] = { ... };
  constexpr uint32_t pack_dst_format[NUM_CBS] = { ... };
  ```

- **Tile dimension arrays** for both unpack and pack.

If your operation produces wrong numerical results, check that the data format arrays match your expectations. A common mistake is configuring the wrong `DataFormat` for a circular buffer, which leads to the unpack hardware misinterpreting the tile data.

### `chlkc_math.cpp`, `chlkc_unpack.cpp`, `chlkc_pack.cpp`

Generated by `jit_build_genfiles_triscs_src()` in [`genfiles.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/genfiles.cpp) (lines 171-247). Each file contains:

1. A `#define` for the TRISC type (`TRISC_MATH`, `TRISC_UNPACK`, or `TRISC_PACK`).
2. An `#include "defines_generated.h"` directive.
3. The kernel source, either:
   - **Simplified syntax**: The `kernel_main()` function is transformed into a namespaced `math_main()` / `unpack_main()` / `pack_main()` via `simple_kernel_syntax::transform_to_legacy_syntax()`.
   - **Legacy syntax**: The source is included directly (for `FILE_PATH` sources, via `#include`).

Inspect these files to verify that:
- Your kernel source was correctly split across the three TRISCs.
- The `TRISC_MATH` / `TRISC_UNPACK` / `TRISC_PACK` define is present (this controls `#ifdef` guards in the Compute API headers).
- `#line` directives point back to your original source file (for `FILE_PATH` kernels).

### `defines_generated.h`

Generated alongside the `chlkc_*.cpp` files (`genfiles.cpp` line 235). Contains all defines added via `ComputeConfig::defines` or `DataMovementConfig::defines` on the host:

```cpp
#define REDUCE_OP PoolType::MAX
#define REDUCE_DIM ReduceDim::REDUCE_ROW
```

These are written using `settings.process_defines()` (line 241-243):

```cpp
settings.process_defines([&gen_defines_file](const string& define, const string& value) {
    gen_defines_file << "#define " << define << " " << value << endl;
});
```

If an operation depends on specific define values (like `REDUCE_OP` or `REDUCE_DIM`), verify them here.

## The `fake_kernels_target` CMake Target

The `jit_kernels_index` target (defined in [`tt_metal/jit_build/fake_kernels_target/CMakeLists.txt`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/fake_kernels_target/CMakeLists.txt)) compiles all kernel `.cpp` files at CMake build time, catching syntax errors and missing headers **without needing hardware**.

### How It Works

The CMakeLists.txt collects all kernel sources from three locations:

```cmake
file(GLOB_RECURSE TT_METAL_KERNELS "${CMAKE_SOURCE_DIR}/tt_metal/kernels/*.cpp" ...)
file(GLOB_RECURSE TT_METAL_DISPATCH_KERNELS "${CMAKE_SOURCE_DIR}/tt_metal/impl/dispatch/kernels/*.cpp" ...)
file(GLOB_RECURSE TTNN_OPERATION_KERNELS ...)  # filtered to */kernels/ and */kernels_ng/
```

It creates an `OBJECT` library target that compiles these files with representative include paths and defines:

```cmake
target_include_directories(jit_kernels_index PRIVATE
    "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/common/inc"
    "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/llk_lib"
    "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/llk_api"
    "${CMAKE_SOURCE_DIR}/tt_metal/hw/ckernels/${ARCH_NAME_LOWER}/metal/llk_api/llk_sfpu"
    "${CMAKE_SOURCE_DIR}/tt_metal/third_party/tt_llk/tt_llk_${ARCH_NAME_LOWER}/common/inc/sfpu"
    # ... plus many more paths
)
```

Since the generated files (`chlkc_descriptors.h`, etc.) do not exist at CMake time, the target uses a [`fake_jit_prelude.h`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/fake_kernels_target/fake_jit_prelude.h) that provides hardcoded placeholder values:

```cpp
// fake_jit_prelude.h (lines 15-17)
// Those values come from generated chlkc_descriptors.h, so hardcode values here for fake JIT builds
// chlkc_descriptors.h (scalars)
constexpr bool DST_ACCUM_MODE = false;
```

The target defaults to compiling as if for the MATH TRISC:

```cmake
target_compile_definitions(jit_kernels_index PRIVATE
    TRISC_MATH=1
    NAMESPACE=chlkc_math
    TENSIX_FIRMWARE
    KERNEL_BUILD
    # ...
)
```

### Using It

Build the target with:

```bash
cmake --build build -- jit_kernels_index
```

This is especially useful in CI. The target is `EXCLUDE_FROM_ALL`, so it does not build by default -- you must request it explicitly. If any kernel has a syntax error, missing include, or type mismatch, the build will fail with a standard GCC error message pointing to the exact source line.

### Limitations

- Only compiles for the MATH TRISC by default. Unpack-only or pack-only code paths are not exercised unless you modify the CMake defines.
- Uses placeholder descriptor values, so compile-time checks that depend on actual data formats (e.g. `static_assert` on `DataFormat` values) may not catch mismatches.
- Does not link or produce runnable binaries -- it only validates compilation.

## Key Preprocessor Defines to Check

When debugging unexpected behavior, verify these defines in the generated files or compiler command line:

| Define | Source | Controls |
|--------|--------|----------|
| `MATH_FIDELITY` | `chlkc_descriptors.h` | Math pipeline fidelity (LoFi, HiFi2, HiFi3, HiFi4) |
| `APPROX` | `chlkc_descriptors.h` | Whether SFPU uses approximation mode |
| `DST_ACCUM_MODE` | `chlkc_descriptors.h` | Whether FP32 destination accumulation is enabled |
| `DST_SYNC_MODE` | `chlkc_descriptors.h` | `DstSync::SyncFull` or `DstSync::SyncHalf` |
| `REDUCE_OP` | `defines_generated.h` | Pool type for reduce operations (MAX, AVG, SUM) |
| `REDUCE_DIM` | `defines_generated.h` | Reduce dimension (REDUCE_ROW, REDUCE_COL, etc.) |
| `ARCH_WORMHOLE` / `ARCH_BLACKHOLE` | Compiler command line | Architecture selector, set by HAL |
| `TRISC_MATH` / `TRISC_UNPACK` / `TRISC_PACK` | `chlkc_*.cpp` prolog | Which TRISC processor this translation unit targets |
| `KERNEL_BUILD` | Compiler command line | Distinguishes kernel builds from firmware builds |
| `KERNEL_COMPILE_TIME_ARGS` | Compiler command line | Comma-separated list of compile-time argument values |
| `ENABLE_LLK_ASSERT` | Compiler command line | Enables LLK runtime assertions |

### Debugging Checklist

1. **Enable verbose compilation**: `export TT_METAL_LOG_KERNELS_COMPILE_COMMANDS=1`
2. **Enable LLK assertions**: `export TT_METAL_LLK_ASSERTS=1`
3. **Inspect `chlkc_descriptors.h`**: Verify data formats match your circular buffer configuration.
4. **Inspect `defines_generated.h`**: Verify host-side defines are present and correct.
5. **Inspect `chlkc_math.cpp`**: Verify the kernel source was correctly transformed.
6. **Check the compile command**: Verify include paths contain the expected architecture directories.
7. **Build `jit_kernels_index`**: Catch compile errors without hardware.
8. **Enable Watcher**: `export TT_METAL_WATCHER=120` to enable the periodic device health checker, which reports assertion failures, hangs, and NOC errors.
