# Compilation Flow End to End

This section traces the complete path from a host-side `CreateKernel()` call to a binary running on a Tensix core's TRISC processor. Understanding this flow is essential for debugging build failures, investigating performance issues caused by unexpected compiler flags, and extending the system with new operations.

## Overview

The compilation of a compute kernel proceeds through these stages:

```
Host: CreateKernel() with ComputeConfig
  --> JitBuildState assembles compiler command
    --> genfiles.cpp generates chlkc_*.cpp and chlkc_descriptors.h
      --> GCC cross-compiles three translation units (one per TRISC)
        --> Linker produces three .elf binaries
          --> Binaries loaded to device L1 memory
```

## Stage 1: Host Calls `CreateKernel()`

On the host, the user creates a compute kernel by calling `CreateKernel()` with a [`ComputeConfig`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/api/tt-metalium/kernel_types.hpp) (lines 76-98):

```cpp
auto kernel_id = CreateKernel(
    program,
    "path/to/my_kernel.cpp",
    core_range,
    ComputeConfig{
        .math_fidelity = MathFidelity::HiFi4,
        .fp32_dest_acc_en = false,
        .math_approx_mode = true,
        .compile_args = {num_tiles, param0},
        .defines = {{"REDUCE_OP", "PoolType::MAX"}},
    });
```

The `ComputeConfig` fields are stored on the `Kernel` object. When `CompileProgram()` is called, these fields flow into the JIT build system.

## Stage 2: `JitBuildEnv` Initialization

Before any kernel is compiled, a `JitBuildEnv` object is constructed once per device open. This happens in [`build.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/build.cpp) (lines 200-303).

The environment captures:
- **Compiler path** (`env_.gpp_`): The RISC-V GCC cross-compiler.
- **Base compiler flags** (`cflags_`): `-std=c++20 -flto=auto -ffast-math -fno-exceptions` and similar.
- **Base defines** (`defines_`): Runtime-conditional defines like `-DENABLE_LLK_ASSERT`, `-DLIGHTWEIGHT_KERNEL_ASSERTS`.
- **Base include paths** (`includes_`): A set of common paths (lines 269-279):
  ```cpp
  std::vector<std::string> includeDirs = {
      ".", "..",
      root_,
      root_ + "ttnn", root_ + "ttnn/cpp",
      root_ + "tt_metal",
      root_ + "tt_metal/hw/inc",
      root_ + "tt_metal/third_party/tt_llk/common",
      root_ + "tt_metal/hostdevcommon/api",
      root_ + "tt_metal/api/"
  };
  ```

## Stage 3: `JitBuildState` Construction (Per-Processor)

For each TRISC processor (unpack, math, pack), a `JitBuildState` is constructed (`build.cpp` lines 305-435). This queries the HAL for architecture-specific configuration:

```cpp
const auto& jit_build_query = hal.get_jit_build_query();

// Architecture-specific includes (appended to base includes)
for (const auto& include : jit_build_query.includes(params)) {
    fmt::format_to(it, "-I{}{} ", env_.root_, include);
}

// Architecture-specific defines
for (const auto& define : jit_build_query.defines(params)) {
    fmt::format_to(it, "-D{} ", define);
}
```

As described in Chapter 7, the HAL's `includes()` method returns paths like:
- `tt_metal/hw/ckernels/wormhole_b0/metal/common`
- `tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/common/inc`
- `tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/llk_lib`
- `tt_metal/hw/ckernels/wormhole_b0/metal/llk_api`
- `tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu`
- `tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/common/inc/sfpu`

The compute TRISC build states use `-O3` optimization (line 319), while data movement processors default to `-Os`.

## Stage 4: Code Generation (`genfiles.cpp`)

Before compilation, `genfiles.cpp` generates several files into the kernel's output directory.

### `chlkc_descriptors.h`

Generated by `generate_all_descriptors()` ([`genfiles.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/genfiles.cpp) lines 488-536). This file translates the `ComputeConfig` fields into compile-time constants:

```cpp
// Math section (guarded by UCK_CHLKC_MATH)
constexpr ckernel::MathFidelity MATH_FIDELITY = static_cast<ckernel::MathFidelity>(4);
constexpr bool APPROX = true;

// Compute section (shared across all TRISCs)
constexpr bool DST_ACCUM_MODE = false;
#define DST_SYNC_MODE DstSync::SyncHalf

// Unpack data format arrays
constexpr uint32_t unpack_src_format[NUM_CBS] = { ... };
constexpr uint32_t unpack_dst_format[NUM_CBS] = { ... };

// Pack data format arrays
constexpr uint32_t pack_src_format[NUM_CBS] = { ... };
constexpr uint32_t pack_dst_format[NUM_CBS] = { ... };
```

The data format arrays are derived from the circular buffer configurations provided by the host program. Each entry corresponds to a circular buffer index and encodes the `DataFormat` enum value.

### `chlkc_math.cpp`, `chlkc_unpack.cpp`, `chlkc_pack.cpp`

Generated by `jit_build_genfiles_triscs_src()` ([`genfiles.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/genfiles.cpp) lines 171-247). Each file has a prolog:

```cpp
#define TRISC_MATH        // or TRISC_UNPACK or TRISC_PACK
#include "defines_generated.h"
```

followed by the kernel source. For kernels using the simplified `kernel_main()` syntax, the source is transformed (lines 108-133): the function is renamed to `math_main` / `unpack_main` / `pack_main` and wrapped in a `namespace chlkc_math { ... }` block. A `#line` directive preserves original source file references in compiler error messages.

### `defines_generated.h`

Contains all host-side defines passed through `ComputeConfig::defines`:

```cpp
#define REDUCE_OP PoolType::MAX
```

### `chlkc_isolate_sfpu.cpp`

A fourth translation unit for Quasar's isolated SFPU processor (not used on Wormhole/Blackhole).

## Stage 5: Cross-Compilation

The `compile_one()` method ([`build.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/build.cpp) lines 464-543) assembles the final GCC command for each source file:

```
cd <out_dir> && <riscv-gcc>
  -O3                              # optimization level (O3 for compute)
  -std=c++20 -flto=auto -ffast-math -fno-exceptions ...  # base flags
  -I<root>/tt_metal/hw/inc         # base includes
  -I<root>/tt_metal/third_party/tt_llk/common
  -I<root>/tt_metal/hw/ckernels/wormhole_b0/metal/common   # HAL includes
  -I<root>/tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/llk_lib
  -I<root>/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api
  -I<root>/tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_sfpu
  -I<root>/tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/common/inc/sfpu
  -I<kernel_source_dir>            # kernel-specific include
  -DTRISC_MATH=1                   # TRISC type
  -DKERNEL_BUILD
  -DARCH_WORMHOLE                  # architecture define
  -DKERNEL_COMPILE_TIME_ARGS=64,42 # compile-time args
  -DENABLE_LLK_ASSERT              # if enabled
  -c -o chlkc_math.o chlkc_math.cpp
```

The compilation happens in parallel across source files using `launch_build_step()` and a thread pool. All three TRISC source files (unpack, math, pack) can compile simultaneously.

### Caching

Before compiling, the build system checks two layers of cache:
1. **Build state hash** (`build.cpp` lines 439-462): An FNV1a hash of all compiler flags, defines, includes, and source paths. If it changes, all cached objects are invalidated.
2. **Dependency hash** (per-object): A hash of the object file and its header dependencies (via the `.d` depfile from GCC). If a header changes, only affected objects recompile.

## Stage 6: Linking

The `link()` method ([`build.cpp`](https://github.com/tenstorrent/tt-metal/blob/main/tt_metal/jit_build/build.cpp) lines 583-629) links each TRISC's object files into an ELF binary:

```
cd <out_dir> && <riscv-gcc>
  -O3
  -Wl,-z,max-page-size=16 -Wl,-z,common-page-size=16 -nostartfiles
  -Wl,--just-symbols=<weakened_firmware.elf>   # import firmware symbols
  -T<linker_script.ld>                         # architecture-specific linker script
  -Wl,--emit-relocs                            # for kernel builds
  <extra_link_objs>                            # pre-compiled HW objects
  chlkc_math.o ... -o trisc2/trisc2.elf
```

Key details:
- The weakened firmware ELF provides symbol addresses for firmware functions that kernels call (like NOC operations). The `weaken()` method (line 635-649) processes the firmware ELF to mark its symbols as weak, preventing duplicate symbol errors.
- `--emit-relocs` preserves relocation information so the runtime can relocate kernels when loading them to different L1 addresses.
- The linker script is architecture-specific and defines the memory layout for the TRISC processor.

The result is three ELF binaries, one per TRISC:
- `trisc0/trisc0.elf` -- Unpack processor
- `trisc1/trisc1.elf` -- Math processor (where SFPU operations run)
- `trisc2/trisc2.elf` -- Pack processor

## Stage 7: Binary Loading

The three ELF binaries are loaded to the target Tensix core's L1 memory by the dispatch infrastructure. Each TRISC processor has a dedicated code region in L1 where its binary is placed. The processor begins execution at the ELF entry point, which calls the TRISC firmware that in turn invokes the kernel's `math_main()` / `unpack_main()` / `pack_main()` function.

## Include Path Resolution Order

When the GCC compiler resolves an `#include` directive, the search order is determined by the order of `-I` flags on the command line. The effective order for a compute kernel is:

1. **JIT output directory** (current working directory `.` and `..`) -- finds `chlkc_descriptors.h`, `defines_generated.h`
2. **TT-Metal root paths** -- `tt_metal/`, `ttnn/`, `tt_metal/hw/inc`, `tt_metal/hostdevcommon/api`, `tt_metal/api/`
3. **Common TT-LLK path** -- `tt_metal/third_party/tt_llk/common` (contains `llk_assert.h`)
4. **HAL architecture-specific paths** (from `jit_build_query.includes()`):
   - `hw/ckernels/{arch}/metal/common` -- `chlkc_list.h`, common headers
   - `third_party/tt_llk/tt_llk_{arch}/common/inc` -- architecture-specific LLK headers
   - `third_party/tt_llk/tt_llk_{arch}/llk_lib` -- LLK implementation headers
   - `hw/ckernels/{arch}/metal/llk_api` -- Metal LLK API wrappers
   - `hw/ckernels/{arch}/metal/llk_api/llk_sfpu` -- Metal SFPU API wrappers
   - `third_party/tt_llk/tt_llk_{arch}/common/inc/sfpu` -- SFPI kernel implementations
   - `hw/inc/{arch_prefix}` -- architecture hardware register definitions
   - `hw/inc/{arch_prefix}/noc` -- NOC definitions
5. **Kernel source directory** -- appended via `settings.process_include_paths()` (line 521), allowing kernels to use relative `#include` directives for co-located headers

This ordering matters when headers of the same name exist in multiple directories. For example, `ckernel_sfpu_relu.h` exists in both `tt-llk` (the SFPI implementation) and `ckernels/{arch}/metal/llk_api/llk_sfpu/` (the Metal wrapper). The Metal wrapper is found first because `llk_api/llk_sfpu/` appears before `common/inc/sfpu/` in the include path. The Metal wrapper then includes the TT-LLK implementation via `sfpu/ckernel_sfpu_relu.h` (a relative path into the SFPU directory).

## Summary Diagram

```
Host Program
    |
    v
CreateKernel(program, "my_kernel.cpp", cores, ComputeConfig{...})
    |
    v
CompileProgram()
    |
    +---> genfiles.cpp generates:
    |       chlkc_descriptors.h  (data formats, math fidelity, DST config)
    |       chlkc_math.cpp       (kernel source with TRISC_MATH define)
    |       chlkc_unpack.cpp     (kernel source with TRISC_UNPACK define)
    |       chlkc_pack.cpp       (kernel source with TRISC_PACK define)
    |       defines_generated.h  (host-side #defines)
    |
    +---> JitBuildState::compile_one() x 3 (parallel)
    |       riscv-gcc -O3 ... -c chlkc_math.cpp -o chlkc_math.o
    |       riscv-gcc -O3 ... -c chlkc_unpack.cpp -o chlkc_unpack.o
    |       riscv-gcc -O3 ... -c chlkc_pack.cpp -o chlkc_pack.o
    |
    +---> JitBuildState::link() x 3
    |       riscv-gcc ... -o trisc0/trisc0.elf  (unpack)
    |       riscv-gcc ... -o trisc1/trisc1.elf  (math)
    |       riscv-gcc ... -o trisc2/trisc2.elf  (pack)
    |
    v
Load ELFs to Tensix core L1 memory
    |
    v
TRISC0 runs unpack_main()  |  TRISC1 runs math_main()  |  TRISC2 runs pack_main()
```
