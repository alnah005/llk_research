# 8.4 Common Gotchas

This section catalogs the most frequent mistakes when implementing new micro-ops in Blaze, each with the specific source-code mechanism that causes the problem and how to fix it.

## 1. CT Arg Name Mismatches (Python Prefix vs C++ Field)

### The Problem

When a BlazeOp declares `ct_args`, each argument has a `name` field. This name must match exactly between the Python emit-side and the C++ `ct_args::` struct. The struct is auto-generated by Metal's JIT compiler from the named compile-time args that Blaze provides.

The Python side uses a prefix to namespace CT args. The prefix-to-struct mapping is handled by `_to_ct_args_type()` in `kernel_codegen.py`:

```python
def _to_ct_args_type(prefix: str) -> str:
    """Map logical op prefix to generated ct_args struct name."""
    return prefix.replace(".", "__").replace("-", "_")
```

So a prefix of `"shared_expert.gu"` maps to `ct_args::shared_expert__gu`.

### How It Manifests

If Python emits `("activation_cb", 5)` but the C++ Op struct reads `ct_args::my_prefix.src`, the JIT build succeeds (the named_args_generated.h has `activation_cb` as a field) but the Op reads the wrong field or a default value, causing silent data corruption. In some cases, the mismatch produces a Metal kernel compile error about missing `ct_args::` struct members.

### Branch Suffix Handling

For branched ops (e.g., KNSlicedMatmul with gate/up branches), the CT prefix includes the branch suffix. The codegen strips it to find the parent prefix:

```python
# blaze/kernel_codegen.py
def _compute_prefix(node: OpNode) -> str:
    prefix = node.kwargs.get("ct_prefix", node.spec.name)
    branch = node.kwargs.get("branch")
    if branch and prefix.endswith(f".{branch}"):
        prefix = prefix[: -len(f".{branch}")]
    return prefix
```

If you manually set `ct_prefix` with a branch suffix but do not set `branch` in kwargs, the codegen will not strip the suffix and will generate a mismatched struct name.

### How to Avoid

- Use the exact same names in `CompileTimeArg(name=...)` as the C++ Op struct expects
- The `kind` field determines the source: `CB(port)` reads the CB ID for that port, `Sem(protocol)` reads a semaphore address, `Grid()` reads a NOC coordinate, `Derived()` reads a user-computed value
- Check `named_args_generated.h` in `~/.cache/tt-metal-cache/` to verify the actual struct fields
- Add a CT arg schema test in `tests/blaze/infra/test_ct_args.py` for your new op

### Diagnosis

Use `blaze.compile_engines(ctx.graph)` to inspect the generated CT tuples and compare against the `named_args_generated.h` in `~/.cache/tt-metal-cache/`. The visualizer (`blaze.visualize()`) also shows CT arg name-value pairs per node.

## 2. Missing Flags (All Flags Must Be Declared)

### The Problem

Blaze ops commonly use per-core boolean flags like `is_sender`, `is_active`, `is_receiver_in_mcast_range`, etc. These are declared in the op's `ct_args` list and emitted via `f.per_core_unified_ct_args()` or variants. The C++ side reads them from the `ct_args` struct.

If a flag is not declared for all cores, the C++ side reads an uninitialized or default value (0). This means a core that should participate does nothing, potentially causing a hang or data corruption.

### How It Manifests

A gather op has `is_active` flag. You set it to `True` for matmul cores but forget to emit it for the sender core. The sender reads `is_active=0` and does not participate in the gather's receive logic, causing the gather to hang waiting for all senders.

Another common case: a new op adds a `pop_input` flag but only declares it when the feature is enabled:

```python
# Wrong: flag only declared when pop is needed
if needs_pop:
    f.per_core_unified_ct_args([
        f.flag(f"{prefix}.pop_input", f.all_cores),
    ])
```

When `needs_pop` is `False`, the flag is not declared at all, and all cores read uninitialized data.

### How to Avoid

Always declare every flag, using `enabled=False` when disabled:

```python
# Correct: flag always declared
f.per_core_unified_ct_args([
    f.flag(f"{prefix}.pop_input", f.all_cores, needs_pop),
])
```

Use `FusedProgram.flag()` to create flag tuples. It explicitly sets the flag to 1 on the given cores when `enabled=True` and 0 when `enabled=False`:

```python
f.per_core_unified_ct_args([
    f.flag("mcast.is_sender", f.sender_grid, enabled=True),
    f.flag("mcast.is_active", f.all_cores, enabled=True),
])
```

### Duplicate Detection

FusedProgram detects duplicate flag names via `_per_core_ct_args_check()`:

```python
def _per_core_ct_args_check(self, args):
    for name, mapping in args:
        if name in self._ct_arg_names_seen:
            raise ValueError(f"Duplicate per-core CT arg: {name!r}")
        self._ct_arg_names_seen.add(name)
```

This catches cases where two ops accidentally use the same flag name. The error is raised at emit() time, before any device interaction. However, this check does NOT detect missing flags -- it only catches duplicates.

## 3. Wrong RISC Struct Assignment

### The Problem

Each CT arg has a `riscs` field (`Risc.NCRISC`, `Risc.TRISC`, `Risc.BRISC`, or combinations via `|`). This determines which RISC processor(s) receive the argument. If an arg is sent to the wrong RISC, the correct RISC reads an old or default value.

The `Risc` enum is a Flag type from `blaze_op.py`:

```python
class Risc(Flag):
    NCRISC = auto()
    BRISC = auto()
    TRISC = auto()
```

### How It Manifests

A matmul op needs `k_num_tiles` on both NCRISC (for the weight reader loop count) and TRISC (for the compute loop count). If you declare it as `riscs=Risc.TRISC` only, NCRISC does not receive the value and loops the wrong number of times (or infinitely).

### How to Avoid

- Check the C++ Op struct to see which RISCs read each field
- Use `Risc.TRISC | Risc.NCRISC` for args shared across processors
- The BRISC processor is used for data movement RISC 0; args for receiver-side gather logic typically go to BRISC
- Test with `blaze.compile_engines(ctx.graph).ct_tuples` to verify each RISC receives the expected args

**RISC assignment guidelines**:
- CB IDs that the data movement reader needs: `Risc.NCRISC` (or `Risc.BRISC` for writer)
- CB IDs that compute uses: `Risc.TRISC`
- CB IDs shared across data movement and compute: `Risc.NCRISC | Risc.TRISC` (or add `Risc.BRISC`)
- NOC coordinates (for multicasting): `Risc.NCRISC`
- Tile counts for compute loops: `Risc.TRISC` (and possibly `Risc.NCRISC` if the reader also loops)
- Semaphore IDs: The RISC that signals/waits on the semaphore

From `test_ct_args.py`, a reference for correct assignments:

```python
def test_matmul_schema_trisc_args(self):
    schema = get_ct_schema("matmul")
    trisc_args = schema.args_for_risc("trisc")
    names = [a.name for a in trisc_args]
    assert "in0" in names
    assert "in1" in names
    assert "out" in names
    assert "k_num_tiles" in names
    assert "out_w_per_core" in names
    assert len(trisc_args) == 5

def test_matmul_schema_ncrisc_args(self):
    schema = get_ct_schema("matmul")
    ncrisc_args = schema.args_for_risc("ncrisc")
    names = [a.name for a in ncrisc_args]
    assert "in0" in names
    assert "in0_is_tensor_backed" in names
    assert "k_num_tiles" in names
    assert "in1" not in names       # in1 is TRISC-only
    assert "out_w_per_core" not in names
    assert len(ncrisc_args) == 3
```

Note: The CT arg schema registration validates that port names exist but does NOT validate RISC assignment -- that is the op author's responsibility.

## 4. CB Page Mismatches: `cb_scratch` vs `cb_from_tensor`

### The Problem

`cb_from_tensor()` creates a tensor-backed CB whose geometry is derived from the tensor itself. `cb_scratch()` creates a scratch CB whose geometry is specified explicitly. If a producer writes to a tensor-backed CB but the consumer treats it as scratch (or vice versa), the page size and layout may not match.

### How It Manifests

You pass a `ttnn.Tensor` to `cb_from_tensor()` for the input, then create a scratch CB for intermediate storage, but set the scratch CB's `page_size` to match the wrong tile. The compute RISC reads tiles with one size but the pack RISC writes with another, corrupting data boundaries.

**Page size formula**: `page_size = tile_height * tile_width * bytes_per_element`

Common errors:
- Using 32x32 page size (2048 bytes for bfloat16) for a 1x32 tile (should be 64 bytes)
- Forgetting packed formats: `bfloat8_b` and `bfloat4_b` have different per-element sizes than `bfloat16`. These packed formats use `ttnn.Tile.get_tile_size(dtype)` rather than simple multiplication.
- Mismatching tile in `cb_scratch` vs the tensor's actual tile

### How to Avoid

- Read `page_size` and `tile_desc` from the input `CBHandle` when creating downstream scratch CBs
- Use `TileInfo.from_tensor(tensor)` to extract tile metadata consistently:

```python
from blaze.fused_program import TileInfo

tile_info = TileInfo.from_tensor(input_tensor)
# tile_info.tile, tile_info.data_format, tile_info.size, tile_info.desc
```

- Or propagate the CBHandle's stored value:

```python
downstream_op(
    input_cb=scratch,
    page_size=scratch.page_size,  # Use the handle's stored value
)
```

## 5. Tile Geometry Mismatches

### The Problem

Different tensors may use different tile shapes (e.g., `[1, 32]` for activations, `[32, 32]` for weights). If an op receives a `[1, 32]` activation handle but configures its compute phase for `[32, 32]` tiles, the compute RISC misinterprets the data layout.

### How It Manifests

PCC is very low (near 0.0) or the output is garbage. The shape of the output tensor appears correct, but the values are scrambled because tiles were read/written with wrong dimensions.

### Shard Shape Alignment

When creating CBs from views, the shard shape must be an exact multiple of the tile shape:

```python
# blaze/fused_program.py
def _view_total_size_from_shape(view, page_size, tile_desc):
    shard_h, shard_w = view.shard_shape
    tile_h, tile_w = tile_desc.height, tile_desc.width
    if shard_h % tile_h or shard_w % tile_w:
        raise ValueError(
            "cb_from_shared_l1_views: view shard_shape "
            f"{view.shard_shape} must be divisible by tile_shape "
            f"{(tile_h, tile_w)}."
        )
```

### How to Avoid

- Propagate tile information through `CBHandle.tile_desc`
- When creating scratch CBs, use the same tile as the input/output handles
- In the C++ Op, read tile shape from the CT args rather than hardcoding
- When converting between tile geometries (e.g., face-view to full tiles), use `dst_tile_info=` to override explicitly
- Pad shard dimensions to tile-aligned boundaries when needed

## 6. Scratch CB Naming: Triple Underscore Delimiter

### The Problem

Scratch CB names use a triple underscore (`___`) delimiter to scope them under the op prefix. This is defined in `BlazeOp`:

```python
class BlazeOp:
    CB_NAME_DELIMITER: str = "___"

    @classmethod
    def cb_name(cls, prefix: str, name: str) -> str:
        return f"{prefix}{cls.CB_NAME_DELIMITER}{name}"
```

This is different from the double underscore used for child prefixes (`CHILD_PREFIX_DELIMITER = "__"`).

If you name a scratch CB with single or double underscores, it will not be recognized by the scratch mapping system, causing scratch-to-tensor mapping (`scratch_mapping`) to fail silently.

### How It Manifests

Scratch CBs are not mapped to pre-allocated tensor memory, causing higher L1 usage or Metal allocation failures. In some cases, two ops both use `name="scratch"` without prefix scoping, and the dedup logic silently merges them even if the sizes differ.

### How to Avoid

Always use `BlazeOp.cb_name(prefix, "my_scratch")` to create scratch CB names:

```python
scratch_handle = f.cb_scratch(
    name=self.cb_name(prefix, "intermed"),
    num_pages=2,
    core_ranges=f.all_cores,
    data_format=data_format,
    tile=tile,
    page_size=page_size,
)
```

Never hardcode the delimiter. The `FusedProgram` tracks scratch CB names via `_cb_scratch_names_seen` and will warn on duplicates.

## 7. `cb_output()` vs `cb_from_tensor()` Confusion

### The Problem

`cb_output()` allocates a CB backed by a tensor AND marks it as the program's output tensor. `cb_from_tensor()` allocates a CB backed by a tensor without marking it as output. Using `cb_from_tensor()` for the output tensor means the program does not know which tensor to return from `run()`.

However, the docstring says "Prefer cb_from_tensor() in emit()" because `FusedProgram.run()` has auto-detection logic: if no explicit output is wired, it uses the last tensor-backed CB from `_io_tensors`:

```python
def run(self, output_tensor=None, noc_mode=None):
    if not self.program._output_tensors:
        if output_tensor is not None:
            ...
        elif self.program._io_tensors:
            eligible = [
                (cb_id, tensor)
                for cb_id, tensor in self.program._io_tensors
                if cb_id not in self._direct_address_cbs
            ]
            self.program._output_tensors.append(eligible[-1][1])
    return self.program.run()
```

### How It Manifests

If the last `cb_from_tensor()` call is not the output tensor, `run()` returns the wrong tensor. This is especially confusing when the test "passes" but returns stale data from a different buffer.

If you explicitly call `cb_output()` inside an `emit()` function, you may interfere with the auto-wiring logic when the op is used as an intermediate stage in a larger pipeline.

### How to Avoid

- **In `emit()` functions**: Use `cb_from_tensor()` for all tensors. Let the pipeline compositor or `f.run()` handle output designation.
- **At the FusedProgram level**: Use `cb_output()` when you know which tensor is the output, or pass `output_tensor` explicitly to `f.run(output_tensor=my_output)`.
- **For the compose path** (BlazeCompiler): The output tensor is passed to `compile()` explicitly.

## 8. Direct-Address vs FIFO Handle Confusion

### The Problem

`CBHandle` has an `access_mode` that is either `CBAccessMode.FIFO` (normal push/pop semantics) or `CBAccessMode.DIRECT_ADDRESS` (the CB's L1 address is read directly, no FIFO semantics).

Ops that consume FIFO handles call `cb_wait_front()` / `cb_pop_front()`. Ops that consume direct-address handles call `handle.buffer_address()` and read from that L1 address. Passing a direct-address handle to an op that expects FIFO semantics causes undefined behavior.

**Source reference** -- `cb_handle.py`:

```python
def require_fifo_handle(handle: CBHandle, op_name: str) -> CBHandle:
    if handle.is_direct_address:
        raise ValueError(
            f"{op_name} requires a FIFO CBHandle, but received a "
            "direct-address handle. Use an op that explicitly reads "
            "handle.buffer_address()."
        )
    return handle
```

This guard is called by `cb_from_tensor()` when passed a `CBHandle`:

```python
def cb_from_tensor(self, tensor, **kwargs) -> CBHandle:
    if isinstance(tensor, CBHandle):
        return require_fifo_handle(tensor,
                                    "FusedProgram.cb_from_tensor")
    ...
```

The constructor also validates that direct-address handles have a backing tensor:

```python
def __post_init__(self):
    if self.is_direct_address and self.backing_tensor is None:
        raise ValueError("Direct-address CBHandle requires a backing_tensor.")
```

### How It Manifests

`ValueError` at build time if the guard catches it. If you bypass the guard by extracting `cb_id` manually, the kernel hangs or produces garbage because FIFO state is not maintained for direct-address handles.

### How to Avoid

- Direct-address handles come from `cb_from_view()` on overlapped views or from explicit `access_mode=CBAccessMode.DIRECT_ADDRESS` construction
- Check `handle.is_fifo` before passing to downstream FIFO-consuming ops
- Read `buffer_address()` for direct-address handles instead of using the CB ID in FIFO operations
- Use `require_fifo_handle(handle, "MyOp")` at the start of every `emit()` that does FIFO operations

## 9. Pop Control and `supports_pop_control`

### The Problem

Some ops support deferred pop control, where the pop of a consumed CB is handled by a downstream op rather than the consuming op itself. This is declared in the OpSpec:

```python
register_op(OpSpec(
    name=cls.name,
    ...
    supports_pop_control=cls.supports_pop_control,
    pop_flags=list(cls.pop_flags),
))
```

If an op declares `supports_pop_control=True` but does not actually implement the pop control logic in its C++ code, the CB is never popped and the producer deadlocks on the next push.

### How It Manifests

Works correctly in isolation but hangs when composed with a preceding op that relies on deferred pop. The preceding op pushes to the CB, expects the consumer to pop, but the consumer skips the pop.

### How to Avoid

- Only declare `supports_pop_control=True` if the C++ Op explicitly handles the pop
- Test the op in a multi-phase pipeline, not just in isolation
- When using ops that support pop control, explicitly set the pop flags in CT args

## 10. `compose()` Prefix and Cores Passing

### The Problem

When a fused op's `compose()` method calls child op `emit()` functions, it must pass the correct `prefix` and `cores` to each child. The prefix namespaces CT args, and cores determines which cores the child op runs on.

The prefix convention uses `__` (double underscore) as the delimiter between parent and child:

```python
@classmethod
def child_prefix(cls, prefix: str, name: str) -> str:
    return f"{prefix}{cls.CHILD_PREFIX_DELIMITER}{name}" \
        if prefix else name

# CHILD_PREFIX_DELIMITER = "__"
```

### How It Manifests

If `compose()` passes the wrong prefix, the CT args are emitted under the wrong namespace. The C++ kernel reads from the wrong `ct_args::` struct and gets garbage values. If `compose()` passes `None` as the prefix, the child op's CT args collide with the parent's. If two fused ops both hardcode `prefix="mcast"`, their CT args and CB names collide.

### How to Avoid

- Always use `cls.child_prefix(prefix, "child_name")` for nested op prefixes
- Pass `prefix=` explicitly to every child `emit()` call
- Use `cls.cb_name(prefix, "scratch_name")` for scratch CB names under the fused op's scope

```python
@classmethod
def compose(cls, f, tensors, output, user_args):
    parent_prefix = user_args.get("prefix", cls.name)

    mcast_prefix = cls.child_prefix(parent_prefix, "mcast")
    matmul_prefix = cls.child_prefix(parent_prefix, "matmul")

    mcast_out = Mcast.emit(f, tensors["src"], prefix=mcast_prefix)
    Matmul.emit(f, mcast_out, tensors["weights"], prefix=matmul_prefix)
```

## 11. CB ID Limit Exhaustion

### The Problem

Blackhole hardware supports a maximum of 64 CBs:

```python
# blaze/cb_engine.py
MAX_CB_ID = 64
```

Complex fused ops that compose many micro-ops can exhaust this limit.

### How It Manifests

```
ValueError: CB ID limit exceeded (64): cannot assign CB for external input 'weights' on 'matmul_3'
```

The CB engine also warns when approaching the limit:

```python
if next_cb_id > self.max_cb_id - 8:
    warnings.warn(
        f"CB usage is high: {next_cb_id}/{self.max_cb_id} CBs assigned.",
        stacklevel=2,
    )
```

### How to Avoid

Enable compact mode which uses interval coloring to reuse CB IDs across non-overlapping lifetimes:

```python
engine = CBEngine(compact=True)
```

## 12. Op Registration Errors

### Missing `register()` Call

MicroOps must be registered before they can be used in graph construction or FusedProgram composition. If you define a MicroOp class but forget to call `MyOp.register()`, the op will not appear in the registry.

### Missing `compose()` on MicroOp

MicroOps require both `emit()` and `compose()`:

```python
class MicroOp(BlazeOp):
    def __init_subclass__(cls, **kwargs):
        if cls.emit is BlazeOp.emit:
            raise TypeError(f"{cls.__name__}: MicroOp must override emit()")
        if cls.compose.__func__ is BlazeOp.compose.__func__:
            raise TypeError(
                f"{cls.__name__}: MicroOp must override compose()"
            )
```

This raises `TypeError` at class definition time if either method is missing.

## Summary Checklist

Before submitting a new micro-op, verify:

- [ ] CT arg `name` fields match the C++ struct field names exactly
- [ ] CT arg `riscs` flags include all RISCs that read each argument
- [ ] All per-core flags are declared for all cores (including disabled=False cores)
- [ ] Scratch CB names use triple underscore delimiter via `BlazeOp.cb_name()`
- [ ] `page_size` and `tile` are consistent between producer and consumer CBs
- [ ] `num_pages` matches between CB descriptor and kernel loop iteration count
- [ ] `cb_from_tensor()` is used in `emit()` (not `cb_output()`); output wired at pipeline level
- [ ] Direct-address handles are not passed to FIFO-consuming ops
- [ ] `init_is_empty` and `teardown_is_empty` in `PhaseInfo` match the actual C++ implementation
- [ ] `compose()` passes correct `prefix` and `cores` to child ops via `child_prefix()`
- [ ] `balanced=False` is set on scratch CBs with unbalanced push/pop patterns
- [ ] `supports_pop_control` matches the C++ implementation
- [ ] Semaphore names are unique per sync point (namespaced with prefix)
- [ ] Emit order matches the data flow dependency order
- [ ] Push/pop balance is correct on every core for every CB
- [ ] A CT arg schema test exists in `tests/blaze/infra/`
- [ ] `register()` is called after class definition
