# 8.2 Debugging Tools

## Environment Variables Overview

| Variable | Purpose | Values |
|----------|---------|--------|
| `BLAZE_DEBUG_KERNELS` | Inject DPRINT phase markers into generated kernels | `0` (off), `1`/`all` (all RISCs), or comma-separated RISC names |
| `BLAZE_L1_PROFILE` | Print CB stats (type, L1 address, dtype, tile, page_size) before every program execution | Any truthy value |
| `BLAZE_DISABLE_TENSOR_CB_SHARE` | Disable cross-tensor format-key CB sharing | `1` to disable |
| `BLAZE_DISABLE_TEMPORAL_REUSE` | Disable temporal CB reuse in multi-phase programs | `1` to disable |
| `BLAZE_EXPORT` | Auto-export visualizer JSON on every `compile()` | `1` to enable |
| `BLAZE_EXPORT_PATH` | Override output directory for auto-export | Directory path (default: `generated/viz`) |
| `TT_METAL_DPRINT_CORES` | Filter which cores produce DPRINT output | Core range syntax |
| `TT_METAL_CACHE` | Root for JIT cache and profiler logs | Directory path |

## BLAZE_DEBUG_KERNELS: DPRINT Phase Markers

Setting `BLAZE_DEBUG_KERNELS` causes the kernel codegen to inject DPRINT markers at the start and end of every phase in the generated kernel. This is the primary tool for diagnosing which phase a hang occurs in.

### How It Works

The codegen reads the environment variable in `kernel_codegen.py` via `_parse_debug_riscs()`:

```python
# From blaze/kernel_codegen.py

_RISC_GUARDS = {
    "brisc": "defined(COMPILE_FOR_BRISC)",
    "ncrisc": "defined(COMPILE_FOR_NCRISC)",
    "trisc": "defined(COMPILE_FOR_TRISC)",
    "trisc0": "defined(COMPILE_FOR_TRISC) && COMPILE_FOR_TRISC == 0",
    "trisc1": "defined(COMPILE_FOR_TRISC) && COMPILE_FOR_TRISC == 1",
    "trisc2": "defined(COMPILE_FOR_TRISC) && COMPILE_FOR_TRISC == 2",
}

def _parse_debug_riscs(env_val: str) -> tuple[bool, str | None]:
    val = env_val.strip().lower()
    if not val or val == "0":
        return False, None
    if val in ("1", "all"):
        return True, None         # All RISCs
    riscs = [r.strip() for r in val.split(",") if r.strip()]
    guards = []
    for r in riscs:
        if r not in _RISC_GUARDS:
            raise ValueError(
                f"Unknown RISC in BLAZE_DEBUG_KERNELS: {r!r} "
                f"(valid: {', '.join(sorted(_RISC_GUARDS))})"
            )
        guards.append(_RISC_GUARDS[r])
    # Single guard or OR-joined multi-guard
    if len(guards) == 1:
        return True, guards[0]
    return True, " || ".join(f"({g})" for g in guards)
```

### Usage

```bash
# All RISCs print phase markers
BLAZE_DEBUG_KERNELS=1 pytest tests/blaze/micro-ops/common/test_mcast.py -v

# Only NCRISC prints
BLAZE_DEBUG_KERNELS=ncrisc pytest ...

# Only TRISC pack processor (trisc2) prints
BLAZE_DEBUG_KERNELS=trisc2 pytest ...

# BRISC and NCRISC print, TRISC silent
BLAZE_DEBUG_KERNELS=brisc,ncrisc pytest ...
```

### What the Generated Kernel Looks Like

With `BLAZE_DEBUG_KERNELS=ncrisc`, the codegen inserts guarded DPRINT statements:

```cpp
// Auto-generated by blaze.kernel_codegen -- do not edit manually.

#include "blaze/kernels/ops.hpp"
#include "blaze/kernels/kernel_op_api.hpp"
#include "blaze/kernels/kernel_utils.hpp"
#if defined(COMPILE_FOR_NCRISC)
#include "api/debug/dprint.h"
#endif

using ActMcast = blaze::Mcast::Op<ct_args::act_mcast>;
using Matmul = blaze::Matmul::Op<ct_args::matmul>;

void kernel_main() {
#if defined(COMPILE_FOR_TRISC)
    deepseek_compute_kernel_init();
#endif

#if defined(COMPILE_FOR_NCRISC)
    DPRINT << "[PHASE] ACT_MCAST start" << ENDL();
#endif
    ActMcast act_mcast;
    act_mcast.init();
    {
        DeviceZoneScopedN("ACT_MCAST");
        act_mcast();
    }
#if defined(COMPILE_FOR_NCRISC)
    DPRINT << "[PHASE] ACT_MCAST done" << ENDL();
#endif

#if defined(COMPILE_FOR_NCRISC)
    DPRINT << "[PHASE] MATMUL start" << ENDL();
#endif
    {
        DeviceZoneScopedN("MATMUL");
        Matmul matmul;
        matmul.init();
        matmul();
        matmul.teardown();
    }
#if defined(COMPILE_FOR_NCRISC)
    DPRINT << "[PHASE] MATMUL done" << ENDL();
#endif
}
```

The debug kernel file name gets a `_debug` suffix to avoid caching conflicts with non-debug runs. From `generate_kernel_file()`:

```python
if os.getenv("BLAZE_DEBUG_KERNELS", "0") != "0":
    out_path = out_dir / f"{label}_{content_hash}_debug.cpp"
```

### Combining with TT_METAL_DPRINT_CORES

To limit DPRINT output to specific cores (critical on large grids):

```bash
BLAZE_DEBUG_KERNELS=1 TT_METAL_DPRINT_CORES="0,0" pytest ...
```

**Important note on DPRINT**: DPRINTs are slow and can change timing enough to mask or introduce timing-sensitive bugs. Use targeted RISC filtering and core filtering to minimize the overhead.

### Valid RISC Names

| Name | Meaning |
|------|---------|
| `brisc` | Data movement RISC 0 |
| `ncrisc` | Data movement RISC 1 (NOC reader) |
| `trisc` | All three compute RISCs |
| `trisc0` | Unpack processor |
| `trisc1` | Math processor |
| `trisc2` | Pack processor |

**When to use each**: Use `BLAZE_DEBUG_KERNELS=ncrisc` to debug data movement issues (reader/writer). Use `BLAZE_DEBUG_KERNELS=trisc` to debug compute. Use `BLAZE_DEBUG_KERNELS=1` to see all phases on all RISCs when you have no hypothesis about the cause.

## BLAZE_L1_PROFILE: CB and Kernel Stats

Setting `BLAZE_L1_PROFILE` to any truthy value causes every `CompiledProgram.run()` and `MeshCompiledProgram.run()` to print CB statistics **before** dispatching the program to the device. The gate is checked in `compiler.py`:

```python
# From blaze/compiler.py, CompiledProgram.run()
def run(self):
    if os.environ.get("BLAZE_L1_PROFILE"):
        from blaze.l1_profile import print_cb_stats
        print_cb_stats(self)
    return _run_program(...)
```

Note: `print_cb_stats()` is called before `_run_program()` returns, so the stats are printed before execution completes. For tensor-backed CBs, the L1 address comes from `buffer->address() + address_offset` from the backing tensor buffer (which is known at allocation time). For scratch CBs, the address reported is `metal-managed` because the actual address is assigned by Metal during program dispatch.

### `print_cb_stats()`

Reports per-CB descriptor information:

```
cb_stats: 5 CB descriptor(s)
  cb[0] type=tensor_backed buffer_index=0 total_size_B=2048 l1_addr=0x12000
    [dtype=DataType.BFLOAT16 page_size_B=64 tile=1x32]
    core_ranges={[(x=10,y=9) - (x=10,y=9)]}
  cb[1] type=tensor_backed buffer_index=1 total_size_B=65536 l1_addr=0x15000
    [dtype=DataType.BFLOAT16 page_size_B=2048 tile=32x32]
    core_ranges={[(x=0,y=0) - (x=9,y=9)]}
  cb[2] type=scratch buffer_index=2 total_size_B=2048 l1_addr=metal-managed
    [dtype=DataType.BFLOAT16 page_size_B=64 tile=1x32]
    core_ranges={[(x=0,y=0) - (x=10,y=9)]}
  cb[3] type=aliased buffer_index=3,4 total_size_B=4096 l1_addr=0x18000
    [dtype=DataType.BFLOAT16 page_size_B=64 tile=1x32;
     dtype=DataType.BFLOAT8_B page_size_B=36 tile=1x32]
    core_ranges={[(x=0,y=0) - (x=9,y=9)]}
```

CB types reported:
- **`tensor_backed`**: Backed by a ttnn.Tensor buffer; `l1_addr` is the buffer's address (possibly with an `address_offset`)
- **`aliased`**: Multiple format descriptors sharing one L1 buffer; all `buffer_index` values listed
- **`scratch`**: No backing tensor; address assigned by Metal at dispatch time (`l1_addr=metal-managed`)

### `print_kernel_stats()`

Reports kernel descriptor information:

```
kernel_stats: 3 kernel(s)
  kernel[0] role=brisc stem=my_kernel_abc123 core_ranges={...}
  kernel[1] role=ncrisc stem=my_kernel_abc123 core_ranges={...}
  kernel[2] role=trisc stem=my_kernel_abc123 core_ranges={...}
```

### Using `print_cb_stats` and `print_kernel_stats` Manually

You can call these functions directly in test code or debugging scripts without setting the environment variable:

```python
from blaze.l1_profile import print_cb_stats, print_kernel_stats

program = f.build()          # CompiledProgram
print_cb_stats(program)      # Per-CB L1 layout
print_kernel_stats(program)  # Per-kernel role and core ranges
program.run()
```

### CB Name Extraction

`extract_cb_names()` maps CB IDs to human-readable names from the shadow graph:

```python
from blaze.l1_profile import extract_cb_names

names = extract_cb_names(program)
# {"0": "act_mcast.out", "1": "matmul.in1", "2": "matmul.out", ...}
```

### Log File Output

Both functions also write to a timestamped log file:

```
$TT_METAL_CACHE/logs/blaze_l1_profiler_log_2026-05-13_143022.log
```

Falls back to `~/.cache/tt-metal-logs/` if `TT_METAL_CACHE` is unset. The path is printed to stderr on first access:

```
[blaze.l1_profile] logging to /home/user/.cache/tt-metal-logs/blaze_l1_profiler_log_2026-05-13_143022.log
```

### Usage

```bash

```

## BLAZE_DISABLE_TENSOR_CB_SHARE: Bisection Escape Hatch

When debugging suspected CB sharing bugs, set `BLAZE_DISABLE_TENSOR_CB_SHARE=1` to disable the format-key cross-tensor sharing optimization in `FusedProgram`. This forces each `cb_from_tensor()` call to allocate a unique CB ID, even when two tensors have matching `(data_format, tile_shape, page_size)` and disjoint core ranges.

From `fused_program.py`:

```python
def _tensor_cb_share_disabled() -> bool:
    """Bisection escape hatch: BLAZE_DISABLE_TENSOR_CB_SHARE=1
    disables format-key (cross-tensor) sharing on tensor-backed
    allocator surfaces. Same-tensor view sharing in cb_from_view
    is unaffected."""
    import os
    return bool(os.environ.get("BLAZE_DISABLE_TENSOR_CB_SHARE"))
```

This is checked in `_try_reuse_cb_id()`:

```python
def _try_reuse_cb_id(self, data_format, tile_desc, core_ranges,
                      page_size=None) -> int | None:
    if _tensor_cb_share_disabled() or core_ranges is None:
        return None
    ...
```

**When to use**: If a test passes with `BLAZE_DISABLE_TENSOR_CB_SHARE=1` but fails without it, the bug is in the CB sharing logic (mismatched format keys, non-disjoint grids treated as disjoint, etc.).

Note: same-tensor view sharing via `cb_from_view()` is unaffected by this flag -- it only controls cross-tensor format-key sharing.

## BLAZE_DISABLE_TEMPORAL_REUSE: Multi-Phase Bisection

Setting `BLAZE_DISABLE_TEMPORAL_REUSE=1` disables temporal CB reuse in the multi-phase CB compaction pass. When active, scratch CBs whose lifetimes do not overlap will still get distinct IDs instead of being recolored.

From `blaze/cb_reconfig_builder.py`:

```python
allow_temporal = not os.environ.get("BLAZE_DISABLE_TEMPORAL_REUSE")
```

**When to use**: If a multi-phase program hangs or produces wrong results and you suspect temporal CB reuse is conflicting, set this to force all CBs to have unique IDs.

## CT Arg Inspection

### Via `compile_engines()`

For graph-based tests, use `blaze.compile_engines()` to inspect the CT arg tuples that would be generated:

```python
import blaze
from blaze import ExternalTensor

with blaze.fuse() as ctx:
    blaze.matmul(ExternalTensor("act"), ExternalTensor("weights"),
                 grid="compute")

result = blaze.compile_engines(ctx.graph)

# Inspect per-RISC CT arg tuples
for risc in ("ncrisc", "trisc", "brisc"):
    print(f"\n{risc} CT args:")
    for name, value in result.ct_tuples.get(risc, []):
        print(f"  {name} = {value}")
```

The `EngineResult` object has:
- `ct_tuples`: `dict[str, list[tuple[str, int]]]` -- per-RISC `(name, value)` pairs
- `cb_assignments`: `dict[str, CBAssignment]` -- CB engine results
- `cb_by_node`: `dict[str, dict[str, int]]` -- `node_id -> {port_name: cb_id}`
- `sem_assignments`: `list[SemAssignment]` -- semaphore engine results
- `sem_by_node`: `dict[str, dict[str, int]]` -- `node_id -> {protocol: sem_id}`

### Via CT Arg Schema

For programmatic inspection of what compile-time arguments an op expects:

```python
from blaze.ct_args import get_ct_schema

schema = get_ct_schema("matmul")
trisc_args = schema.args_for_risc("trisc")
for arg in trisc_args:
    print(f"  {arg.name}: type={arg.type}, source={arg.source}")
```

### Via FusedProgram Shadow Graph

For FusedProgram-based tests, the shadow graph captures CT arg values as they are emitted. You can compare engine-derived values against manually-emitted values:

```python
f = FusedProgram(kernel=None, device=device, name="test")
# ... emit ops ...
diffs = f._verify_engine_equivalence()
assert diffs["ok"], f"Engine/manual mismatch: {diffs}"
```

### Strict Mode in CTArgEngine

For graph-API compilation, the `CTArgEngine` validates CT arg tuples against schemas registered by each op. When `strict=True` (not the default -- must be passed explicitly), any CT arg declared in the schema that cannot be resolved triggers an error rather than a warning:

```python
ct_tuples = CTArgEngine().generate_tuples(
    graph, cb_for_ct, sem_for_ct,
    grid_context=grid_context,
    user_overrides=merged_overrides,
    strict=True,  # raises on missing args
)
```

## Generated Kernel Inspection

### Accessing the Generated Source

After building a FusedProgram, the generated kernel source can be read via `program.generated_kernel`:

```python
f = FusedProgram(kernel=None, device=device, name="test")
# ... emit ops ...
compiled = f.build()

# The BlazeProgram property (not the CompiledProgram)
source = f.program.generated_kernel
if source:
    print(source)
```

To see the generated C++ kernel without running it on device:

```python
from blaze.kernel_codegen import generate_kernel
import blaze

with blaze.fuse() as ctx:
    blaze.mcast(blaze.ExternalTensor(name="input"))

kernel_source = generate_kernel(ctx.graph)
print(kernel_source)
```

### Generated Kernel Files on Disk

Generated kernels are written to `generated/kernels/` (relative to the Blaze root) with content-hash filenames for caching:

```
generated/kernels/
    test_mcast_a1b2c3d4e5f6.cpp
    shared_expert_9f8e7d6c5b4a.cpp
    shared_expert_9f8e7d6c5b4a_debug.cpp  # when BLAZE_DEBUG_KERNELS is set
```

The hash is derived from the kernel source content, so identical graphs produce the same file (skipping regeneration). The write uses atomic write-to-temp + rename so concurrent processes never see a partial file. From `generate_kernel_file()`:

```python
source = generate_kernel(graph)
content_hash = hashlib.sha256(source.encode()).hexdigest()[:12]
label = name or "generated"
out_path = out_dir / f"{label}_{content_hash}.cpp"
```

The generated kernel contains:
1. `#include` directives for `ops.hpp`, `kernel_op_api.hpp`, `kernel_utils.hpp`
2. Type aliases mapping op prefixes to C++ Op structs (e.g., `using ActMcast = blaze::Mcast::Op<ct_args::act_mcast>;`)
3. `kernel_main()` with the phase execution sequence
4. `DeviceZoneScopedN` markers for Tracy profiling
5. Init/operator/teardown lifecycle calls for ops that have them
6. DPRINT markers if `BLAZE_DEBUG_KERNELS` is set

### `named_args_generated.h` in JIT Cache

When Metal JIT-compiles the kernel, it generates a `named_args_generated.h` header file containing the `ct_args::` struct definitions. These are the structs that the kernel's Op templates consume.

Location: `~/.cache/tt-metal-cache/<kernel_hash>/kernels/<kernel_name>/<risc>/named_args_generated.h`

Each RISC gets its own header with only the args routed to that processor (`brisc/`, `ncrisc/`, `trisc0/`, `trisc1/`, `trisc2/`).

To find the generated header for a specific kernel:

```bash
find ~/.cache/tt-metal-cache -name "named_args_generated.h" -newer /tmp/my_test_start_marker | head -20
```

This header contains struct definitions like:

```cpp
namespace ct_args {
namespace act_mcast {
    static constexpr uint32_t in0 = 0;
    static constexpr uint32_t num_tiles = 16;
    static constexpr uint32_t noc_x = 12;
    static constexpr uint32_t noc_y = 9;
}
namespace matmul {
    static constexpr uint32_t in0 = 0;
    static constexpr uint32_t in1 = 1;
    static constexpr uint32_t out = 2;
    static constexpr uint32_t k_num_tiles = 16;
    static constexpr uint32_t out_w_per_core = 2;
}
}
```

This is useful for debugging CT arg name mismatches -- if the Python side emits `("activation_cb", 5)` but the C++ struct expects `src`, the name mismatch causes a compile error or silent data corruption. Cross-reference CB IDs, tile counts, NOC coordinates, and semaphore addresses against expected values.

### Shadow Graph

Every `FusedProgram` maintains a shadow graph (`_shadow_graph`) that records the op composition as it happens. This graph is used for:
- Kernel codegen (when `kernel=None`)
- Visualizer export
- L1 profiler CB name extraction
- CT arg schema validation

Access it after compilation:

```python
compiled = compiler.compile(graph, tensors, output_tensor, ...)
shadow = compiled.shadow_graph
# shadow.nodes, shadow.edges are available for inspection
```

## Visualizer

The `blaze.visualize()` function generates an interactive HTML visualization of a Blaze graph.

```python
import blaze
from blaze import ExternalTensor

with blaze.fuse() as ctx:
    blaze.mcast(ExternalTensor("src"))

# Generate and open in browser
blaze.visualize(ctx.graph, name="my_pipeline")
```

From `visualizer.py`:

```python
def visualize(
    graph: BlazeGraph,
    engine_result: EngineResult | None = None,
    grid_config: GridConfig | None = None,
    *,
    name: str = "",
    tensors: dict | None = None,
    output_dir: str | pathlib.Path | None = None,
    open_browser: bool = True,
) -> pathlib.Path:
```

If `engine_result` is not provided, `compile_engines()` is called automatically. The `tensors` parameter accepts a dict of `{tensor_name: ttnn_tensor}` for extracting shape, dtype, memory config, and shard spec metadata into the visualization.

**What the visualizer shows**:
- Graph topology: nodes (ops), edges (data flow), external inputs/outputs
- Core grid map: which cores run which op phases, sender/receiver roles
- CB details: ID, type, dtype, page size, tile shape, L1 address
- Semaphore events: protocol, direction, associated nodes
- Phase timeline: execution order, DeviceZoneScoped regions
- CT arg values: per-RISC named compile-time arguments
- Runtime args: per-core runtime argument values (when captured)
- Op nodes with color-coded types (red=mcast, green=matmul, purple=gather, etc.)

### Auto-Export via BLAZE_EXPORT

Setting `BLAZE_EXPORT=1` causes every `BlazeCompiler.compile()` call to auto-export visualizer JSON:

```bash
BLAZE_EXPORT=1 pytest tests/blaze/fused_ops/mlp/test_dense_mlp.py -s
# Prints: [BLAZE_EXPORT] generated/viz/dense_mlp_viz.json
# Also prints a URL to the hosted visualizer page
```

From `compiler.py`:

```python

    self._auto_export(graph, program, tensors)
```

The export directory can be overridden with `BLAZE_EXPORT_PATH`:

```bash
BLAZE_EXPORT=1 BLAZE_EXPORT_PATH=/tmp/my_viz pytest ...
```

## Tracy Profiler: DeviceZoneScopedN Markers

The generated kernel wraps every phase in a `DeviceZoneScopedN` scope, which produces Tracy profiler markers when Tracy is enabled:

```cpp
{
    DeviceZoneScopedN("ACT_MCAST");
    ActMcast act_mcast;
    act_mcast.init();
    act_mcast();
    act_mcast.teardown();
}
```

The zone name is the uppercased prefix of the phase (e.g., `ACT_MCAST`, `MATMUL`, `GATHER`). These markers appear in Tracy's timeline view and allow you to measure per-phase execution time on each core.

For noop kernel nodes (where `noop_kernel=True` is set on the graph node), a `_NOOP` zone is emitted instead:

```cpp
{
    DeviceZoneScopedN("ACT_MCAST_NOOP");
}
```

### Using Tracy

To profile with Tracy:

1. Build tt-metal with Tracy support enabled
2. Run your test/program -- Tracy captures the `DeviceZoneScopedN` zones
3. Open the Tracy profiler and load the trace
4. Look for `ACT_MCAST`, `MATMUL`, `GATHER` etc. in the timeline view

The zone names correspond directly to the op prefixes you pass to `emit()`. For example, `Mcast.emit(f, ..., prefix="act_mcast")` generates the zone `ACT_MCAST`.

With Tracy you can:
- Identify which phase is taking the most time
- Spot phases that are unexpectedly long (potential hangs or serialization points)
- Correlate device-side timing with host-side dispatch
- Compare per-phase timing across different core positions to detect load imbalance

## The Capture Plugin (conftest_capture.py)

The `conftest_capture.py` pytest plugin provides deep post-mortem analysis of CB layouts. It hooks `ttnn.generic_op` to intercept the `MeshProgramDescriptor` and `io_tensors` after each test.

### Activation

```bash
PYTHONPATH=tests/blaze:$PYTHONPATH pytest -p conftest_capture tests/blaze/micro-ops/common/test_matmul.py -v -s
```

### What It Provides

- Full CB blueprint JSON dump
- Overlap analysis: which CBs share L1 address ranges (uses union-find on address intervals via `_build_overlap_groups`)
- Tensor-to-CB mapping table
- Multi-device comparison (checks if all devices have identical CB layouts)
- For Blitz programs: auto-detects and decodes the CbReconfig tensor to get real post-reconfig CB addresses

### Output Location

Defaults to `/tmp/blaze_capture/`. Override with `BLAZE_CAPTURE_DIR`. Set `BLAZE_CAPTURE_NAMED_DIR` for a secondary copy.

## Debugging Workflow Summary

1. **Test fails with wrong output**: Use `comp_pcc()` to quantify the error. Check if the PCC is close (rounding/fidelity issue) or far off (structural bug). Use `BLAZE_L1_PROFILE=1` to verify CB shapes and addresses. Inspect `named_args_generated.h` for CT arg correctness.

2. **Test hangs**: Set `BLAZE_DEBUG_KERNELS=1` and `TT_METAL_DPRINT_CORES=0,0` to identify which phase the hang occurs in. Check the section on common hang causes (Section 8.3).

3. **Suspected CB sharing bug**: Set `BLAZE_DISABLE_TENSOR_CB_SHARE=1`. If the test passes, the bug is in format-key sharing.

4. **Suspected multi-phase reconfig bug**: Set `BLAZE_DISABLE_TEMPORAL_REUSE=1`. If the test passes, temporal CB reuse is conflicting.

5. **CT arg mismatch**: Use `blaze.compile_engines(ctx.graph).ct_tuples` to inspect generated CT args. Check `named_args_generated.h` in `~/.cache/tt-metal-cache/` for the C++ struct definition. Verify Python prefix matches C++ struct names.

6. **Visual pipeline debugging**: Use `blaze.visualize(ctx.graph)` or `BLAZE_EXPORT=1` to generate an interactive graph diagram showing nodes, edges, CB assignments, and CT args.

7. **L1 memory pressure**: Set `BLAZE_L1_PROFILE=1` to see per-CB sizes and addresses. Look for CBs that are unexpectedly large or overlapping. Use the capture plugin for overlap analysis.

8. **Performance analysis**: Use Tracy with `DeviceZoneScopedN` zones to identify which phases are bottlenecks.

### Quick Reference Table

| Symptom | First Tool to Try |
|---------|-------------------|
| Wrong numerical output | `comp_pcc()` with golden, then inspect `named_args_generated.h` |
| Program hangs | `BLAZE_DEBUG_KERNELS=1` to identify stalling phase |
| CB address conflicts | `BLAZE_L1_PROFILE=1` + capture plugin for overlap analysis |
| Unexpected CB sharing | `BLAZE_DISABLE_TENSOR_CB_SHARE=1` to bisect |
| Multi-phase reconfig bug | `BLAZE_DISABLE_TEMPORAL_REUSE=1` to bisect |
| Graph topology confusion | `blaze.visualize()` or `BLAZE_EXPORT=1` |
| Per-phase timing | Tracy with `DeviceZoneScopedN` zones |
| CT arg mismatch | Inspect generated kernel + `strict=True` errors from CTArgEngine |
