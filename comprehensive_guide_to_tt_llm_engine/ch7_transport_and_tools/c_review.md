## Structural Suggestions

### 1. Consider merging loopback_kernel.md and loopback_microbenchmark.md
- The kernel file (341 lines) and microbenchmark file (464 lines) describe two halves of a single system. The kernel is only meaningful in the context of the benchmark, and the benchmark is only meaningful with the kernel.
- Combined they would be ~750 lines after deduplication (~700 lines), which is still manageable.
- **Counter-argument**: The current split mirrors the two-process architecture (device-side vs host-side), which is pedagogically clean. The separation also lets readers skip the kernel details if they only care about the scheduler API.
- **Verdict**: Keep separate. The split is architecturally motivated. Add Level-N labels at the top of each file per Missing Cross-References #2.

### 2. Move CMake build details from mooncake_transport.md to Ch8
- The "Build System Gating" section (87 lines) in mooncake_transport.md overlaps with Ch8's domain. The Mooncake-specific flag hierarchy and feature flags belong here, but the vendored dependency wiring is build-system infrastructure.
- **Recommendation**: Split the section. Keep flag hierarchy + feature flags (~30 lines) in mooncake_transport.md. Move vendored dependency wiring + post-build patching (~55 lines) to Ch8, or add a cross-reference if Ch8 already covers it.

### 3. Standardize metric definitions
- **loopback_microbenchmark.md** lines 348-383: Defines TTFT, ITL, TPOT, throughput metrics
- **deepseek_inference_runner.md** lines 533-543: Metric Definitions table with formulas
- The runner's table is a superset of the connector's definitions (same metrics + spec decode stats).
- **Recommendation**: In the connector file, reference the runner's metric definitions table: "Metric definitions (TTFT, ITL, TPOT, throughput) are defined in [deepseek_inference_runner.md](deepseek_inference_runner.md#metric-definitions)." Keep the connector's brief prose descriptions of how it collects each metric (the collection logic differs).

### 4. Add a "Quick Start" section to deepseek_inference_runner.md
- The runner is flagged as the "Production user" entry point in index.md, but it launches directly into "Two Operating Modes" without a minimal invocation example.
- The CLI examples in the "Two Operating Modes" section partially serve this purpose, but a 3-line "Quick Start" before the operating modes section would improve scanability.
