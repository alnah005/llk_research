# The DeepSeek Inference Runner

[Prev: Loopback Microbenchmark](loopback_microbenchmark.md) | [Next: Mooncake Transport](mooncake_transport.md)

---

The DeepSeek inference runner (`tools/decode/deepseek_inference_runner.cpp`) is the production-oriented benchmark and inference tool for DeepSeek models running on Tenstorrent hardware. It connects to a pre-launched DeepSeek model pipeline via H2D/D2H sockets, instantiates a `DecodeScheduler` with the DeepSeek wire format, and drives multi-user multi-turn inference sessions with comprehensive metrics collection. Unlike the loopback microbenchmark (which uses a trivial echo kernel), this tool operates against a real model and produces real text output when running in tokenizer mode.

**Source file:** `tools/decode/deepseek_inference_runner.cpp`

---

## Two Operating Modes

### Synthetic Mode

Generates dummy token IDs for ISL-controlled benchmarking. Provides exact control over prompt length without tokenization overhead.

```bash
deepseek_inference_runner --h2d-socket-id deepseek_h2d --d2h-socket-id deepseek_d2h \
    --prompt-len 128 --max-decode 64 --max-turns 5 --num-users 16
```

Prompts are generated by `make_prompt(len, seed)` (lines 228-234):

```cpp
std::vector<uint32_t> make_prompt(uint32_t len, uint32_t seed) {
    std::vector<uint32_t> prompt(len);
    for (uint32_t i = 0; i < len; i++) {
        prompt[i] = seed + i + 2;
    }
    return prompt;
}
```

Each user and turn receives a distinct seed (`user_idx * max_turns + turn_idx`), producing non-overlapping token ID sequences.

### Tokenizer Mode

Uses a HuggingFace-format `tokenizer.json` to encode real prompts and decode output tokens. Supports two input methods:

**Single prompt (applied to all users):**
```bash
deepseek_inference_runner --h2d-socket-id deepseek_h2d --d2h-socket-id deepseek_d2h \
    --tokenizer /path/to/tokenizer.json --prompt "Explain quantum computing" \
    --max-decode 256 --max-turns 1
```

**Per-user, per-turn prompts from JSON file:**
```bash
deepseek_inference_runner --h2d-socket-id deepseek_h2d --d2h-socket-id deepseek_d2h \
    --tokenizer /path/to/tokenizer.json --prompts-file prompts.json \
    --max-decode 256
```

The prompts file supports two formats:
- **2D array** (multi-turn): `[["u0t0","u0t1",...], ["u1t0","u1t1",...], ...]`
- **Flat array** (single-turn per user): `["prompt0", "prompt1", ...]`

When using `--prompts-file`, `--num-users` and `--max-turns` are inferred from the file dimensions if not explicitly provided. Each user can have a different number of turns, controlled by a per-user `turn_limits` vector (lines 528-534).

## DeepSeek Chat Template

All prompts in tokenizer mode are wrapped with the DeepSeek R1/V3 chat template before encoding (lines 316-323):

```cpp
std::string apply_deepseek_chat_template(const std::string& user_content) {
    static const std::string BOS = "<\xef\xbd\x9c" "begin\xe2\x96\x81" "of\xe2\x96\x81"
                                    "sentence\xef\xbd\x9c>";
    static const std::string USER_TAG = "<\xef\xbd\x9c" "User\xef\xbd\x9c>";
    static const std::string ASST_TAG = "<\xef\xbd\x9c" "Assistant\xef\xbd\x9c>";
    return BOS + USER_TAG + user_content + ASST_TAG;
}
```

The special tokens use Unicode fullwidth bar U+FF5C and lower-one-eighth-block U+2581, encoded as UTF-8 byte sequences. The rendered template format is: `<|begin_of_sentence|><|User|>{content}<|Assistant|>`. The assistant tag at the end is the `add_generation_prompt=True` convention -- it primes the model to begin generating from the assistant role.

## Tokenizer Integration

The runner wraps `tokenizers_cpp` (a C++ binding for HuggingFace's Rust-based tokenizers crate) in a simple `Tokenizer` struct (lines 331-353):

```cpp
struct Tokenizer {
    std::unique_ptr<tokenizers::Tokenizer> tok;

    void load(const std::string& json_path) {
        auto blob = read_file_bytes(json_path);
        tok = tokenizers::Tokenizer::FromBlobJSON(blob);
    }

    std::vector<uint32_t> encode(const std::string& text) const {
        auto ids = tok->Encode(text);
        return {ids.begin(), ids.end()};
    }

    std::string decode(const std::vector<uint32_t>& ids) const {
        std::vector<int> signed_ids(ids.begin(), ids.end());
        return tok->Decode(signed_ids);
    }
};
```

Building with tokenizer support requires Rust >= 1.80 via rustup; the CMake build conditionally fetches `tokenizers-cpp` when cargo is available. The `DS_NO_TOKENIZER=ON` flag builds without tokenizer support (synthetic mode only).

### EOS and Thinking Token Detection

In tokenizer mode, three special token IDs are auto-detected from the tokenizer (lines 486-509):

**EOS token:** The DeepSeek end-of-sentence token `<|end_of_sentence|>` is encoded and must resolve to exactly one token ID. If encoding produces multiple IDs, the default `pm::DEFAULT_EOS_TOKEN` (1) is kept as a fallback.

**Thinking tokens:** `<think>` and `</think>` bracket the reasoning phase. Both must encode to single tokens; otherwise the thinking-phase gate is disabled with a warning. When the gate is disabled (token IDs remain `EMPTY_TOKEN`), the scheduler treats every decode token as outside the thinking phase, falling back to strict acceptance as described in [Ch4 thinking_phase_and_relaxed_acceptance.md](../ch4_speculative_decode/thinking_phase_and_relaxed_acceptance.md).

## Scheduler Configuration

The DecodeScheduler is configured with DeepSeek-specific settings (lines 594-602):

```cpp
pm::DecodeScheduler mgr(
    pl::SocketConfig{h2d_id, d2h_id, connect_timeout_ms,
                     /*use_deepseek_md_format=*/true},
    pm::SchedulerParams{
        .max_users = effective_batch_size,
        .eos_token = eos_token_id,
        .think_open_token_id = think_open_token_id,
        .think_close_token_id = think_close_token_id,
    });
```

Key differences from the loopback connector:
- `use_deepseek_md_format=true` selects the DeepSeek wire format (see [Ch5 wire_format.md](../ch5_pipeline_and_wire_format/wire_format.md))
- `max_users` is set to `effective_batch_size` (not total `num_users`) because the scheduler is reused across batches
- EOS and thinking token IDs are forwarded from tokenizer detection

## Batching Architecture

The runner supports processing users in batches via `--batch-size N`. When batching is enabled:

```
+----- Batch 1 -----+   +----- Batch 2 -----+   +----- Batch 3 -----+
| Users 0..N-1      |   | Users N..2N-1     |   | Users 2N..end     |
| Allocate slots     |   | Allocate slots    |   | Allocate slots    |
| Run all turns      |   | Run all turns     |   | Run all turns     |
| CANCEL (fire&forget)|   | CANCEL (fire&forget)|  | CANCEL + drain   |
+----+---------------+   +----+---------------+   +-------------------+
     |                        |
     |  Prior-batch CANCELs   |
     |  drained on demand by  |
     |  next batch's allocator|
     +------------------------+
```

### Batch Size Resolution

```cpp
// Lines 572-580
const uint32_t effective_batch_size =
    (batch_size_arg > 0) ? std::min(batch_size_arg, num_users) : num_users;
const uint32_t num_batches =
    (num_users + effective_batch_size - 1) / effective_batch_size;
```

If `--batch-size` is 0 or omitted, all users run in a single batch (the historical behavior). Values larger than `--num-users` are clamped with a warning.

### Pipelined Teardown: allocate_one_with_eviction

The key innovation in the batching design is **pipelined inter-batch teardown**. At the end of each batch, CANCEL requests are issued fire-and-forget, and the cancelled slot IDs are added to an `evictable` list. The next batch's slot allocation drains these CANCEL acks on demand.

The `allocate_one_with_eviction` function (lines 191-226) implements this:

```cpp
uint32_t allocate_one_with_eviction(
    pm::DecodeScheduler& mgr,
    uint32_t& req_id,
    std::vector<uint32_t>& evictable) {
    while (true) {
        // Try ALLOCATE
        const uint32_t alloc_req_id = req_id++;
        pm::ISRequest req{};
        req.type = pm::RequestType::ALLOCATE;
        req.request_id = alloc_req_id;
        push_request(mgr, req);

        pm::SchedulerResponse resp{};
        if (!poll_response_for(mgr, alloc_req_id, evictable, resp))
            throw std::runtime_error("ALLOCATE response timeout");

        if (resp.error_code == 0) {
            // Success -- remove from evictable if present
            auto it = std::find(evictable.begin(), evictable.end(), resp.slot_id);
            if (it != evictable.end()) evictable.erase(it);
            return resp.slot_id;
        }

        // ALLOCATE failed (PM out of free slots)
        if (evictable.empty())
            throw std::runtime_error("capacity exhausted");

        // Wait for a CANCEL ack to free a slot, then retry
        if (!drain_one_cancel_ack(mgr, evictable))
            throw std::runtime_error("timeout waiting for CANCEL ack");
    }
}
```

The state machine for a single allocation attempt:

```
  ALLOCATE --> success (error_code == 0)?
     |             |
     | yes         | no
     |             v
     |         evictable empty?
     |             |
     |       yes   |   no
     |        |    |    |
     |   THROW |    v
     |         |  drain_one_cancel_ack()
     |         |    |
     |         |    v
     |         |  retry ALLOCATE
     v         v
   return slot_id
```

The `poll_response_for` helper (lines 131-156) is critical: while waiting for the ALLOCATE response, it also drains any interleaved CANCEL ack responses from the previous batch. Each CANCEL ack signals that a slot has completed the deferred cancellation protocol ([Ch6 deferred_cancellation.md](../ch6_lifecycle_is_integration_kv_tiering/deferred_cancellation.md)) and is now truly free:

```cpp
bool poll_response_for(
    pm::DecodeScheduler& mgr,
    uint32_t expected_req_id,
    std::vector<uint32_t>& evictable,
    pm::SchedulerResponse& matching) {
    for (int i = 0; i < MAX_POLL_ITERATIONS; i++) {
        pm::SchedulerResponse resp{};
        if (mgr.try_pop_response(resp)) {
            if (resp.request_id == expected_req_id) {
                matching = resp;
                return true;
            }
            // Treat as CANCEL ack: remove from evictable
            auto it = std::find(evictable.begin(), evictable.end(), resp.slot_id);
            if (it != evictable.end()) evictable.erase(it);
            continue;
        }
        std::this_thread::yield();
    }
    return false;
}
```

This design eliminates idle time between batches: the allocator only blocks when the PM has not yet freed a slot by the time the next batch needs it.

### End-of-Batch CANCEL and Last-Batch Drain

Each batch ends with fire-and-forget CANCELs (lines 849-856):

```cpp
for (uint32_t u = batch_start; u < batch_end; u++) {
    pm::ISRequest req{};
    req.type = pm::RequestType::CANCEL;
    req.request_id = req_id++;
    req.slot_id = users[u].slot_id;
    push_request(mgr, req);
    evictable.push_back(users[u].slot_id);
}
```

The last batch's CANCEL acks are drained explicitly after all batches complete (lines 898-910):

```cpp
while (!evictable.empty()) {
    if (!drain_one_cancel_ack(mgr, evictable)) {
        std::cerr << "WARNING: " << evictable.size()
                  << " slot(s) did not receive CANCEL ack during cleanup";
        break;
    }
}
```

## Worked Example: Tracing a Batch

Consider a run with `--num-users 4 --batch-size 2 --max-turns 2 --max-decode 32`:

```
  === Batch 1/2: users 0..1 (2 slots) ===

  1. ALLOCATE(req_id=0) -> slot_id=0
  2. ALLOCATE(req_id=1) -> slot_id=1

  3. SUBMIT(req_id=2, slot=0, tokens=[prompt_u0_t0], max_new=32)
  4. SUBMIT(req_id=3, slot=1, tokens=[prompt_u1_t0], max_new=32)

  5. [decode loop -- OutputMessage tokens arrive]
     User 0: token, token, ..., is_complete=true
     -> CONTINUE(req_id=4, slot=0, tokens=[prompt_u0_t1], max_new=32)
     User 1: token, ..., is_complete=true
     -> CONTINUE(req_id=5, slot=1, tokens=[prompt_u1_t1], max_new=32)

  6. [decode loop continued]
     User 0: tokens..., is_complete=true (turn 2 done)
     User 1: tokens..., is_complete=true (turn 2 done)
     batch_users_done == 2

  7. Snapshot spec stats for slots 0, 1
  8. CANCEL(req_id=6, slot=0)  -- fire-and-forget
     CANCEL(req_id=7, slot=1)  -- fire-and-forget
     evictable = [0, 1]

  === Batch 2/2: users 2..3 (2 slots) ===

  9. allocate_one_with_eviction(req_id=8):
     -> ALLOCATE(req_id=8) -> error_code != 0 (slots still in CANCEL drain)
     -> drain_one_cancel_ack() -> CANCEL ack for slot 0 arrives, evictable = [1]
     -> retry ALLOCATE(req_id=9) -> slot_id=0 (recycled)
     -> evictable = [1] (slot 0 removed from evictable)

  10. allocate_one_with_eviction(req_id=10):
      -> ALLOCATE(req_id=10) -> slot_id=1 (CANCEL ack for slot 1 arrived in poll)
      -> evictable = []

  11. [repeat SUBMIT/decode/CONTINUE/decode cycle for users 2,3]
  12. CANCEL fire-and-forget for slots 0, 1
  13. Drain final CANCEL acks in cleanup
```

## Speculative Decode Support

The runner supports assigning a subset of users to speculative decode via `--num-spec-decode-users N`. Within each batch, `N` users are randomly selected:

```cpp
// Lines 691-709
const uint32_t batch_spec = std::min(num_spec_decode_users, batch_size_actual);
std::vector<uint32_t> indices(batch_size_actual);
std::iota(indices.begin(), indices.end(), batch_start);
std::mt19937 rng(std::random_device{}());
std::shuffle(indices.begin(), indices.end(), rng);
for (uint32_t i = 0; i < batch_spec; i++) {
    users[indices[i]].spec_decode = true;
}
```

The `spec_decode` flag is passed through `GenerationParams` on both SUBMIT and CONTINUE requests. Per-slot accept/reject counters are snapshotted before CANCEL frees the slot (lines 837-843), because `mgr.get_spec_accepts(slot_id)` reads slot-indexed state that would be recycled across batches:

```cpp
for (uint32_t u = batch_start; u < batch_end; u++) {
    snapshot_accepts[u] = mgr.get_spec_accepts(users[u].slot_id);
    snapshot_rejects[u] = mgr.get_spec_rejects(users[u].slot_id);
    batch_accepts += snapshot_accepts[u];
    batch_rejects += snapshot_rejects[u];
}
```

## Multi-Turn Conversation Flow

Each user processes multiple turns through the SUBMIT/CONTINUE lifecycle:

```
Turn 0: SUBMIT with prompt[0]
  -> prefill -> decode -> COMPLETE
Turn 1: CONTINUE with prompt[1]
  -> prefill (from current_position) -> decode -> COMPLETE
  ...
Turn N-1: CONTINUE with prompt[N-1]
  -> prefill -> decode -> COMPLETE
  -> mark user done
```

The CONTINUE request (lines 812-826) reuses the existing KV cache for the slot (see [Ch6 multi_turn_continue.md](../ch6_lifecycle_is_integration_kv_tiering/multi_turn_continue.md)). Sampling and acceptance parameters are applied uniformly to every user and every turn. A user is marked done when either all turns are completed (`u.current_turn >= turn_limits[user_idx]`) or the context window is filled (`u.ctx_exhausted`).

## End-to-End Data Flow

Here is a traced sequence for one user through a single turn in tokenizer mode:

```
1. Raw text: "Explain quantum computing"
   | apply_deepseek_chat_template()
2. Templated: "<|begin_of_sentence|><|User|>Explain quantum computing<|Assistant|>"
   | tokenizer.encode()
3. Token IDs: [151646, 151644, 35078, 34420, 26340, 151645]  (example IDs)
   | push_request(SUBMIT, slot=0, tokens=[...], gen={max_new_tokens=256, ...})

4. DecodeScheduler API thread:
   -> Store prompt in PromptTable
   -> Set state = PREFILL
   -> Push to PrefillQueue

5. DecodeScheduler Writer thread:
   -> Chunked prefill (24 tokens/chunk)
   -> serialize_inject() with use_deepseek_md_format=true
   -> H2D socket write

6. Device model pipeline:
   -> Process tokens, build KV cache
   -> On last prefill token: sample first decode token
   -> Write ResultPage to D2H socket

7. DecodeScheduler Reader thread:
   -> deserialize_result()
   -> Detect prefill result -> skip (prefill_in_flight)
   -> Detect first sampled token -> OutputMessage{slot=0, token_id=T1}

8. Host (runner) main loop:
   -> try_pop_output() -> OutputMessage{token_id=T1}
   -> user_output_ids[0][0].push_back(T1)
   -> Record timing (TTFT on first token)

   ... decode loopback continues until EOS or max_decode ...

9. Completion:
   -> OutputMessage{is_complete=true}
   -> If more turns: push_request(CONTINUE, tokens=[next_turn_prompt])
   -> If done: mark user done

10. End of batch:
    -> Snapshot spec stats
    -> Issue CANCEL (fire-and-forget)

11. Output display (tokenizer mode):
    -> tokenizer.decode(user_output_ids[0][0])
    -> "Quantum computing uses quantum bits..."
```

## Output and Metrics

### Decoded Text Output (Tokenizer Mode)

In tokenizer mode, output token IDs are collected per-user per-turn and decoded at the end (lines 1000-1011). Token IDs are accumulated during the decode loop, then bulk-decoded after all batches complete to avoid per-token decode overhead during the benchmark-sensitive completion loop.

### Per-User Summary

Each user gets a summary line showing ISL, output tokens, average TTFT, average TPOT, and spec accept stats:

```
--- Per-User Summary ---
  User  0 | slot  0 | SD | turns   5 | ISL    1280 | tokens    1024 | TTFT   45.32 ms | TPOT    2.15 ms | spec 42/50 (84.0% accepted)
  User  1 | slot  1 |    | turns   5 | ISL     640 | tokens     512 | TTFT   38.71 ms | TPOT    2.28 ms | spec 0/0 (0.0% accepted)
```

The `SD` column indicates speculative decode users. Cumulative ISL accounts for all turns' prompt lengths.

### Metric Definitions

| Metric | Formula | Scope |
|--------|---------|-------|
| **TTFT** | Time from `turn_start` to first non-empty `OutputMessage` | Per-turn, per-user |
| **TPOT** | `(last_token_time - first_token_time) / (turn_tokens - 1)` | Per-turn, per-user |
| **ITL** | Wall-clock time between consecutive output tokens | Global, all users pooled |
| **Request throughput** | `grand_total_turns / bench_duration_s` | Aggregate |
| **Output token throughput** | `grand_total_tokens / bench_duration_s` | Aggregate |
| **Peak output token throughput** | Max tokens in any 1-second sliding window | Aggregate |
| **Total token throughput** | `(total_input + total_output) / bench_duration_s` | Aggregate |

## Comparison with the Loopback Connector

| Feature | Loopback Connector | DeepSeek Runner |
|---------|-------------------|-----------------|
| Wire format | Default (non-DeepSeek) | DeepSeek MD (`use_deepseek_md_format=true`) |
| Prompt source | Synthetic (`make_prompt`) | Synthetic or tokenizer (HuggingFace) |
| Chat template | None | DeepSeek R1/V3 chat template |
| EOS handling | Default EOS token | Auto-detected from tokenizer |
| Speculative decoding | Not supported | Supported (`--num-spec-decode-users`) |
| Thinking tokens | Not supported | Auto-detected (`<think>`/`</think>`) |
| Batching | Single batch | Multi-batch with pipelined teardown |
| Slot allocation | Upfront, fail on error | `allocate_one_with_eviction` |
| Cancellation | Synchronous wait for INACTIVE | Fire-and-forget with eviction |
| Sampling parameters | Defaults only | Full control (temperature, top_p, top_k) |
| Multi-turn | Fixed turns for all users | Per-user turn limits from prompts file |
| Decoded output | None | Full detokenized text in tokenizer mode |
| Spec stats | None | Per-user + per-batch + aggregate |
