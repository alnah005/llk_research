# Agent B Review: Chapter 4 — The Unpack-Math-Pack Pipeline — Pass 1

1. **Matmul register reuse direction is inverted** (`unpack_operations.md`, lines 60-61). The doc states: "when `ct_dim >= rt_dim`, Source A (which holds `in1`/`inB`) is reused across multiple Source B loads; otherwise Source B (which holds `in0`/`inA`) is reused." The source code (`llk_unpack_AB_matmul.h` and `llk_math_matmul.h`) shows the opposite: when `reuse_a` (`ct_dim >= rt_dim`) is true, `CLR_DVALID_SrcB_Disable` is set, meaning **Source B** (holding `in0`/`inA`) stays valid and is reused, while Source A (holding `in1`/`inB`) cycles through new tiles. The variable `reuse_a` refers to reusing `inA` (the logical A operand), not the SrcA register. A reader implementing matmul tile scheduling based on this doc would get the reuse strategy backwards.

2. **`relu_config` bit layout is inaccurate** (`pack_operations.md`, lines 128-130). The doc states: "Bits [3:0] select the `ReluType`. The remaining bits encode the threshold value." The source (`llk_pack_common.h`, `_llk_pack_relu_config_`) shows `(config & 0xf)` for mode (bits [3:0] -- correct) but `(config >> 16)` for threshold (bits [31:16]). Bits [15:4] are unused. Saying "the remaining bits" implies bits [31:4] all encode the threshold, which would cause someone constructing a `relu_config` value to place the threshold in the wrong bit positions. The doc should state "Bits [31:16] encode the threshold value."
