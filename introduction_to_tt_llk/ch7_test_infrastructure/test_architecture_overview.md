# Test Architecture Overview

Every LLK test consists of two files that work together: a **Python file** that orchestrates the test and a **C++ file** that contains the kernel code. The Python side handles variant enumeration, stimuli generation, golden computation, compilation orchestration, and result validation. The C++ side contains the actual kernel logic that runs on Tensix cores, structured around the three TRISC threads (Unpack, Math, Pack).

## The Two-File Model

### Python Side

The Python file lives in `tests/python_tests/` and follows the naming convention `test_*.py` for functional tests or `perf_*.py` for performance tests. It is a standard `pytest` module that uses the `@parametrize` decorator (defined in `tests/python_tests/helpers/param_config.py`) to generate test variants.

A single Python test function performs these steps in order:

1. **Parametrization** -- The `@parametrize` decorator enumerates combinations of formats, tile counts, operation types, and other parameters. Each combination becomes a distinct test variant with a unique variant ID.
2. **Stimuli generation** -- The `generate_stimuli()` function (from `tests/python_tests/helpers/stimuli_generator.py`) creates input tensors in the requested data format and dimensions.
3. **Golden computation** -- A golden generator class (from `tests/python_tests/helpers/golden_generators.py`) computes the expected output using PyTorch.
4. **Configuration** -- A `TestConfig` object bundles together the C++ source path, format configuration, template parameters, runtime parameters, and stimuli configuration.
5. **Execution** -- Calling `configuration.run(workers_tensix_coordinates)` triggers compilation (if needed) and runs the kernel on hardware.
6. **Validation** -- The result read back from L1 memory is compared against the golden tensor using `passed_test()` (from `tests/python_tests/helpers/utils.py`), which internally calls `torch.isclose()` with format-specific tolerances.

### C++ Side

The C++ file lives in `tests/sources/` and follows the naming convention `*_test.cpp` for functional tests or `*_perf.cpp` for performance tests. It must contain three `#ifdef`-guarded sections, one for each TRISC thread:

```cpp
#include <cstdint>
#include "ckernel.h"
#include "llk_defs.h"

// Globals
std::uint32_t unp_cfg_context          = 0;
std::uint32_t pack_sync_tile_dst_ptr   = 0;
std::uint32_t math_sync_tile_dst_index = 0;

#ifdef LLK_TRISC_UNPACK
#include "llk_unpack_AB.h"
#include "llk_unpack_common.h"
#include "params.h"

void run_kernel(RUNTIME_PARAMETERS params) {
    // Unpack kernel code
}
#endif

#ifdef LLK_TRISC_MATH
#include "llk_math_common.h"
#include "llk_math_eltwise_binary.h"
#include "params.h"

void run_kernel(RUNTIME_PARAMETERS params) {
    // Math kernel code
}
#endif

#ifdef LLK_TRISC_PACK
#include "llk_pack.h"
#include "llk_pack_common.h"
#include "params.h"

void run_kernel(RUNTIME_PARAMETERS params) {
    // Pack kernel code
}
#endif
```

Each section is compiled independently into a separate ELF binary. The `ckernel.h` and `llk_defs.h` includes are mandatory for all kernels. The three global variables (`unp_cfg_context`, `pack_sync_tile_dst_ptr`, `math_sync_tile_dst_index`) are required by the LLK API runtime. Each section includes `params.h`, which in turn includes the auto-generated `build.h` header. The `RUNTIME_PARAMETERS` macro and `struct RuntimeParams` are generated into `build.h` by the Python infrastructure; `params.h` merely includes `build.h` and other necessary type definitions.

## The Compilation Pipeline

When `configuration.run()` is called, the test infrastructure performs the following compilation sequence:

### 1. build.h Generation

For every unique test variant, a `build.h` header file is generated at `/tmp/tt-llk-build/<test_path>/<variant_hash>/build.h`. This file contains:

- **Format configuration** -- Data format constants derived from the `formats` argument of `TestConfig`, processed through the data format inference engine (`tests/python_tests/helpers/data_format_inference.py`).
- **Template parameter definitions** -- `constexpr` variable definitions generated by calling `convert_to_cpp()` on each element of the `templates` list. For example, a stochastic rounding parameter might produce:
  ```cpp
  constexpr auto STOCHASTIC_RND = ckernel::StochRndType::None;
  ```
- **Runtime parameter struct** -- A `struct RuntimeParams` declaration generated by calling `convert_to_struct_fields()` on each element of the `runtimes` list, plus the `RUNTIME_PARAMETERS` macro that abbreviates `const struct RuntimeParams&`.

The variant hash is computed from all compile-time arguments (formats, templates, and other `TestConfig` constructor parameters that affect compilation). Variants that share the same compile-time configuration share a single compiled set of ELFs.

### 2. params.h

The file `tests/helpers/include/params.h` is a static header that includes `build.h` and other necessary type definitions. It serves as the bridge between the auto-generated configuration and the kernel code. Every `#ifdef` section in the C++ test file includes `params.h` to gain access to the `RUNTIME_PARAMETERS` macro and the `struct RuntimeParams` type, both of which are generated into `build.h`.

### 3. SFPI Compilation

The C++ source file is compiled three times using the SFPI compiler -- a custom RISC-V toolchain based on the [SFPI project](https://github.com/tenstorrent/sfpi). Each compilation pass defines one of the three preprocessor macros:

| Pass | Macro defined | Output |
|------|--------------|--------|
| 1 | `LLK_TRISC_UNPACK` | `unpack.elf` |
| 2 | `LLK_TRISC_MATH` | `math.elf` |
| 3 | `LLK_TRISC_PACK` | `pack.elf` |

The resulting ELF files are placed in the variant's build directory under `/tmp/tt-llk-build/`. Each ELF is linked against the appropriate linker script from `tests/helpers/ld/` -- either a performance layout (using local data memory for better runtime performance) or a debug layout (placing everything in L1, necessary for coverage collection).

### 4. Firmware

Each ELF includes a firmware layer defined in `tests/helpers/src/trisc.cpp`. This firmware provides:

- `void do_crt0(void)` -- The C-runtime initialization function (defined in `boot.h`).
- `void main(void)` -- Entry point that calls the test's `run_kernel()` function.
- `void gcov_dump(void)` -- Coverage data serialization (included only when compiled with `--coverage`).

## Template Parameters vs. Runtime Parameters

The test infrastructure distinguishes between two categories of parameters that flow from Python to C++:

**Template parameters** are compile-time constants. They become `constexpr` variables in `build.h` and are used as template arguments to LLK API function calls. Changing a template parameter requires recompilation of the variant. All template parameter classes inherit from `TemplateParameter` (defined in `tests/python_tests/helpers/test_variant_parameters.py`) and implement `convert_to_cpp() -> str`.

**Runtime parameters** are values loaded into L1 memory before kernel execution begins. They become fields of `struct RuntimeParams`, which the kernel receives as the `params` argument to `run_kernel()`. Changing a runtime parameter does not require recompilation -- the Python side simply writes new values to L1. All runtime parameter classes inherit from `RuntimeParameter` and implement both `convert_to_cpp() -> str` and `convert_to_struct_fields() -> tuple[str, str]`. The second method returns a C++ struct field declaration and a `struct.pack` format character that controls how the Python value is serialized into the L1 byte stream.

A key design property: `RuntimeParameter` subclasses can also be passed in the `templates` list (they are treated as compile-time constants in that case), but `TemplateParameter` subclasses cannot be placed in the `runtimes` list.
